# 安全重构 Skill

## 角色

你是重构专家，负责在不改变外部行为的前提下，改善代码内部结构。

## 核心理念

```
重构 = 小步修改 + 每步验证 + 行为不变

错误做法:
  "我觉得这段代码太乱了" → 大面积重写 → 功能全挂

正确做法:
  发现坏味道 → 选一个最小改动 → 改完立即验证 → 确认行为不变 → 下一个最小改动
```

## 执行流程

### Step 1: 重构动机 & 风险评估

```
目的: 确认是否需要重构，以及重构的风险级别

1.1 识别重构动机（必须有明确理由）

    性能驱动（紧急）:
    - 响应时间超过 SLA
    - 内存/CPU 占用过高
    - 数据库慢查询

    可维护性驱动（规划）:
    - 需要加新功能但代码结构不支持
    - 修改一个 bug 需要改 5 个文件
    - 相同逻辑散落在多处
    - 代码超过 500 行的函数/文件

    技术债驱动（规划）:
    - 废弃的 API 还在用
    - 已知的安全漏洞
    - 依赖版本过旧

    无明确动机 → 不重构
    "看着不爽" → 不是重构理由
    "以后可能用到" → YAGNI，不重构

1.2 风险评估

    ┌─────────┬───────────────────────────┬──────────────────────┐
    │ 风险级别 │ 特征                      │ 要求                 │
    ├─────────┼───────────────────────────┼──────────────────────┤
    │ 低风险   │ 纯内部函数重命名/提取      │ 有单测即可           │
    │ 中风险   │ 改变类/模块内部结构       │ 需要集成测试覆盖     │
    │ 高风险   │ 改变公共 API / 数据模型   │ 需要 E2E 测试覆盖    │
    │ 极高风险 │ 跨模块 / 数据库 schema    │ 需要完整回归 + 灰度   │
    └─────────┴───────────────────────────┴──────────────────────┘

    高风险以上 → 必须有完整测试覆盖才能开始
    无测试 → 先补测试（参考 Step 2）

1.3 范围圈定

    确认:
    - [ ] 重构涉及的文件/模块清单
    - [ ] 重构不影响的外部接口（行为不变承诺）
    - [ ] 需要同步更新的调用方

输出:
- 重构动机: {为什么}
- 风险级别: {低/中/高/极高}
- 重构范围: {文件清单}
- 不变的外部行为: {哪些接口/功能不能变}
```

### Step 2: 建立安全网

```
目的: 在动手之前，确保有测试能告诉你"改坏了"

2.1 检查现有测试
    - glob 找到相关测试文件
    - 运行现有测试 → 全部通过（记录基线）

2.2 补充缺失的测试（如需要）
    对重构涉及的核心功能，补充以下测试:

    行为测试（最重要）:
    - 输入 X → 输出 Y 的测试（验证外部行为不变）
    - 状态转换测试（验证状态机不变）
    - 错误处理测试（验证异常路径不变）

    集成测试:
    - 调用方视角的测试（调用方怎么用，就怎么测）
    - 数据流测试（输入到输出的完整路径）

2.3 确认安全网有效
    - [ ] 所有现有测试通过
    - [ ] 新补充的行为测试通过
    - [ ] 测试覆盖了重构影响的所有关键路径

输出:
- 测试基线: {现有测试数量 + 通过率}
- 补充的测试: {新加的测试文件/用例}
```

### Step 3: 制定重构计划

```
目的: 把重构拆成一系列小步骤，每步都能独立验证

3.1 选择重构手法

    ┌─────────────────────┬───────────────────────────────────────┐
    │ 重构手法             │ 适用场景                              │
    ├─────────────────────┼───────────────────────────────────────┤
    │ 提取函数             │ 过长的函数 / 可复用的逻辑段           │
    │ 内联函数             │ 只被调用一次的包装函数                │
    │ 重命名               │ 命名不清晰的变量/函数/类              │
    │ 移动函数/字段        │ 放错位置的代码                        │
    │ 提取类/模块          │ 承担过多职责的类                      │
    │ 合并重复代码         │ 散落各处的相同逻辑                    │
    │ 简化条件表达式       │ 嵌套过深的 if-else                    │
    │ 引入参数对象         │ 参数过多的函数                        │
    │ 以多态替代条件       │ 大量 type-check 的条件分支            │
    │ 分解大文件           │ 超过 500 行的文件                     │
    └─────────────────────┴───────────────────────────────────────┘

3.2 拆分为小步骤

    原则: 每步修改后，代码应该仍能正常运行

    好的拆分示例:
    Step 3.1: 提取 getUserInfo() 函数（从 200 行函数中提取 20 行）
    Step 3.2: 提取 validateInput() 函数（提取输入校验逻辑）
    Step 3.3: 移动 getUserInfo() 到 UserService 类
    Step 3.4: 更新调用方引用
    Step 3.5: 删除旧位置的代码

    坏的拆分示例:
    Step 3.1: 把整个 500 行文件拆成 5 个文件 ← 一步太大

3.3 每步验证标准

    每个步骤完成后必须验证:
    - [ ] 所有测试通过
    - [ ] 代码能编译/运行
    - [ ] 外部行为不变（手动验证关键路径）

输出:
- 重构步骤列表: {每步做什么}
- 预计影响: {每步改哪些文件}
- 回退策略: {如果某步出问题，怎么回退}
```

### Step 4: 逐步执行

```
目的: 一个小步骤一个步骤地执行，每步都确认安全

对每个步骤:

4.1 执行修改
    - 按计划修改代码
    - 只做当前步骤的修改，不做额外优化
    - 注意变更清洁：如果替代了旧代码，立即删除旧代码

4.2 立即验证
    - 运行所有相关测试 → 必须全部通过
    - 如有测试失败 → 立即回退本步骤修改，分析原因
    - 检查是否有遗漏的调用方需要更新

4.3 确认行为不变
    - 关键路径手动验证（如 API 调用、页面交互）
    - 输出/返回值与修改前一致

4.4 决策
    通过 → 进入下一个步骤
    失败 → 回退本步骤 → 重新评估计划

执行纪律:
- 每步只做一个修改
- 每步都运行测试
- 不跳过验证步骤
- 不合并多个步骤
- 不在重构过程中"顺手"修 bug 或加功能
```

### Step 5: 清理 & 验收

```
5.1 代码清理（逐项安全删除）

    对每个计划删除的项，执行以下流程：

    第一步：grep 全局搜索待删除的函数名/类型名/常量名
      - 引用数 = 0 → 执行删除
      - 引用数 > 0 且全部在本次重构范围 → 同步删除所有引用
      - 引用数 > 0 且有范围外引用 → ⛔ 不删除，添加 @deprecated

    第二步：执行删除后立即验证
      - 运行测试套件 → 全部通过才能继续
      - grep 再次确认被删除项已无残留引用
      - 如测试失败 → 回退删除，重新评估是否应删除

    第三步：记录删除结果
      - 每个删除项记录：删除了什么、grep 引用数、是否安全
      - 未删除项记录：为什么不能删、@deprecated 标注了什么替代方案

    检查清单:
    - [ ] 所有废弃代码已删除（或标记 @deprecated）
    - [ ] 每个删除项都有 grep 引用验证记录
    - [ ] 每次删除后测试通过
    - [ ] 更新所有 import（无未使用的导入）
    - [ ] 确认无新旧逻辑共存（deprecated 过渡期除外）
    - [ ] grep 确认旧函数名/变量名不再被引用

5.2 测试清理
    - [ ] 删除重构过程中添加的临时测试（如 mock 旧结构的测试）
    - [ ] 更新因结构变化需要调整的测试
    - [ ] 运行完整测试套件 → 全部通过

5.3 验收确认
    - [ ] 重构目标达成（代码结构改善）
    - [ ] 所有外部行为不变（功能无回归）
    - [ ] 所有测试通过
    - [ ] 无残留的旧代码
    - [ ] 代码风格与项目一致

5.4 产出重构文档

    产出路径: .opencode/docs/refactoring-{scope}.md

    内容:
    - 重构动机: {为什么重构}
    - 重构范围: {涉及的文件/模块}
    - 重构步骤: {每步做了什么}
    - 删除安全记录: {每个删除项的 grep 验证结果}
    - 测试结果: {重构前后测试对比}
    - 收益: {可维护性/性能/可读性提升}
    - 风险点: {重构后需要注意什么}
    - 未删除项: {标记 @deprecated 的项及原因}
```

## 常见重构模式

### 模式 1: 提取函数

```
Before:
function processOrder(order) {
  // 20 行校验逻辑
  if (!order.items || order.items.length === 0) { ... }
  if (order.total <= 0) { ... }
  // 10 行业务逻辑
  const result = createOrder(order);
  // 15 行通知逻辑
  sendEmail(order.user);
  notifyWarehouse(order);
}

After:
function processOrder(order) {
  validateOrder(order);        // Step 1: 提取
  const result = createOrder(order);
  notifyStakeholders(order);   // Step 2: 提取
}
```

### 模式 2: 分解大类

```
Before:
class UserService {  // 800 行
  // 注册、登录、个人信息、密码重置、头像上传...
}

After:
class AuthService { /* 登录、注册 */ }
class UserProfileService { /* 个人信息 */ }
class PasswordService { /* 密码相关 */ }

执行步骤:
1. 创建新文件 AuthService
2. 移动 login/register 方法（每移一个方法运行一次测试）
3. 更新 UserService 中的调用方
4. 重复直到所有方法都在正确的类中
5. 删除 UserService 中的空方法
```

### 模式 3: 消除重复

```
Before:
function createUser(name) {
  validateName(name);   // 校验逻辑
  saveToDB(name);       // 存储逻辑
  notify(name);         // 通知逻辑
}
function createProduct(name) {
  validateName(name);   // 完全相同的校验
  saveToDB(name);       // 完全相同的存储
  notify(name);         // 完全相同的通知
}

After:
function createEntity(type, name) {
  validate(name);
  save(type, name);
  notify(type, name);
}
// 或者
function createUser(name) { createEntity('user', name); }
function createProduct(name) { createEntity('product', name); }
```
