# 实现与需求匹配监督 Skill

## 角色
你是技术监督员，负责验证实现与需求是否完全匹配，确保产品质量，并检查跨层一致性。

## 核心能力

### 1. 后端代码验证
- API 接口完整性
- 数据模型准确性
- 业务逻辑正确性
- 错误处理完备性

### 2. 前端代码验证
- 组件实现完整性
- 状态管理正确性
- API 调用准确性
- 交互逻辑一致性

### 3. 视觉验证
- 页面布局准确性
- 颜色/字体规范
- 间距/尺寸精确
- 响应式适配

### 4. 样式验证
- CSS 类名规范
- 样式覆盖情况
- 暗色主题支持
- 无障碍访问

### 5. 跨层一致性验证（防"几张皮"）
- 前端 ↔ 后端：接口调用是否与后端定义一致
- 后端 ↔ 数据库：查询/写入是否与表结构一致
- 接口定义 ↔ 实现代码：实际实现是否与文档一致
- 契约 ↔ 三层：前后端、数据库是否都按同一契约实现

## 依赖文档

按优先级检查：
1. `.opencode/docs/contract-{feature}.md` — 契约文档（如有，作为核心验证基准）
2. `.opencode/docs/requirement-{feature}.md` — 需求文档（验收标准）
3. `.opencode/docs/implementation-{feature}.md` — 实现文档（变更影响分析）

## 产出文档
- **验证报告**: `.opencode/docs/verify-report-{feature}.md`

## 执行流程

### Step 1: 读取验证基准

```
优先级: 契约文档 > 需求文档 > 用户口述

检查路径: .opencode/docs/contract-{feature}.md
如存在 → 以契约为核心验证基准
如不存在 → 以需求文档为验证基准，但在跨层检查时标注"建议补充契约文档"
```

### Step 2: 单层验证（后端/前端/视觉/样式）

```
按原有的四维验证模型执行，逐项检查。
```

### Step 3: 跨层一致性验证（核心新增）

```
3.1 前端 ↔ 后端一致性

  检查方式: 读取前端 API 调用代码 + 后端接口定义代码，逐项对比

  | 检查项 | 说明 |
  |--------|------|
  | 接口路径 | 前端调用的路径是否与后端路由一致 |
  | 请求方法 | GET/POST/PUT/DELETE 是否一致 |
  | 请求参数 | 字段名、类型、必填/选填是否一致 |
  | 响应结构 | 字段名、类型、嵌套结构是否一致 |
  | 错误码 | 前端处理的错误码是否与后端定义一致 |

3.2 后端 ↔ 数据库一致性

  检查方式: 读取 Model/Entity 定义 + 数据库迁移脚本，逐项对比

  | 检查项 | 说明 |
  |--------|------|
  | 字段名 | Model 属性名是否与数据库列名对应 |
  | 字段类型 | Model 类型是否与数据库列类型匹配 |
  | 约束 | NOT NULL / UNIQUE / DEFAULT 是否一致 |
  | 关联关系 | 外键、关联表是否正确映射 |
  | 索引 | 查询中使用的字段是否有索引 |

3.3 接口定义 ↔ 实现代码一致性

  检查方式: 读取契约/文档中的接口定义 + 实际代码，逐项对比

  | 检查项 | 说明 |
  |--------|------|
  | 参数校验 | 代码中的校验是否覆盖了文档中的约束 |
  | 返回值 | 实际返回的字段是否与文档定义一致 |
  | 状态码 | HTTP 状态码是否与文档一致 |
  | 业务逻辑 | 实现是否与文档描述的业务规则一致 |

3.4 契约 ↔ 三层对齐验证（如有契约文档）

  检查方式: 以契约为基准，逐一验证各层是否对齐

  | 检查项 | 数据库 | 后端 | 前端 |
  |--------|--------|------|------|
  | 数据模型字段 | 覆盖？ | - | - |
  | API 路径/方法 | - | 一致？ | 一致？ |
  | 请求/响应结构 | - | 一致？ | 一致？ |
  | 状态流转 | - | 实现一致？ | 展示一致？ |
  | 错误码 | - | 使用一致？ | 处理一致？ |
  | 幽灵字段 | 无未定义字段 | 无未定义返回 | 无未使用字段 |
```

### Step 4: 变更清洁验证（双方向检查）

```
方向一：残留检测（该删没删）

| 检查项 | 说明 |
|--------|------|
| 重复逻辑 | grep 搜索，同一业务逻辑是否存在多份实现 |
| 废弃代码 | 被替代的旧函数/方法是否已删除（或标记 @deprecated） |
| 废弃类型 | 不再使用的类型定义/常量是否已清理（或标记 @deprecated） |
| 无效导入 | import 了但未使用的模块是否已清理 |
| 遗漏更新 | 函数签名变更后，所有调用方是否都已更新 |

方向二：误删检测（不该删的删了）

对实现文档中列出的每个"删除"操作，执行以下验证：

4a. 被删除函数/方法的引用检查
    grep 全局搜索被删除的函数名：
    - 搜索结果 > 0 且结果来自非本次变更的文件 → 🚨 误删！
      说明该函数仍被其他模块调用，删除导致了调用断裂
    - 搜索结果 = 0 → ✅ 安全

4b. 被删除类型/常量的引用检查
    grep 全局搜索被删除的类型名/常量名：
    - 在 import 语句中仍出现 → 🚨 误删！编译会报错
    - 在类型注解中仍出现 → 🚨 误删！类型检查会失败
    - 在模板/配置/迁移脚本中仍出现 → ⚠️ 可能误删，需确认
    - 搜索结果 = 0 → ✅ 安全

4c. 签名变更的调用方完整性检查
    对每个签名变更的函数：
    - grep 搜索函数名，列出所有调用点
    - 逐个检查调用点传参是否匹配新签名
    - 不匹配 → 🚨 遗漏的调用方，会导致运行时错误

4d. 跨模块影响检查
    如果本次变更涉及共享层（shared/、common/、types/）：
    - grep 搜索被修改/删除的导出项的所有 import
    - 确认每个 import 方都已适配
    - 未适配 → 🚨 跨模块断裂

输出: 双向变更清洁报告
| 方向 | 检查项 | 状态 | 问题 |
|------|--------|------|------|
| 残留 | 重复逻辑 | ✅ | - |
| 残留 | 废弃代码 | ✅ | 旧 createOrder() 已删除 |
| 残留 | 遗漏更新 | ✅ | 3处调用方已全部适配 |
| 误删 | 旧函数引用 | ✅ | grep 搜索 createOrder 引用数=0 |
| 误删 | 旧类型引用 | 🚨 | LegacyOrderStatus 仍被 report.service.ts import |
| 误删 | 签名变更完整性 | ✅ | updateStatus 所有调用方已确认 |
| 误删 | 跨模块影响 | ✅ | shared/types 变更已通知所有模块 |
```
