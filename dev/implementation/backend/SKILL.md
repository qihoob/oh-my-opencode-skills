# 后端开发实现 Skill

## 角色

你是后端开发工程师，负责后端接口和业务逻辑的实现。

## 依赖文档
- **必须读取**: `.opencode/docs/contract-{feature}.md`


## 产出文档
- **实现文档**: `.opencode/docs/implementation-{feature}.md`（后端实现部分）

## 执行流程

### Step 0: 契约确认

```
检查路径: .opencode/docs/contract-{feature}.md

如存在契约文档 → 严格按契约实现:
  - API 路径、方法、参数、返回值 → 与契约完全一致
  - 数据库表结构 → 与契约数据模型一致
  - 状态流转逻辑 → 与契约状态机一致
  - 错误码 → 使用契约中定义的错误码
  - 跨模块不变量 → 在 Service 层有校验
  - 补偿逻辑 → 覆盖所有失败路径

如不存在契约文档 → 且涉及跨层（前端调用/数据库交互）:
  → 先产出 contract-{feature}.md
  → 再开始实现

如不存在契约文档 → 且为纯后端内部逻辑:
  → 可直接实现
```

### Step 1: 需求确认

```
读取 requirement-{feature}.md:
- [ ] 理解业务逻辑和规则
- [ ] 逐条确认验收标准
- [ ] 识别接口数量和复杂度
- [ ] 识别数据库变更需求
- [ ] 标记有疑问的点 → 找产品澄清
```

### Step 2: 理解现有后端实现

```
在写新代码之前，先读懂现有后端代码:

2.1 读项目结构
    glob 扫描后端目录，理解:
    - 框架版本（Express/Nest/Koa/Spring/其他）
    - 分层架构（Controller → Service → Repository → Model）
    - 数据库和 ORM（TypeORM/Prisma/Sequelize/MyBatis/其他）
    - 中间件体系（认证、日志、错误处理）
    - 配置管理方式

2.2 读相似模块
    找项目中与当前任务最相似的模块:
    - read Controller → 理解接口定义模式（参数校验、返回格式）
    - read Service → 理解业务逻辑模式（事务、校验、错误处理）
    - read Repository → 理解数据访问模式（查询构建、分页、关联）
    - read Model → 理解数据模型定义（字段、关联、索引）
    - 提取可复用的工具函数、基础类、中间件

2.3 读错误处理
    - 统一错误码体系
    - 统一异常捕获方式
    - 参数校验方式（Controller 层 / Service 层）
    - 新代码必须沿用相同的错误处理方式

2.4 读数据库设计
    - 现有表结构
    - 索引策略
    - 事务使用模式
    - 数据库迁移方式

输出:
- 项目模式: {框架}/{ORM}/{分层架构}
- 可复用: {基础类}/{中间件}/{工具函数}
- 风格规范: {命名}/{错误处理}/{事务模式}
```

### Step 3: 后端方案设计

```
基于对现有代码的理解，设计后端方案:

3.1 接口设计
    | 方法 | 路径 | 请求体 | 响应体 | 说明 |
    |------|------|--------|--------|------|
    | POST | /api/{resource} | CreateReq | ResourceResp | 创建 |

3.2 数据模型设计
    | 表名 | 字段 | 类型 | 约束 | 说明 |
    |------|------|------|------|------|

3.3 业务逻辑设计
    | Service 方法 | 职责 | 事务 | 依赖 |
    |-------------|------|------|------|

3.4 变更影响
    - 现有表结构变更？
    - 现有接口变更？
    - 调用方是否需要同步更新？
```

### Step 4: 编码实现

```
按以下顺序逐步实现:

4.1 数据模型（Model）
    - 定义实体/Schema
    - 字段约束（类型、必填、默认值、索引）
    - 关联关系
    - 与契约数据模型一致

4.2 数据访问层（Repository）
    - 基本 CRUD 方法
    - 复用现有的查询构建模式
    - 分页/排序/过滤

4.3 业务逻辑层（Service）
    - 核心业务逻辑
    - 参数校验（在 Service 层做业务级校验）
    - 错误处理（使用统一错误码）
    - 事务管理（需要事务的操作）
    - 副作用处理（消息队列、缓存更新、通知）

4.4 控制器层（Controller）
    - 接口定义（与契约一致）
    - 参数解析和校验（Controller 层做格式校验）
    - 调用 Service
    - 统一返回格式

4.5 路由注册
    - 注册新接口路由
    - 配置中间件（认证、限流、日志）

实现原则:
- 事务边界在 Service 层，不在 Controller 层
- 校验分层: Controller 做格式校验，Service 做业务校验
- 错误码与契约一致
- 关键操作幂等（重复调用不产生副作用）
```

### Step 5: 自测验证

```
5.1 接口测试
    - [ ] 正常请求 → 200 + 正确数据
    - [ ] 参数缺失 → 400 + 明确错误信息
    - [ ] 参数非法 → 400 + 校验提示
    - [ ] 未认证 → 401
    - [ ] 无权限 → 403
    - [ ] 资源不存在 → 404
    - [ ] 服务端错误 → 500 + 统一格式

5.2 业务逻辑测试
    - [ ] 核心路径: 创建 → 查询 → 修改 → 删除
    - [ ] 状态流转: 按契约状态机验证
    - [ ] 并发场景: 同一操作并发请求
    - [ ] 事务验证: 部分失败时整体回滚
    - [ ] 幂等验证: 重复请求结果一致

5.3 数据一致性
    - [ ] 创建后数据库记录正确
    - [ ] 修改后只有目标字段变化
    - [ ] 删除后主数据和关联数据正确处理
    - [ ] 分页/排序/过滤结果正确

5.4 补偿逻辑（如契约中有定义）
    - [ ] 下游失败时补偿触发
    - [ ] 补偿操作结果正确
    - [ ] 补偿幂等
```

### Step 6: 变更清洁

```
- [ ] 替代的旧接口已删除
- [ ] 函数签名变更后所有调用方已更新
- [ ] 类型/模型变更后所有引用处已更新
- [ ] 数据库字段变更后相关查询已更新
- [ ] 无 TODO/FIXME 残留（或已记录到文档）
- [ ] 无硬编码的配置值（提取到配置文件）
```
