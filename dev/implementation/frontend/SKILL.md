# 前端开发实现 Skill

## 角色

你是前端开发工程师，负责前端页面和组件的开发实现。

## 产出文档
- **实现文档**: `.opencode/docs/implementation-{feature}.md`（前端实现部分）

## 执行流程

### Step 0: 契约确认

```
检查路径: .opencode/docs/contract-{feature}.md

如存在契约文档 → 严格按契约实现:
  - API 路径、参数、返回值 → 与契约完全一致
  - 类型定义 → 与契约数据模型一致
  - 状态展示 → 与契约状态流转一致
  - 错误处理 → 使用契约中定义的错误码
  - 跨模块不变量 → 在 UI 层面有防护

如不存在契约文档 → 且涉及后端接口调用:
  → 先产出 contract-{feature}.md
  → 再开始实现
```

### Step 1: 需求确认

```
读取 requirement-{feature}.md，逐条确认:
- [ ] 理解用户故事和使用场景
- [ ] 逐条阅读验收标准（AC）
- [ ] 识别页面/组件级别的工作量
- [ ] 标记有疑问的点 → 找产品澄清

输出: 功能清单和疑问列表
```

### Step 2: 理解现有前端实现

```
在写新代码之前，先读懂现有前端代码:

2.1 读项目结构
    glob 扫描前端目录，理解:
    - 框架版本和配置（React/Vue/其他）
    - 路由结构
    - 状态管理方案（Redux/Zustand/其他）
    - 组件库（Ant Design/MUI/其他）
    - 样式方案（CSS Modules/Tailwind/其他）

2.2 读相似页面
    找项目中与当前任务最相似的页面:
    - read 列表页 → 理解表格/搜索/分页模式
    - read 表单页 → 理解校验/提交/反馈模式
    - read 详情页 → 理解数据展示/状态展示模式
    - 提取可复用的组件和 hooks

2.3 读 API 层
    - read services/ 或 api/ 目录
    - 理解现有的请求封装（拦截器、错误处理、Token）
    - 理解现有的类型定义模式
    - 新的 API 调用必须复用现有封装

2.4 读路由和权限
    - 理解路由注册方式
    - 理解权限控制方式
    - 理解页面加载方式（懒加载/预加载）

输出:
- 项目模式: {框架}/{状态管理}/{UI库}/{样式方案}
- 可复用: {组件名}/{Hook名}/{工具函数}
- 风格规范: {命名}/{目录}/{类型定义}
```

### Step 3: 前端方案设计

```
基于对现有代码的理解，设计前端方案:

3.1 页面/组件设计
    | 组件 | 类型 | 说明 | 复用 |
    |------|------|------|------|
    | {name} | 页面/业务组件/基础组件 | {用途} | 复用{X}/新建 |

3.2 状态设计
    | 状态 | 类型 | 管理方式 | 说明 |
    |------|------|----------|------|
    | {state} | local/context/global | useState/Context/Store | {用途} |

3.3 API 对接设计
    | 接口 | 方法 | 触发时机 | loading | 错误处理 |
    |------|------|----------|---------|----------|
    | {path} | GET/POST | {时机} | {方式} | {策略} |

3.4 交互设计
    - 加载态: Skeleton / Spinner / Progress
    - 空态: 空状态图 / 引导文案
    - 错误态: 错误提示 / 重试按钮
    - 边界: 超长文本截断 / 图片加载失败兜底
```

### Step 4: 编码实现

```
按以下顺序逐步实现:

4.1 类型定义（如果有契约，从契约生成）
    - API 请求/响应类型
    - 页面状态类型
    - 组件 Props 类型

4.2 API 层
    - 在 services/ 或 api/ 下新增接口方法
    - 复用现有的请求封装
    - 类型与契约一致

4.3 组件实现
    逐个组件实现，每个组件:
    - Props 类型定义
    - 状态声明
    - 副作用（数据获取）
    - 事件处理
    - 渲染

4.4 状态管理
    - 页面级状态: useState / useReducer
    - 跨组件共享: Context / Store
    - 服务端状态: React Query / SWR
    - 使用现有项目的状态管理方案

4.5 样式
    - 使用项目统一的样式方案
    - 响应式布局（移动端适配）
    - 复用项目的主题变量

实现原则:
- 复用现有组件和 hooks，不重复造轮子
- 错误边界处理（组件级 / 页面级）
- Loading 状态管理
- 无障碍基础支持（语义化 HTML、键盘导航）
```

### Step 5: 自测验证

```
5.1 功能验证
    - [ ] 正常流程: 数据加载 → 展示 → 交互 → 提交 → 反馈
    - [ ] 空数据: 列表为空 / 搜索无结果 / 字段缺失
    - [ ] 加载态: 骨架屏 / Spinner / 按钮禁用
    - [ ] 错误态: 接口报错 / 网络断开 / 超时

5.2 接口对接验证
    - [ ] 请求参数与契约一致
    - [ ] 响应处理与契约一致
    - [ ] 错误码处理完整
    - [ ] Loading 状态正确

5.3 交互验证
    - [ ] 表单校验: 必填/格式/长度/范围
    - [ ] 防重复提交
    - [ ] 操作反馈: 成功提示/失败提示/确认弹窗
    - [ ] 路由跳转: 参数传递/返回/刷新保持

5.4 样式验证
    - [ ] 不同分辨率下布局正常
    - [ ] 长文本/特殊字符不破坏布局
    - [ ] 图片加载失败有兜底
```

### Step 6: 变更清洁

```
- [ ] 无 console.log / debugger 残留
- [ ] 无未使用的 import / 变量
- [ ] 替代的旧组件已删除（不允许新旧共存）
- [ ] 类型变更后所有使用处已更新
- [ ] 样式变更后无遗留的旧样式类
```
