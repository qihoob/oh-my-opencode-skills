# Skill: dev-implementation

**角色**: 开发 (Dev)
**功能**: 功能实现
**触发关键词**: 实现功能、编写代码、开发功能、做功能、写代码

## 产出文档
**路径**: `.opencode/docs/implementation-{feature-name}.md`
**说明**: 实现完成后记录技术细节、实现决策和自测结果，供测试/验收参考

## 依赖文档
**必须读取**: `.opencode/docs/requirement-{feature-name}.md`
- 从中获取验收标准
- 理解功能边界条件
- 确认优先级

**可选读取**:
- `.opencode/docs/handoff-product-to-dev.md` — 交接单（技术方案）
- `.opencode/docs/module-requirement-{module}.md` — 模块需求（接口定义）

## 核心能力

1. **需求理解** — 从需求文档获取验收标准，确认实现范围
2. **方案设计** — 设计技术实现方案，明确数据流和接口
3. **代码实现** — 遵循项目规范，编写清晰可维护的代码
4. **自测验证** — 验证功能、边界和错误处理
5. **Bug 修复** — 从 Bug 分配单获取上下文，先分析根因再最小修复

## 两种执行模式

| 模式 | 触发条件 | 依赖文档 | 修复原则 |
|------|----------|----------|----------|
| **功能实现** | 需求驱动 | requirement-*.md | 按验收标准完整实现 |
| **Bug 修复** | bug-{id}.md 存在 | bug 分配单 + 原始需求 | 先分析根因，最小修复 |

## 执行流程

> 以下为**功能实现**模式的流程。**Bug 修复**模式见文末「Bug 修复专项流程」。

### Step 1: 需求确认

```
操作: 读取 requirement-{feature}.md

确认项:
- [ ] 理解用户故事和业务价值
- [ ] 逐条阅读验收标准（AC1, AC2, ...）
- [ ] 识别边界条件和异常场景
- [ ] 确认优先级（Must/Should/Could Have）
- [ ] 标记有疑问的点 → 需要与产品澄清

如有疑问 → 暂停实现，提示用户先执行 collab-product-to-dev 澄清
```

### Step 2: 方案设计

```
输出: 实现方案（写在实现文档的开头）

设计要点:
1. 确定涉及的文件和模块
2. 设计数据流（输入 → 处理 → 输出）
3. 定义接口（函数签名 / API 端点）
4. 确认技术选型（与项目现有技术栈一致）
5. 评估对现有功能的影响

方案模板:
```
```markdown
## 实现方案

### 涉及文件
| 文件 | 操作 | 说明 |
|------|------|------|
| src/modules/{m}/service.ts | 修改 | 新增业务逻辑 |
| src/modules/{m}/controller.ts | 修改 | 新增接口 |
| src/types/{m}.ts | 修改 | 新增类型定义 |

### 数据流
输入 → [Controller] → [Service] → [Repository] → 数据库
                         ↓
                      [缓存/消息]

### 接口设计
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/{resource} | 创建资源 |

### 关键决策
- {为什么选择方案 A 而不是方案 B}
```

### Step 3: 变更影响分析（变更清洁机制）

```
目的: 确保每次变更是「替换」而非「追加」，避免旧代码残留导致重复逻辑

操作: 在编码之前，先分析这次变更的完整影响面

3.1 识别要改什么
  - 这个功能/修复/优化涉及哪些现有代码？
  - grep 搜索相关函数名、变量名、类型名，定位所有引用点

3.2 识别要删什么（关键步骤）
  - 哪些旧函数/方法会被新实现替代？→ 标记删除
  - 哪些接口签名会变更？→ 标记所有调用方需同步更新
  - 哪些类型定义/常量/配置会废弃？→ 标记清理
  - 哪些 import 会变为无效？→ 标记清理

3.3 识别连带影响
  - 改了这个函数，调用方是否需要同步更新？
  - 改了这个字段/类型，赋值/查询/展示是否都已覆盖？
  - 改了这个状态/枚举，状态机的所有转换路径是否都正确？
  - 改了这个接口，前端调用是否需要同步？

输出: 变更影响清单（写入实现文档的「变更影响」章节）
| 文件 | 操作 | 说明 |
|------|------|------|
| service.ts | 修改+删除 | 新增 createOrderV2()，删除旧的 createOrder() |
| service.ts | 修改 | updateStatus() 签名变更，3处调用方需适配 |
| types.ts | 修改+删除 | 新增 OrderStatusV2，删除废弃的 LegacyOrderStatus |
| controller.ts | 修改 | 适配新接口签名 |

原则:
- 凡是被替代的代码，必须删除，不允许新旧共存
- 凡是签名变更，必须找到所有调用方逐一更新
- 凡是废弃的类型/常量，必须清理，不允许留空不用
```

### Step 4: 获取代码上下文

```
如已执行 dev-context-first → 直接使用上下文报告
如未执行 → 快速获取最小上下文:
1. glob 扫描相关模块目录
2. read 读取核心文件
3. grep 搜索相关代码模式
```

### Step 5: 编码实现

```
编码原则:
- 遵循项目现有代码风格和命名规范
- 保持函数/方法单一职责
- 必要的注释（复杂逻辑、业务规则、TODO）
- 错误处理：不忽略异常，不使用空 catch
- 类型安全：避免 any，使用明确的类型定义

逐条对照验收标准实现:
- AC1: {实现方式} → ✅ 完成
- AC2: {实现方式} → ✅ 完成
- ...

编码时同步执行变更清洁:
- 每完成一个文件的修改，立即执行 Step 3 中标记的删除操作
- 不允许出现「先加新的，稍后删旧的」—— 新旧不能共存
- 修改函数签名后，立即 grep 搜索所有调用方逐一更新
```

### Step 6: 自测验证

```
验证清单:
- [ ] 基本功能: 正常输入 → 正常输出
- [ ] 边界条件: 空值/最大值/最小值/特殊字符
- [ ] 错误处理: 无效输入 → 明确错误提示
- [ ] 异常场景: 网络超时/服务不可用/数据不存在
- [ ] 权限校验: 未登录/无权限/越权访问
```

### Step 7: 代码规范检查

```
快速检查（完整检查由 dev-code-review 执行）:
- [ ] 无 console.log / debugger 残留
- [ ] 无硬编码的敏感信息（密钥/密码/Token）
- [ ] 无未使用的变量和导入
- [ ] 函数复杂度合理（无过深嵌套）
- [ ] 命名清晰有意义
- [ ] 无新旧逻辑共存（变更清洁检查）
- [ ] 无重复实现（grep 搜索确认同一逻辑只有一份实现）
```

## 最小上下文原则

1. **先查文档** — 检查 `.opencode/docs/` 是否有相关需求文档
2. **按需加载** — 只读取与当前功能相关的文档和代码
3. **不重复获取** — 已有的需求文档不再重新分析
4. **聚焦实现** — 不过度设计，按验收标准逐条实现

## Bug 修复专项流程

当存在 `bug-{id}.md` 分配单时，执行以下流程（替代上述功能实现流程）：

### Bug Step 1: 读取 Bug 分配单

```
检查路径: .opencode/docs/bug-{id}.md

必须获取:
- 问题描述（预期 vs 实际）
- 复现步骤
- 根因分析方向
- 关联的代码文件
- 修复约束

如 Bug 分配单不存在 → 说明还未经过 bug-coordinator，先执行 bug-coordinator
```

### Bug Step 2: 根因分析（不要直接改）

```
原则：先搞清楚为什么坏了，再决定怎么修

1. 按分配单中的根因方向逐一排查
2. 读取相关代码，定位具体出错位置
3. 确认根因：
   - 是逻辑错误？（条件判断/计算错误）
   - 是数据问题？（数据格式/缺失/不一致）
   - 是接口问题？（前后端不匹配/超时/异常未处理）
   - 是环境问题？（配置/依赖版本）

输出根因记录（写在实现文档中）:
- 根因：{一句话描述}
- 出错位置：{文件:行号}
- 为什么出错：{原因分析}
```

### Bug Step 3: 最小修复

```
原则：只改该改的，不做额外优化

1. 针对根因，设计最小修复方案
2. 不重构周边代码（除非根因就是结构问题）
3. 不添加新功能（即使发现优化机会）
4. 变更清洁：如果修复替代了旧逻辑，删除旧代码

修复 vs 重写的判断:
- 根因是局部逻辑错误 → 修复（改几行）
- 根因是架构/设计缺陷 → 标注需要重构，先做最小修复止血，建议纳入下迭代
```

### Bug Step 4: 验证修复

```
对照 Bug 分配单中的验证标准:
- [ ] 原 Bug 复现步骤不再出现
- [ ] 修复未影响关联功能
- [ ] 边界场景测试通过
- [ ] 更新 Bug 状态为 "待验证"
```

### Bug Step 5: 产出修复文档

```
更新 implementation-bugfix-{id}.md:

## Bug 修复记录

### Bug 信息
- Bug ID: bug-{module}-{seq}
- 严重程度: P{N}
- 根因: {一句话}

### 修复方案
- 修改文件: {文件列表}
- 修改内容: {简要说明}
- 修复类型: 逻辑修复 / 数据修复 / 接口修复

### 根因分析
- 出错位置: {文件:行号}
- 原因: {详细分析}
- 为什么原有测试没发现: {分析}

### 影响评估
- 影响范围: {仅本模块 / 跨模块}
- 风险: {可能的风险}

### 回归建议
- 必测: {原 Bug 复现场景}
- 建议: {关联功能回归}
```

### Bug Step 6: 自动推荐

```
修复完成 → 推荐 test-executor (回归测试)
回归通过 → Bug 状态更新为 "已关闭"
回归失败 → 退回 Bug Step 2，标注新问题
```
