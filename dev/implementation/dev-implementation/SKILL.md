# Skill: dev-implementation

**角色**: 开发 (Dev)
**功能**: 功能实现
**触发关键词**: 实现功能、编写代码、开发功能、做功能、写代码

## 产出文档
**路径**: `.opencode/docs/implementation-{feature}.md`
**说明**: 实现完成后记录技术细节、实现决策和自测结果，供测试/验收参考

## 依赖文档
**必须读取**: `.opencode/docs/requirement-{feature}.md`
- 从中获取验收标准
- 理解功能边界条件
- 确认优先级

**可选读取**:
- `.opencode/docs/handoff-product-to-dev.md` — 交接单（技术方案）
- `.opencode/docs/module-requirement-{module}.md` — 模块需求（接口定义）

## 核心能力

1. **需求理解** — 从需求文档获取验收标准，确认实现范围
2. **理解现有实现** — 深入阅读目标位置已有代码，理解当前逻辑、风格和模式
3. **方案设计** — 基于现有代码理解，设计契合项目的新方案
4. **代码实现** — 遵循项目规范，编写清晰可维护的代码
5. **自测验证** — 基于实际代码逻辑验证功能、边界和错误处理
6. **Bug 修复** — 从 Bug 分配单获取上下文，先分析根因再最小修复

## 两种执行模式

| 模式 | 触发条件 | 依赖文档 | 修复原则 |
|------|----------|----------|----------|
| **功能实现** | 需求驱动 | requirement-*.md | 按验收标准完整实现 |
| **Bug 修复** | bug-{module}-{seq}.md 存在 | bug 分配单 + 原始需求 | 先分析根因，最小修复 |
| **快速修复** | quickfix 触发 | test-report / code-review | 直接修，不走完整流程 |
| **TDD 模式** | 用户指定 | requirement-*.md | 先写测试，再写实现 |

## TDD 开发模式（可选）

当用户指定"测试驱动" / "TDD" / "先写测试"时，使用此模式替代标准功能实现流程：

### TDD Step 1: 需求确认（同标准 Step 1）

### TDD Step 2: 理解现有实现（同标准 Step 2.1）

### TDD Step 3: 红-绿-重构循环

```
对每个验收标准（AC），执行以下循环：

3.1 红（Red）— 写一个会失败的测试
    - 根据 AC 写测试用例
    - 运行测试 → 确认失败（测试本身没写错）
    - 只写一个测试，不要一次写多个

3.2 绿（Green）— 写最少的代码让测试通过
    - 只写让当前测试通过的最少代码
    - 不做过度设计
    - 允许硬编码、简单实现（后续重构）

3.3 重构（Refactor）— 在测试保护下改善代码
    - 消除 3.2 中的硬编码
    - 提取重复逻辑
    - 改善命名
    - 运行测试 → 确认仍然通过

3.4 下一个 AC → 回到 3.1

TDD 纪律:
- 每次只写一个测试
- 测试失败才写代码
- 代码只为了让测试通过
- 重构时测试必须全程通过
- 不跳过红阶段直接写代码
```

### TDD Step 4: 集成验证

```
所有 AC 的红绿重构循环完成后:
- 运行完整测试套件 → 全部通过
- 验证测试覆盖了所有 AC
- 验证代码与现有代码风格一致
- 产出实现文档（同标准 Step 8）
```

## 执行流程

> 以下为**功能实现**模式的流程。**Bug 修复**模式见文末「Bug 修复专项流程」。

### Step 1: 需求确认

```
操作: 读取 requirement-{feature}.md

确认项:
- [ ] 理解用户故事和业务价值
- [ ] 逐条阅读验收标准（AC1, AC2, ...）
- [ ] 识别边界条件和异常场景
- [ ] 确认优先级（Must/Should/Could Have）
- [ ] 标记有疑问的点 → 需要与产品澄清

如有疑问 → 暂停实现，提示用户先执行 collab-product-to-dev 澄清
```

### Step 2: 理解现有实现 + 方案设计

> 这是整个流程中最关键的步骤。一个合格的开发者在写新代码之前，必须先读懂已有代码。

#### 2.1 理解现有实现（必须执行）

```
目的: 在设计新方案前，先搞清楚现有代码是怎么做的

必须执行的操作:
- read 目标位置已有的代码，理解当前逻辑
- grep 搜索相关函数名、变量名、类型名的使用情况
- 分析新功能与现有逻辑如何衔接

理解清单（逐项回答）:
- [ ] 目标位置已有的逻辑是什么？
      → 读取涉及的 service / controller / model 等文件
      → 梳理当前的数据流：输入从哪来，经过哪些处理，输出到哪去

- [ ] 有没有可以复用的现有逻辑？
      → grep 搜索相似功能的实现（同类型的 CRUD、同模式的校验）
      → 找到可复用的工具函数、中间件、基础类
      → 原则：能复用的不新建，能扩展的不重写

- [ ] 替换现有逻辑时，哪些调用方受影响？
      → grep 搜索函数名、方法名的所有引用
      → 找到所有 import / require 的位置
      → 确认改动是否会破坏下游调用方

- [ ] 现有代码的风格和模式是什么？
      → 观察命名规范（camelCase / snake_case）
      → 观察错误处理模式（try/catch / Result 类型 / 错误码）
      → 观察分层架构（controller → service → repository）
      → 新代码必须与现有风格保持一致

- [ ] 现有代码的错误处理方式是什么？
      → 是否有统一的错误码体系？
      → 是否有统一的异常捕获中间件？
      → 参数校验是在 controller 层还是 service 层？
      → 新代码必须沿用相同的错误处理方式
```

#### 2.2 方案设计（基于现有实现理解）

```
前提: 已完成 2.1，对现有代码有充分理解

输出: 实现方案（写在实现文档的开头）

必须回答的设计问题:
- 新数据从哪来、怎么处理、存到哪？（完整数据流）
- 新接口与现有接口风格是否一致？（URL 命名、参数格式、返回结构）
- 能复用现有逻辑的，不新建（引用 2.1 中找到的可复用部分）
- 改了哪些文件，影响哪些模块（引用 2.1 中的调用方分析）
- 哪些旧代码需要删除（变更清洁）

方案模板:
```
```markdown
## 实现方案

### 现有实现理解
- 当前逻辑: {目标位置的已有逻辑概述}
- 可复用部分: {已有的工具函数/模式/基础设施}
- 受影响调用方: {哪些模块会受此次改动影响}
- 代码风格: {命名规范、分层模式、错误处理方式}

### 涉及文件
| 文件 | 操作 | 说明 |
|------|------|------|
| src/modules/{m}/service.ts | 修改 | 新增业务逻辑 |
| src/modules/{m}/controller.ts | 修改 | 新增接口 |
| src/types/{m}.ts | 修改 | 新增类型定义 |

### 数据流
输入 → [Controller] → [Service] → [Repository] → 数据库
                         ↓
                      [缓存/消息]

### 接口设计
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/{resource} | 创建资源 |

### 关键决策
- {为什么选择方案 A 而不是方案 B}
- {为什么复用 X 而不是新建 Y}
```

### Step 3: 变更影响分析（变更清洁机制）

```
目的: 确保每次变更是「替换」而非「追加」，避免旧代码残留导致重复逻辑

操作: 在编码之前，先分析这次变更的完整影响面

3.1 识别要改什么
  - 这个功能/修复/优化涉及哪些现有代码？
  - grep 搜索相关函数名、变量名、类型名，定位所有引用点

3.2 识别要删什么（关键步骤）
  - 哪些旧函数/方法会被新实现替代？→ 标记删除
  - 哪些接口签名会变更？→ 标记所有调用方需同步更新
  - 哪些类型定义/常量/配置会废弃？→ 标记清理
  - 哪些 import 会变为无效？→ 标记清理

3.3 删除安全验证（防止误删 — 必须执行）

  对 3.2 中每个标记删除的项，执行以下验证：

  a) 全局引用扫描
     对每个待删除的函数名/类型名/常量名，grep 全局搜索（不含 node_modules、.git）：
     - 引用数 = 0 → 安全，可以删除
     - 引用数 > 0 且所有引用都在本次变更范围内 → 安全，同步删除
     - 引用数 > 0 且有引用在本次变更范围外 → ⛔ 不能删除

  b) 不可删除的处理
     对 ⛔ 不能删除的旧代码，不能强删，采取以下策略：
     - 在旧代码上添加 @deprecated 注释，说明替代方案
     - 在新代码的注释中标注旧代码仍存在的原因
     - 将此条目记入变更影响清单的「暂不删除」区域

  c) 签名变更的影响面
     对每个签名变更的函数，grep 搜索所有调用方：
     - 逐个确认调用方是否已适配新签名
     - 未适配的调用方必须纳入本次变更范围
     - 无法纳入的 → 不改签名，改用新函数名 + 旧函数标记 deprecated

3.4 识别连带影响
  - 改了这个函数，调用方是否需要同步更新？
  - 改了这个字段/类型，赋值/查询/展示是否都已覆盖？
  - 改了这个状态/枚举，状态机的所有转换路径是否都正确？
  - 改了这个接口，前端调用是否需要同步？

输出: 变更影响清单（写入实现文档的「变更影响」章节）
| 文件 | 操作 | 说明 | 删除安全 |
|------|------|------|----------|
| service.ts | 修改+删除 | 新增 createOrderV2()，删除旧的 createOrder() | ✅ 引用数=0 |
| service.ts | 修改 | updateStatus() 签名变更，3处调用方需适配 | ✅ 3处均已纳入 |
| types.ts | 修改+删除 | 新增 OrderStatusV2，删除废弃的 LegacyOrderStatus | ✅ 引用数=0 |
| types.ts | 保留 | 旧的 LegacyStatusMap 暂不删除 | ⛔ 仍有2处外部引用 |
| controller.ts | 修改 | 适配新接口签名 | - |

原则:
- 凡是被替代的代码，**验证无外部引用后**才删除
- 凡是签名变更，必须找到所有调用方逐一更新，**遗漏的调用方阻止变更**
- 凡是废弃但仍有外部引用的代码，标记 @deprecated 而非强删
- 不允许新旧逻辑共存（deprecated 标记的过渡期除外）
```

```

如未执行 → 快速获取最小上下文:
```

### Step 5: 编码实现

```
编码原则:
- 遵循项目现有代码风格和命名规范（来自 Step 2.1 的理解）
- 保持函数/方法单一职责
- 必要的注释（复杂逻辑、业务规则、TODO）
- 错误处理：不忽略异常，不使用空 catch，沿用项目统一错误处理方式
- 类型安全：避免 any，使用明确的类型定义

逐条对照验收标准实现:
- AC1: {实现方式} → 完成
- AC2: {实现方式} → 完成
- ...

编码时同步执行变更清洁（按 Step 3 的安全验证结果执行）:
- 删除前再次 grep 确认引用数（防止编码过程中产生了新引用）
- 只删除 Step 3.3 中标记为 ✅ 的项
- Step 3.3 中标记为 ⛔ 的项 → 添加 @deprecated 注释，不删除
- 不允许出现「先加新的，稍后删旧的」—— 新旧不能共存（deprecated 过渡期除外）
- 修改函数签名后，立即 grep 搜索所有调用方逐一更新
- 每次删除后运行相关测试确认无回归

边写边测（推荐）:
- 每完成一个函数/方法 → 立即写一个对应的测试
- 每完成一个接口 → 立即用 curl/Postman 测试
- 发现问题 → 立即修复，不要攒到最后
- 这样自测阶段（Step 6）只需做汇总验证
```

### Step 6: 自测验证（基于代码验证）

```
原则: 自测不是凭空列清单，而是回到实际代码中逐条验证

6.1 对照代码中实际的校验逻辑
  - 读 service 代码，确认每个 if / validate 都测到了
  - 读 controller 代码，确认每个参数校验都测到了
  - 读中间件代码，确认每个拦截条件都触发了

6.2 对照代码中的错误处理路径
  - 读 try/catch 块，确认每个异常路径都测到了
  - 读错误码定义，确认每个错误码都能被正确触发
  - 读兜底逻辑（fallback / default），确认边界行为正确

6.3 对照数据流（端到端验证）
  - 创建操作: 数据是否正确写入（查数据库 / 查列表接口）
  - 修改操作: 只有被修改的字段变了，其他字段保持不变
  - 删除操作: 主数据和关联数据都正确处理（级联删除 or 软删除）
  - 查询操作: 过滤条件、排序、分页都正确

自测验证清单:
- [ ] 基本功能: 正常输入 → 正常输出
- [ ] 边界条件: 空值/最大值/最小值/特殊字符
- [ ] 错误处理: 无效输入 → 明确错误提示（对照代码中的错误码）
- [ ] 异常场景: 网络超时/服务不可用/数据不存在（对照 try/catch 路径）
- [ ] 权限校验: 未登录/无权限/越权访问
- [ ] 数据一致性: 创建/修改/删除后数据状态正确
- [ ] 校验覆盖: 代码中每个 if/validate 都有对应测试场景
```

### Step 7: 代码规范检查

```
快速检查（完整检查由 dev-code-review 执行）:
- [ ] 无 console.log / debugger 残留
- [ ] 无硬编码的敏感信息（密钥/密码/Token）
- [ ] 无未使用的变量和导入
- [ ] 函数复杂度合理（无过深嵌套）
- [ ] 命名清晰有意义
- [ ] 无新旧逻辑共存（变更清洁检查）
- [ ] 无重复实现（grep 搜索确认同一逻辑只有一份实现）
- [ ] 新代码与现有代码风格一致（对照 Step 2.1 的风格分析）
```

## 最小上下文原则

1. **先查文档** — 检查 `.opencode/docs/` 是否有相关需求文档
2. **按需加载** — 只读取与当前功能相关的文档和代码
3. **不重复获取** — 已有的需求文档不再重新分析
4. **聚焦实现** — 不过度设计，按验收标准逐条实现
5. **先读后写** — 任何修改前先读懂目标位置的现有代码

## Bug 修复专项流程

当存在 `bug-{module}-{seq}.md` 分配单时，执行以下流程（替代上述功能实现流程）：

### Bug Step 1: 读取 Bug 分配单

```
检查路径: .opencode/docs/bug-{module}-{seq}.md

必须获取:
- 问题描述（预期 vs 实际）
- 复现步骤
- 根因分析方向
- 关联的代码文件
- 修复约束

如 Bug 分配单不存在 → 说明还未经过 bug-coordinator，先执行 bug-coordinator
```

### Bug Step 2: 根因分析（不要直接改）

```
原则：先搞清楚为什么坏了，再决定怎么修

1. 按分配单中的根因方向逐一排查
2. 读取相关代码，定位具体出错位置
3. 确认根因：
   - 是逻辑错误？（条件判断/计算错误）
   - 是数据问题？（数据格式/缺失/不一致）
   - 是接口问题？（前后端不匹配/超时/异常未处理）
   - 是环境问题？（配置/依赖版本）

输出根因记录（写在实现文档中）:
- 根因：{一句话描述}
- 出错位置：{文件:行号}
- 为什么出错：{原因分析}
```

### Bug Step 3: 最小修复

```
原则：只改该改的，不做额外优化

1. 针对根因，设计最小修复方案
2. 不重构周边代码（除非根因就是结构问题）
3. 不添加新功能（即使发现优化机会）
4. 变更清洁（轻量版，见下方清单）

修复 vs 重写的判断:
- 根因是局部逻辑错误 → 修复（改几行）
- 根因是架构/设计缺陷 → 标注需要重构，先做最小修复止血，建议纳入下迭代
```

### Bug Step 3.5: 变更清洁验证（Bug修复专用）

```
目的：Bug修复是最容易产生残留的场景——急、改得少、容易忘清理。

对本次修复涉及的每个变更，执行以下检查：

3.5.1 修复是否替代了旧逻辑？
  → 否（纯新增逻辑）：跳过清理
  → 是：grep 搜索被替代的旧函数/变量名的全局引用
    - 引用数 = 0 → 删除旧代码
    - 引用数 > 0 且全部在本次修复范围 → 同步删除
    - 引用数 > 0 且有范围外引用 → ⛔ 不删除，在旧代码添加 @deprecated

3.5.2 修复是否改变了函数签名？
  → 否：跳过
  → 是：grep 搜索所有调用方，逐个确认兼容
    - 不兼容的调用方必须纳入本次修复
    - 无法纳入 → 不改签名，改用新函数名

3.5.3 修复后运行回归测试
  → 通过：继续
  → 失败：回退修改，重新评估

输出：变更清洁结果（写入修复文档）
| 待清理项 | 操作 | 全局引用数 | 决定 | 原因 |
|----------|------|-----------|------|------|
| oldValidateLogin() | 删除 | 0 | ✅ 已删除 | 无外部引用 |
| retryCount 参数 | 保留 | 3（范围外） | ⛔ @deprecated | 旧接口仍被调用 |
```

### Bug Step 4: 验证修复

```
对照 Bug 分配单中的验证标准:
- [ ] 原 Bug 复现步骤不再出现
- [ ] 修复未影响关联功能
- [ ] 边界场景测试通过
- [ ] 更新 Bug 状态为 "待验证"
```
