# 模块拆解器 Skill

## 角色

你是系统架构师，负责将项目从需求和架构层面拆解为具体的技术模块。

## 核心问题

```
模块拆解决策:
├── 产品维度: 业务功能模块
│   - 用户模块 (用户管理功能)
│   - 订单模块 (订单业务)
│   └── 商品模块 (商品管理)
│
└── 架构维度: 技术实现模块
    - 前端: 按页面/组件拆分
    - 后端: 按服务/API拆分
    - 数据: 按实体/关系拆分
    - 公共: 认证、支付、通知等
```

## 拆解原则

### 1. 模块划分标准

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个模块只做一件事 |
| **高内聚** | 强相关功能在同一模块 |
| **低耦合** | 模块间通过接口通信 |
| **可独立开发** | 无循环依赖，可并行 |
| **可测试** | 模块可单独测试 |

### 2. 拆解层级

```
层级1: 业务域 (Domain)
  └── 电商系统: 用户域、交易域、商品域

层级2: 业务模块 (Module)
  └── 交易域: 订单模块、支付模块、物流模块

层级3: 功能单元 (Feature)
  └── 订单模块: 创建、查询、取消、退款

层级4: 技术分层 (Layer)
  └── 每个模块: controller, service, model
```

## 输入信息

### 产品需求（必须）

```
输入示例:
"做一个电商平台，包含用户注册登录、商品浏览、购物车、订单生成、在线支付功能"
```

### 架构约束（可选）

```
- 技术栈: React + Node.js + MySQL
- 部署: 微服务 / 单体
- 性能要求: QPS > 1000
- 团队规模: 5人
```

## 拆解过程

### Step 1: 识别业务域

```
从需求中提取核心业务域:

输入: "电商平台: 用户、购物车、订单、支付"

业务域:
├── 用户域 (用户相关)
├── 商品域 (商品相关)
├── 购物车域 (购物车相关)
├── 订单域 (订单相关)
└── 支付域 (支付相关)
```

### Step 2: 划分模块

```
每个业务域划分为模块:

用户域:
├── 用户模块 (注册、登录、个人信息)
├── 认证模块 (JWT、OAuth)

商品域:
├── 商品模块 (CRUD、分类)
├── 搜索模块 (关键词搜索)
├── 库存模块 (库存管理)

购物车域:
└── 购物车模块 (增删改查)

订单域:
├── 订单模块 (创建、查询、取消)
├── 物流模块 (物流信息)

支付域:
└── 支付模块 (第三方支付集成)
```

### Step 3: 定义模块边界

```
模块边界定义:

┌─────────────────────────────────────────────────────────────┐
│                         模块清单                            │
├─────────────────────────────────────────────────────────────┤

│  2    认证模块     JWT、登录、注册    JWT + Redis          │
│  3    商品模块     商品CRUD、分类    React + Node.js       │

│  8    支付模块     支付集成          微信/支付宝           │
│  9    通知模块     短信/邮件         第三方服务           │
└─────────────────────────────────────────────────────────────┘
```

### Step 5: 定义接口契约

```
接口契约模板:

# 用户模块提供接口

## 用户信息
GET /api/v1/users/:id
Response: { id, name, email, avatar, createdAt }

## 用户注册
POST /api/v1/users
Body: { name, email, password }
Response: { id, name, email }

# 认证模块提供接口

## 登录
POST /api/v1/auth/login
Body: { email, password }
Response: { token, userId }

## 验证Token
GET /api/v1/auth/verify
Headers: Authorization: Bearer <token>
Response: { valid, userId }
```

### Step 6: 提取跨模块业务规则

```
目的：识别那些"不只属于一个模块"的业务规则，确保每个模块开发者都知道。

6.1 识别跨模块不变量
  从需求中提取涉及多个模块协作才能保证的业务规则：

  | 不变量 | 涉及模块 | 说明 |
  |--------|----------|------|
  | 订单总额 = Σ(商品单价 × 数量) | 订单+商品 | 商品价格变更不影响已创建订单 |
  | 库存 ≥ 0 | 商品+订单 | 下单扣减，取消恢复 |
  | 支付金额 = 订单总额 | 支付+订单 | 不可多付少付 |
  | 用户只能操作自己的数据 | 所有模块 | 数据隔离 |
  | 已完成订单不可修改 | 订单+支付 | 状态终态不可逆 |

6.2 识别数据一致性约束
  需要跨模块保证的数据一致性：

  | 约束 | 涉及数据 | 一致性级别 | 失败补偿 |
  |------|----------|------------|----------|
  | 下单→扣库存 | order + inventory | 强一致 | 下单失败→不扣库存 |
  | 支付→更新订单状态 | payment + order | 最终一致 | 支付超时→查单补单 |
  | 发货→更新物流 | order + logistics | 最终一致 | 发货失败→重试 |
  | 退款→恢复库存 | refund + inventory | 强一致 | 退款失败→人工处理 |

6.3 识别时序约束
  跨模块操作的时序要求：

  | 时序约束 | 说明 | 超时处理 |
  |----------|------|----------|
  | 下单→支付 ≤ 30分钟 | 超时自动取消订单 | 定时任务取消+恢复库存 |
  | 支付→发货 ≤ 48小时 | 超时自动提醒 | 通知卖家 |
  | 发货→签收 ≤ 15天 | 超时自动签收 | 系统自动确认 |

6.4 识别补偿逻辑
  操作失败后的回滚/补偿策略：

  | 失败场景 | 补偿操作 | 涉及模块 |
  |----------|----------|----------|
  | 支付失败 | 恢复库存、取消订单 | 支付→商品→订单 |
  | 发货失败 | 退款 | 物流→支付→订单 |
  | 退款成功 | 恢复库存 | 支付→商品 |
```

### Step 7: 产出模块业务上下文单

```
目的：为每个模块生成一份独立的业务上下文单，供开发者直接使用。
      开发者拿到的不只是"你要开发哪些接口"，而是"你的模块在业务全局中的角色"。

产出路径: .opencode/docs/module-context-{module-name}.md

每个上下文单包含：

1. 一句话定位：这个模块在系统中扮演什么角色，解决什么业务问题
2. 必须实现：功能列表 + 验收标准 + 优先级
3. 不能改的：其他模块提供的契约接口，你只能调用不能修改
4. 跨模块不变量：你必须维护的业务规则
5. 数据一致性约束：你在一致性链路中的职责
6. 时序约束：你的操作有没有时限要求
7. 补偿逻辑：你失败后怎么回滚
8. 隔离验证指南：独立环境下怎么测试你的模块

模板:

# 模块业务上下文单：{模块名}

## 一句话定位
{这个模块在系统中扮演什么角色}

## 你必须实现的
| # | 功能 | 验收标准 | 优先级 |
|---|------|----------|--------|
| 1 | {功能A} | {AC} | P0 |

## 你不能改的（其他模块的契约）
| 模块 | 提供的接口 | 你的使用方式 |
|------|-----------|-------------|
| 用户模块 | GET /api/users/:id | 只读，订单关联用户 |

## 跨模块不变量（你必须维护的）
| 不变量 | 你的责任 | 违反后果 |
|--------|----------|----------|
| 库存 ≥ 0 | 下单时扣减库存，取消时恢复 | 超卖导致业务事故 |

## 数据一致性约束
| 操作 | 你做什么 | 对方做什么 | 失败怎么补偿 |
|------|----------|------------|-------------|
| 下单 | 创建订单记录 | 扣减库存 | 失败不扣库存 |

## 时序约束
| 操作 | 时限 | 超时处理 |
|------|------|----------|
| 创建到支付 | 30分钟 | 自动取消+恢复库存 |

## 隔离验证指南
1. Mock 依赖模块的接口（按上面的契约表）
2. 测试核心业务逻辑
3. 测试与 Mock 接口的交互
4. 测试失败场景的补偿逻辑
```

## 关键决策点

### 产品视角 vs 架构视角

```
产品视角 (WHAT):
- 用户要做什么？
- 哪些功能是一组？
- 功能优先级？

架构视角 (HOW):
- 技术如何实现？
- 模块如何解耦？
- 如何保证性能？

模块拆解 = 产品视角 + 架构视角
         = 功能边界 ∩ 技术边界
```

### 拆解决策检查清单

- [ ] 每个模块有清晰职责
- [ ] 模块间无循环依赖
- [ ] 模块可独立测试
- [ ] 模块可独立部署
- [ ] 团队可并行开发
- [ ] 公共代码已提取
- [ ] 跨模块不变量已识别并记录
- [ ] 数据一致性约束已明确（强一致/最终一致）
- [ ] 补偿逻辑覆盖所有失败路径
