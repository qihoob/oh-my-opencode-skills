## 契约先行机制（核心防偏措施）

**目的**：解决前后端、后端与数据库、接口与实现"几张皮"的问题。

**原则**：契约是唯一的真相源，前后端、数据库、接口定义都从契约派生。

```

| 方法 | 路径 | 请求体 | 响应体 | 说明 |
|------|------|--------|--------|------|
| POST | /api/{resource} | CreateReq | ResourceResp | 创建 |
| GET | /api/{resource}/:id | - | ResourceResp | 查询 |

### 请求/响应体定义
```typescript
interface CreateOrderReq {

interface OrderResp {
  id: string;
  status: OrderStatus;
  total: number;
  createdAt: string;
}
```

## 状态流转（驱动业务逻辑 + 前端状态展示）

  [pending] --支付成功--> [paid] --发货--> [shipped] --签收--> [completed]
     |                       |
     └--超时取消--> [cancelled]  └--退款--> [refunded]

### 状态转换规则
| 当前状态 | 允许转换 | 触发条件 |
|----------|----------|----------|
| pending | paid / cancelled | 支付成功 / 超时 |
| paid | shipped / refunded | 发货 / 退款 |

## 错误码（驱动前后端错误处理）

| 错误码 | HTTP状态 | 说明 | 前端展示 |
|--------|----------|------|----------|
| ORDER_NOT_FOUND | 404 | 订单不存在 | "订单不存在" |
| ORDER_EXPIRED | 400 | 订单已过期 | "请重新下单" |

## 跨模块业务规则（驱动业务对齐）

### 跨模块不变量
| 不变量 | 涉及模块 | 验证方式 | 违反影响 |
|--------|----------|----------|----------|
| 订单总额 = Σ(商品单价 × 数量) | 订单+商品 | 下单时计算并锁定价格 | 价格不一致 |
| 库存 ≥ 0 | 商品+订单 | 下单扣减+取消恢复 | 超卖 |
| 支付金额 = 订单总额 | 支付+订单 | 支付前校验 | 财务差错 |

### 数据一致性约束
| 操作 | 涉及模块 | 一致性级别 | 实现方式 | 失败补偿 |
|------|----------|-----------|----------|----------|
| 下单→扣库存 | 订单+商品 | 强一致 | 分布式事务/本地事务表 | 下单失败不扣 |
| 支付→更新状态 | 支付+订单 | 最终一致 | 消息队列+幂等 | 超时查单补单 |
| 退款→恢复库存 | 支付+商品 | 强一致 | 同步调用+重试 | 人工补偿 |

### 时序约束
| 流程 | 时限 | 超时处理 | 实现方式 |
|------|------|----------|----------|
| 创建订单→完成支付 | 30分钟 | 自动取消订单+恢复库存 | 定时任务扫描 |
| 支付成功→发货 | 48小时 | 自动提醒卖家 | 延迟消息 |

### 补偿逻辑（Saga 模式）
当跨模块操作链路中某一步失败时：
| 步骤 | 失败场景 | 补偿操作 | 补偿顺序 |
|------|----------|----------|----------|
| 1.扣库存 | 失败 | 直接返回错误 | - |
| 2.创建订单 | 失败 | 恢复库存 | 2→1 |
| 3.发起支付 | 失败 | 取消订单+恢复库存 | 3→2→1 |
| 4.支付成功 | 回调超时 | 定时查单补单 | 重试步骤4 |

## 契约验证清单（开发完成后逐项验证）

- [ ] 数据库字段覆盖了数据模型所有字段
- [ ] 前端类型定义与 API 响应体一致
- [ ] 后端接口实现与 API 定义一致
- [ ] 状态流转覆盖了所有业务场景
- [ ] 错误码前后端共用同一份定义
- [ ] 无契约外的"幽灵字段"（数据库有但模型没定义、接口返回但前端不用）
- [ ] 跨模块不变量在所有场景下都能保证（正常/异常/并发）
- [ ] 数据一致性约束的实现方式已明确（强一致/最终一致）
- [ ] 时序约束的超时处理已实现
- [ ] 补偿逻辑已覆盖所有失败路径
- [ ] 幂等性：重复调用不会产生副作用
```

### 契约驱动的开发顺序

```
正确流程:
  契约定义 → 后端实现（按契约）→ 前端实现（按契约）→ 联调验证（对照契约）

错误流程:
  后端先写 → 前端猜接口 → 联调时发现不一致 → 反复修改 → 几张皮
```

### 2. 接口定义规范

**注意**: 接口定义必须基于 Step 0 产出的契约文档 `contract-{feature}.md`，不允许脱离契约单独定义接口。

**API 接口文档模板** (`docs/API.md`):

```markdown
# API 接口定义

## 模块: [模块名]

### 1. [功能点]

#### 请求
- **URL**: `POST /api/v1/xxx`
- **Headers**: 
  - `Authorization: Bearer <token>`
  - `Content-Type: application/json`
- **Body**:
```json
{
  "field1": "string|required",
  "field2": "number|optional",
  "field3": "boolean|default:true"
}
```

#### 响应
- **成功** `200`:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": "xxx",
    "createdAt": "2024-01-01T00:00:00Z"
  }
}
```
- **错误** `4xx/5xx`:
```json
{
  "code": 1001,
  "message": "错误描述"
}
```

### 数据模型

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 唯一标识 |
| name | string | 是 | 名称 |
| status | number | 否 | 状态 0-禁用 1-启用 |
```

## 工作流程

### Step 1: 初始化模块

```bash
# 1. 创建模块目录
mkdir -p modules/{module}/{frontend,backend,docs,tests}

# 2. 初始化前端
cd modules/{module}/frontend
npm create vite@latest . -- --template react-ts

# 3. 初始化后端
cd modules/{module}/backend
npm init -y
```

### Step 2: 定义接口

```bash
# 1. 后端先定义 API
# 2. 生成 API 文档
# 3. 前端根据文档开发
# 4. 联调测试
```

### Step 3: 开发迭代

```
每日流程:
├── 晨会: 同步进度，阻塞问题
├── 开发: 按模块开发
├── 提交: 代码 code review
├── 测试: 自动化测试
└── 集成: 模块间联调
```

### Step 4: 模块集成

```bash
# 1. 确保各模块测试通过
# 2. 执行集成测试
# 3. 部署预环境
# 4. 产品验收
# 5. 部署生产
```
