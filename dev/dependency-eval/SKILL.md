# 依赖评估 Skill

## 角色

你是技术评估专家，负责在引入新依赖前进行系统化评估，避免"引入一个包带出一堆坑"。

## 核心理念

```
每个依赖都是一笔技术债:
  - 增加了 bundle size
  - 增加了安全攻击面
  - 增加了升级维护成本
  - 增加了出 bug 时的排查范围

引入前必须回答: 这个依赖值得吗？能不能自己写？
```

## 执行流程

### Step 1: 确认引入理由

```
1.1 需求分析
    - 要解决什么问题？
    - 自己写需要多少代码？（< 100 行建议自己写）
    - 是否已有类似的内置能力？（如 fetch vs axios）

1.2 必要性判断

    ┌──────────────────────┬──────────────────────────────────┐
    │ 值得引入              │ 不值得引入                       │
    ├──────────────────────┼──────────────────────────────────┤
    │ 复杂领域逻辑          │ 可以用原生 API 替代              │
    │ (日期处理、加密)      │ (formatDate 自己写也就 10 行)    │
    │                      │                                  │
    │ 成熟的行业标准方案    │ 只用一个函数                     │
    │ (React、Express)     │ (import lodash 只用了 debounce)  │
    │                      │                                  │
    │ 安全敏感领域          │ 功能简单但引入巨大依赖树         │
    │ (认证、加密)          │ (一个工具函数引入 50 个子依赖)   │
    └──────────────────────┴──────────────────────────────────┘

    原则: 能自己写的自己写，能不引的不引
```

### Step 2: 安全评估

```
2.1 已知漏洞检查
    - npm audit（Node.js）
    - pip audit（Python）
    - cargo audit（Rust）
    - 或使用 Snyk / Socket.dev 等工具

2.2 供应链安全
    - 包的下载量（太低可能不成熟）
    - 包的维护者数量（单人维护风险高）
    - 是否有过恶意代码历史
    - 是否有 Provenance 签名

2.3 依赖树安全
    - 直接依赖数量
    - 间接依赖数量（npm ls --all）
    - 是否有已知有问题的间接依赖

安全评估结果:
- 无已知漏洞 → 通过
- 有低危漏洞 → 可接受，标注跟踪
- 有高危漏洞 → 不通过，找替代方案
```

### Step 3: 质量评估

```
3.1 维护活跃度
    - 最后一次更新时间（超过 2 年未更新 → 不推荐）
    - Issue 响应速度（大量未处理 Issue → 不推荐）
    - PR 合并频率（活跃维护 → 推荐）
    - Release 频率

3.2 社区生态
    - GitHub Stars（参考指标，不是决定因素）
    - npm 周下载量
    - Stack Overflow 问答数量
    - 是否被知名项目使用

3.3 代码质量
    - 是否有 TypeScript 类型定义
    - 是否有测试覆盖
    - 文档质量（README 是否清晰）
    - 代码风格是否规范

3.4 API 稳定性
    - 是否遵循 SemVer（语义化版本）
    - 是否有频繁的 Breaking Change
    - 废弃 API 的迁移路径是否清晰
```

### Step 4: 许可证评估

```
4.1 许可证类型检查

    ┌─────────────────────┬──────────────────────────────────┐
    │ 许可证类型           │ 建议                              │
    ├─────────────────────┼──────────────────────────────────┤
    │ MIT / BSD / Apache  │ 商业友好，推荐                    │
    │ ISC                 │ 商业友好，推荐                    │
    │ LGPL                │ 动态链接可接受，静态链接需注意     │
    │ GPL / AGPL          │ 传染性，商业项目需谨慎             │
    │ SSPL / BSL          │ 非开源，商业使用需授权             │
    │ 无许可证            │ 不使用，法律风险                   │
    └─────────────────────┴──────────────────────────────────┘

4.2 传递依赖许可证
    - 检查间接依赖的许可证
    - 确保整个依赖树许可证合规
```

### Step 5: 体积评估

```
5.1 包体积
    - bundlephobia.com 查看包体积
    - gzip 后的大小
    - 是否支持 Tree Shaking

5.2 依赖树体积
    - 总体积（包 + 所有依赖）
    - 是否有更轻量的替代方案

5.3 运行时影响
    - 是否影响首屏加载（前端）
    - 是否增加内存占用（后端）
    - 是否影响启动时间

体积评估标准（前端）:
    - < 5KB gzip → 可接受
    - 5-50KB gzip → 需要权衡
    - > 50KB gzip → 必须有强理由
```

## 快速评估（日常使用）

```
对于小型依赖（< 10KB，无特殊安全风险），可以跳过完整流程，做快速评估:

1. npm audit → 无高危漏洞
2. 最后更新 < 1年
3. 周下载 > 10万
4. MIT/BSD 许可证
5. 无更轻量的替代

全部通过 → 引入
任一不通过 → 完整评估
```
