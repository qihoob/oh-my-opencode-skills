# 模块拆解器 Skill

## 角色

你是系统架构师，负责将项目从需求和架构层面拆解为具体的技术模块。

## 核心问题

```
模块拆解决策:
├── 产品维度: 业务功能模块
│   - 用户模块 (用户管理功能)
│   - 订单模块 (订单业务)
│   └── 商品模块 (商品管理)
│
└── 架构维度: 技术实现模块
    - 前端: 按页面/组件拆分
    - 后端: 按服务/API拆分
    - 数据: 按实体/关系拆分
    - 公共: 认证、支付、通知等
```

## 拆解原则

### 1. 模块划分标准

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个模块只做一件事 |
| **高内聚** | 强相关功能在同一模块 |
| **低耦合** | 模块间通过接口通信 |
| **可独立开发** | 无循环依赖，可并行 |
| **可测试** | 模块可单独测试 |

### 2. 拆解层级

```
层级1: 业务域 (Domain)
  └── 电商系统: 用户域、交易域、商品域

层级2: 业务模块 (Module)
  └── 交易域: 订单模块、支付模块、物流模块

层级3: 功能单元 (Feature)
  └── 订单模块: 创建、查询、取消、退款

层级4: 技术分层 (Layer)
  └── 每个模块: controller, service, model
```

## 输入信息

### 产品需求（必须）

```
输入示例:
"做一个电商平台，包含用户注册登录、商品浏览、购物车、订单生成、在线支付功能"
```

### 架构约束（可选）

```
- 技术栈: React + Node.js + MySQL
- 部署: 微服务 / 单体
- 性能要求: QPS > 1000
- 团队规模: 5人
```

## 拆解过程

### Step 1: 识别业务域

```
从需求中提取核心业务域:

输入: "电商平台: 用户、购物车、订单、支付"

业务域:
├── 用户域 (用户相关)
├── 商品域 (商品相关)
├── 购物车域 (购物车相关)
├── 订单域 (订单相关)
└── 支付域 (支付相关)
```

### Step 2: 划分模块

```
每个业务域划分为模块:

用户域:
├── 用户模块 (注册、登录、个人信息)
├── 认证模块 (JWT、OAuth)

商品域:
├── 商品模块 (CRUD、分类)
├── 搜索模块 (关键词搜索)
├── 库存模块 (库存管理)

购物车域:
└── 购物车模块 (增删改查)

订单域:
├── 订单模块 (创建、查询、取消)
├── 物流模块 (物流信息)

支付域:
└── 支付模块 (第三方支付集成)
```

### Step 3: 定义模块边界

```
模块边界定义:

┌─────────────────────────────────────────────────────────────┐
│                         模块清单                            │
├─────────────────────────────────────────────────────────────┤
│  #    模块名      职责范围           技术栈               │
│  ────────────────────────────────────────────────────────   │
│  1    用户模块     用户CRUD、档案    React + Node.js       │
│  2    认证模块     JWT、登录、注册    JWT + Redis          │
│  3    商品模块     商品CRUD、分类    React + Node.js       │
│  4    搜索模块     全文检索          Elasticsearch         │
│  5    库存模块     库存管理          Node.js + 事务        │
│  6    购物车模块   购物车操作        React + Node.js       │
│  7    订单模块     订单生命周期      Node.js + 状态机      │
│  8    支付模块     支付集成          微信/支付宝           │
│  9    通知模块     短信/邮件         第三方服务           │
└─────────────────────────────────────────────────────────────┘
```

### Step 4: 分析依赖关系

```
模块依赖矩阵:

调用\提供   用户 认证 商品 库存 购物车 订单 支付 通知
用户模块     -   ✓    -    -     -     -    -    -
认证模块     -   -    -    -     -     -    -    -
商品模块     -   -    -    ✓     -     -    -    -
库存模块     -   -    -    -     -     ✓    -    -
购物车模块   -   -    ✓   ✓     -     -    -    -
订单模块     ✓   -    -   ✓     ✓     -    ✓    ✓
支付模块     -   -    -    -     -     ✓    -    -
通知模块     ✓   -    -    -     ✓     ✓    ✓    -

说明: ✓ 表示该行模块调用该列模块
```

### Step 5: 定义接口契约

```
接口契约模板:

# 用户模块提供接口

## 用户信息
GET /api/v1/users/:id
Response: { id, name, email, avatar, createdAt }

## 用户注册
POST /api/v1/users
Body: { name, email, password }
Response: { id, name, email }

# 认证模块提供接口

## 登录
POST /api/v1/auth/login
Body: { email, password }
Response: { token, userId }

## 验证Token
GET /api/v1/auth/verify
Headers: Authorization: Bearer <token>
Response: { valid, userId }
```

## 关键决策点

### 产品视角 vs 架构视角

```
产品视角 (WHAT):
- 用户要做什么？
- 哪些功能是一组？
- 功能优先级？

架构视角 (HOW):
- 技术如何实现？
- 模块如何解耦？
- 如何保证性能？

模块拆解 = 产品视角 + 架构视角
         = 功能边界 ∩ 技术边界
```
