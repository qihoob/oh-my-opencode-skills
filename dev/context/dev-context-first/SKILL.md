# Skill: dev-context-first

**角色**: 开发 (Dev)
**功能**: 开发前获取最小上下文（含代码行为分析）
**触发关键词**: 给我上下文、先看看代码、先了解下、给我看看、看看结构

## 产出文档

- **上下文报告（可选持久化）**: `.opencode/docs/context-report-{feature}.md`
- 如处于自驱动链路中（chain-executor 执行），必须持久化到文件，以触发下游技能
- 如为单次交互（用户直接说"给我上下文"），可即时输出不写文件

## 依赖文档

按优先级检查：
1. `.opencode/docs/requirement-{feature}.md` — 需求文档（如有）
2. `.opencode/docs/handoff-product-to-dev.md` — 交接单（如有）
3. `.opencode/docs/project-overview.md` — 项目概览（如有）

## 核心原则

**最小上下文原则**：只获取完成任务所需的最低限度信息，避免过早优化和过度设计，快速启动开发。

**行为理解优先**：不只告诉"有哪些文件"，必须理解"这些文件做了什么"。文件结构是手段，代码行为才是目的。

## 复杂度分级

不同任务对应不同的上下文获取策略：

| 复杂度 | 适用场景 | 上下文范围 | 获取深度 |
|--------|----------|------------|----------|
| **小** | 修改样式、修Bug、调整文案 | 单文件或相邻 2-3 个文件 | 只读目标文件 |
| **中** | 新增功能、接口变更、组件开发 | 单个模块（前端页面或后端 Service） | 读模块入口+核心逻辑 |
| **大** | 跨模块功能、架构调整、新模块 | 多模块 + 共享层 | 读模块结构+接口定义+数据流 |

## 执行步骤

### Step 1: 检查需求文档

```
检查路径: .opencode/docs/requirement-{feature}.md
- 如存在 → 读取验收标准章节
- 如不存在 → 向用户确认需求，提示先执行 product-requirement-analysis
```

### Step 2: 评估任务复杂度

```
根据用户描述判断：
- 涉及多少文件？ → 单文件/多文件/跨模块
- 是否需要新增接口？ → 是 → 中/大
- 是否涉及数据库变更？ → 是 → 中/大
- 是否涉及多个模块协作？ → 是 → 大
```

### Step 3: 扫描项目结构

根据复杂度分级执行不同范围的扫描。

**小任务**：
- 定位目标文件路径（glob 搜索）
- 读取目标文件内容

**中任务**：
- 扫描目标模块目录结构
- 读取模块入口文件
- 识别模块内依赖关系

**大任务**：
- 扫描涉及的所有模块目录
- 读取各模块入口和接口定义
- 分析模块间依赖关系
- 检查共享代码层

### Step 4: 定位关键文件

```
优先关注：
- 入口文件: main.ts, index.ts, app.ts
- 配置文件: config.*, .env.*
- 路由定义: router.*, routes.*
- 核心逻辑: service.*, controller.*, model.*
- 类型定义: types.*, interfaces.*, *.d.ts
```

### Step 5: 深入分析代码行为

这一步是上下文获取的核心。不只是识别文件结构，必须理解代码的实际行为。

#### 5.1 理解业务逻辑

对目标模块的核心文件，必须理解：

- **核心数据流**：数据从哪来 → 怎么处理 → 到哪去
- **校验规则**：每个入口（API/表单）的参数校验逻辑
- **状态机**：业务状态的定义和转换规则（如果涉及状态）
- **错误处理**：异常路径和错误响应方式
- **副作用**：操作触发的连带行为（发通知、写日志、更新缓存）

执行方式：
- read 目标模块的 service/controller/model 文件
- 对每个核心函数，理解：入参 → 处理逻辑 → 返回值
- 对每个 API 端点，理解：请求 → 校验 → 处理 → 响应

#### 5.2 理解依赖关系

不只是列出"依赖了什么模块"，而是深入分析：

- **反向依赖**：这个模块被谁调用？改了会影响谁？
- **正向依赖**：这个模块调用谁？对方接口稳定吗？
- **共享数据**：共享数据模型是否被多方修改？是否存在隐式耦合？
- **循环依赖**：有没有循环依赖？如有，标注其位置和影响范围。

#### 5.3 识别实现风险

基于代码分析，识别：

- **技术债**：重复代码、过深的嵌套、过长的函数
- **并发问题**：共享状态、锁、事务边界
- **缺少错误处理**：空值未处理、超时未捕获、异常未兜底
- **硬编码和配置**：硬编码的常量、环境相关值写死在代码中
- **已知标记**：TODO/FIXME/HACK 注释

### Step 6: 输出上下文报告

将 Step 1-5 的分析结果整合为结构化报告。

## 使用场景

### 场景 1: 简单修改（小任务）
```
用户: "修改登录页面的提交按钮样式"

执行:
1. glob 搜索 login 相关文件
2. 定位到 LoginPage.tsx 和 Login.module.scss
3. 读取两个文件
4. 输出精简上下文（只包含这 2 个文件）
```

### 场景 2: 功能实现（中任务）
```
用户: "实现用户头像上传功能"

执行:
1. 检查 requirement-avatar.md
2. 扫描用户模块目录
3. 读取 UserService.ts、UserController.ts、User.ts(model)
4. 检查文件上传相关的工具和中间件
5. 分析上传流程的完整调用链和校验逻辑
6. 输出中等上下文（包含模块结构、关键文件、业务逻辑概要）
```

### 场景 3: Bug 修复（中任务）
```
用户: "修复首页加载慢的问题"

执行:
1. 扫描首页相关组件
2. 读取首页入口文件和数据获取逻辑
3. 分析数据流和 API 调用
4. 识别潜在性能瓶颈（N+1查询、重复渲染、阻塞操作）
5. 输出分析报告（包含关键代码路径和实现风险）
```

### 场景 4: 跨模块功能（大任务）
```
用户: "实现订单到支付的完整流程"

执行:
1. 检查需求文档
2. 扫描订单模块和支付模块
3. 分析两个模块间的接口依赖
4. 检查共享层（支付网关、消息队列）
5. 理解订单状态机、支付回调流程、异常处理策略
6. 识别跨模块事务一致性和并发风险
7. 输出完整上下文（包含多模块结构、关键代码路径、实现风险）
```

## 最小上下文原则

1. **先查文档** — 检查 `.opencode/docs/` 是否有相关需求文档，有则直接使用
2. **按需加载** — 根据复杂度分级决定读取深度，不盲目读取全部代码
3. **增量获取** — 先扫描结构，再按需读取具体文件内容
4. **不重复获取** — 已有的文档内容不再重新分析
5. **行为优先** — 文件结构只是手段，理解代码行为才是目的

## 工具可用
- glob: 扫描文件结构
- grep: 搜索关键词和代码模式
- read: 读取关键文件内容
