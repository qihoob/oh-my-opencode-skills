# Skill: dev-context-first

**角色**: 开发 (Dev)
**功能**: 开发前获取最小上下文
**触发关键词**: 给我上下文、先看看代码、先了解下、给我看看、看看结构

## 产出文档

无固定产出文件。输出为即时上下文报告，供后续技能（dev-implementation、dev-code-review）直接使用。

## 依赖文档

按优先级检查：
1. `.opencode/docs/requirement-{feature}.md` — 需求文档（如有）
2. `.opencode/docs/handoff-product-to-dev.md` — 交接单（如有）
3. `.opencode/docs/project-overview.md` — 项目概览（如有）

## 核心原则

**最小上下文原则**：只获取完成任务所需的最低限度信息，避免过早优化和过度设计，快速启动开发。

## 复杂度分级

不同任务对应不同的上下文获取策略：

| 复杂度 | 适用场景 | 上下文范围 | 获取深度 |
|--------|----------|------------|----------|
| **小** | 修改样式、修Bug、调整文案 | 单文件或相邻 2-3 个文件 | 只读目标文件 |
| **中** | 新增功能、接口变更、组件开发 | 单个模块（前端页面或后端 Service） | 读模块入口+核心逻辑 |
| **大** | 跨模块功能、架构调整、新模块 | 多模块 + 共享层 | 读模块结构+接口定义+数据流 |

## 执行步骤

### Step 1: 检查需求文档

```
检查路径: .opencode/docs/requirement-{feature}.md
- 如存在 → 读取验收标准章节
- 如不存在 → 向用户确认需求，提示先执行 product-requirement-analysis
```

### Step 2: 评估任务复杂度

```
根据用户描述判断：
- 涉及多少文件？ → 单文件/多文件/跨模块
- 是否需要新增接口？ → 是 → 中/大
- 是否涉及数据库变更？ → 是 → 中/大
- 是否涉及多个模块协作？ → 是 → 大
```

### Step 3: 扫描项目结构

根据复杂度分级执行不同范围的扫描：

**小任务**：
- 定位目标文件路径（glob 搜索）
- 读取目标文件内容

**中任务**：
- 扫描目标模块目录结构
- 读取模块入口文件
- 识别模块内依赖关系

**大任务**：
- 扫描涉及的所有模块目录
- 读取各模块入口和接口定义
- 分析模块间依赖关系
- 检查共享代码层

### Step 4: 定位关键文件

```
优先关注：
- 入口文件: main.ts, index.ts, app.ts
- 配置文件: config.*, .env.*
- 路由定义: router.*, routes.*
- 核心逻辑: service.*, controller.*, model.*
- 类型定义: types.*, interfaces.*, *.d.ts
```

### Step 5: 分析代码模式

```
识别项目中已有的代码模式：
- 代码组织方式（按功能/按类型）
- 命名规范（文件名、变量名、函数名）
- 框架使用方式（React Hooks 模式、状态管理方案）
- API 调用模式（请求封装、错误处理方式）
```

### Step 6: 输出上下文报告

## 输出格式

```markdown
## 开发上下文报告：{任务描述}

### 任务信息
- **复杂度**: 小/中/大
- **涉及文件数**: {N}
- **需求来源**: requirement-{feature}.md / 用户口述

### 需求要点
- {从需求文档提取的验收标准，每条一行}
- {如无需求文档，标注"需先执行需求分析"}

### 项目结构概览
```
{相关的目录树，只展示与当前任务相关的部分}
```

### 关键文件清单

| 文件 | 作用 | 需要读取 | 需要修改 |
|------|------|----------|----------|
| `src/modules/{module}/service.ts` | 业务逻辑 | ✅ | ✅ |
| `src/modules/{module}/controller.ts` | 接口控制 | ✅ | ❌ |
| `src/types/{module}.ts` | 类型定义 | ✅ | ✅ |

### 代码模式
- **组织方式**: {按功能模块 / 按技术层}
- **状态管理**: {useState / Zustand / Redux}

- **命名规范**: {小驼峰 / 短横线}

### 依赖关系
- **内部依赖**: {依赖的模块/工具}

### 风险提示
```

## 使用场景

### 场景 1: 简单修改（小任务）
```
用户: "修改登录页面的提交按钮样式"

执行:
1. glob 搜索 login 相关文件
2. 定位到 LoginPage.tsx 和 Login.module.scss
3. 读取两个文件
4. 输出精简上下文（只包含这 2 个文件）
```

### 场景 2: 功能实现（中任务）
```
用户: "实现用户头像上传功能"

执行:
1. 检查 requirement-avatar.md
2. 扫描用户模块目录
3. 读取 UserService.ts、UserController.ts、User.ts(model)
4. 检查文件上传相关的工具和中间件
5. 输出中等上下文（包含模块结构和关键文件）
```

### 场景 3: Bug 修复（中任务）
```
用户: "修复首页加载慢的问题"

执行:
1. 扫描首页相关组件
2. 读取首页入口文件和数据获取逻辑
3. 分析数据流和 API 调用
4. 识别潜在性能瓶颈
5. 输出分析报告
```

### 场景 4: 跨模块功能（大任务）
```
用户: "实现订单到支付的完整流程"

执行:
1. 检查需求文档
2. 扫描订单模块和支付模块
3. 分析两个模块间的接口依赖
4. 检查共享层（支付网关、消息队列）
5. 输出完整上下文（包含多模块结构和数据流图）
```

## 最小上下文原则

1. **先查文档** — 检查 `.opencode/docs/` 是否有相关需求文档，有则直接使用
2. **按需加载** — 根据复杂度分级决定读取深度，不盲目读取全部代码
3. **增量获取** — 先扫描结构，再按需读取具体文件内容
4. **不重复获取** — 已有的文档内容不再重新分析

## 工具可用
- glob: 扫描文件结构
- grep: 搜索关键词和代码模式
- read: 读取关键文件内容
