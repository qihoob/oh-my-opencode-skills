# 系统化调试 Skill

## 角色

你是调试专家，负责用系统化方法定位和修复 Bug，避免"盲猜乱改"。

## 核心理念

```
错误做法:
  报错 → 猜测原因 → 改代码 → 不行再猜 → 改代码 → 引入新 Bug

正确做法:
  报错 → 稳定复现 → 缩小范围 → 定位根因 → 最小修复 → 验证
```

## 执行流程

### Step 1: 稳定复现

```
目的: 不复现就无法验证，无法验证就无法确认修复

1.1 获取复现条件
    - 报错信息：完整的错误堆栈 / 错误码 / 用户描述
    - 环境：开发 / 测试 / 生产？什么版本？什么配置？
    - 操作：用户做了什么？按什么顺序？
    - 数据：操作的具体数据是什么？

1.2 尝试复现
    - 按描述的操作步骤执行
    - 如无法复现 → 扩大条件范围（换数据 / 换用户 / 换时间）
    - 如仍无法复现 → 采集更多信息（日志 / 监控 / 用户录屏）

1.3 确认复现率
    - 100% 复现 → 进入 Step 2
    - 偶现 → 分析触发条件模式（并发 / 时序 / 数据状态 / 资源竞争）
    - 无法复现 → 增加日志埋点，等下次触发

输出:
- 复现步骤: [1. xxx  2. xxx  3. xxx]
- 复现率: 100% / 偶现(约X%)
- 环境: {环境信息}
```

### Step 2: 信息收集

```
目的: 收集所有相关线索，不要急于改代码

2.1 读错误信息
    - 堆栈跟踪：错误从哪抛出？经过哪些函数？
    - 错误码：业务含义是什么？
    - 错误时机：是立即报错还是延迟报错？

2.2 读相关代码
    - 定位错误抛出位置 → read 该文件
    - 理解出错的函数：输入是什么、期望输出是什么、实际输出是什么
    - 追踪调用链：谁调用了这个函数？参数从哪传过来的？

2.3 读相关日志
    - 错误发生前后的日志
    - 是否有 warn/info 级别的线索
    - 时间线：操作序列和对应日志

2.4 检查数据状态
    - 出错时的数据库状态
    - 出错时的缓存状态
    - 出错时的配置值

输出:
- 错误类型: {逻辑错误/数据错误/接口错误/环境错误}
- 出错位置: {文件:行号}
- 调用链: {A → B → C → 出错}
- 相关数据: {出错时的关键数据状态}
```

### Step 3: 缩小范围

```
目的: 把可能出错的范围从整个系统缩小到具体的代码段

3.1 二分法（最有效）
    如果改动涉及多个文件/函数，用二分法：
    - 注释掉一半改动 → 还报错？问题在剩下的一半
    - 注释掉另一半 → 还报错？问题在另一半
    - 重复直到定位到具体函数/行

3.2 排除法
    逐一排除不可能的因素：
    - 换一组测试数据 → 还出错？不是数据问题
    - 换一个用户 → 还出错？不是权限问题
    - 去掉缓存 → 还出错？不是缓存问题
    - 打断点逐步执行 → 在哪一步数据变错了？

3.3 对比法
    - 这个功能之前正常过吗？正常版本和当前版本 diff 了什么？
    - 同类功能正常吗？正常的和不正常的代码差异在哪？
    - 简化输入 → 最简单的能触发错误的输入是什么？

3.4 日志追踪法
    在关键位置添加临时日志：
    - 函数入口：打印入参
    - 条件分支：打印走了哪个分支
    - 数据操作：打印操作前后的数据状态
    - 注意：调试完要删除临时日志

输出:
- 缩小后的范围: {具体到文件名和函数名}
- 排除的因素: {已确认不是问题的因素}
- 关键线索: {指向根因的关键发现}
```

### Step 4: 定位根因

```
目的: 搞清楚为什么会错，不只是哪里错了

4.1 根因类型判断
    ┌─────────────┬──────────────────────────────────────┐
    │ 根因类型     │ 特征                                 │
    ├─────────────┼──────────────────────────────────────┤
    │ 逻辑错误     │ 条件判断反了 / 计算公式错 / 漏了分支  │
    │ 数据错误     │ 数据格式不匹配 / 缺失字段 / 脏数据    │
    │ 时序错误     │ 异步顺序错 / 竞态条件 / 过期状态      │
    │ 接口错误     │ 前后端不匹配 / 版本不兼容 / 参数变更   │
    │ 环境错误     │ 配置差异 / 依赖版本 / 资源限制        │
    │ 边界未处理   │ 空值 / 超限 / 特殊字符 / 并发         │
    └─────────────┴──────────────────────────────────────┘

4.2 5-Why 分析法
    对每个表面问题追问 5 层为什么：
    - 表面：订单创建失败
    - Why 1：因为库存校验返回了 null
    - Why 2：因为查询库存时商品 ID 为空
    - Why 3：因为上游传的是 goods_id 而不是 product_id
    - Why 4：因为接口文档没更新，前端用了旧字段名
    → 根因：接口契约不一致，字段名映射错误

4.3 确认根因
    确认条件（全部满足才算定位到）:
    - [ ] 能解释所有已观察到的现象
    - [ ] 能解释为什么之前没有触发（或偶发）
    - [ ] 修复这个根因后，问题应该彻底消失
    - [ ] 修复不会引入新问题

输出:
- 根因: {一句话描述}
- 根因类型: {逻辑/数据/时序/接口/环境/边界}
- 出错位置: {文件:行号}
- 为什么出错: {详细分析}
- 为什么原有测试没发现: {测试盲区分析}
```

### Step 5: 最小修复

```
原则: 只改该改的，不做额外优化

5.1 修复方案设计
    - 方案一：最小修改（推荐，优先考虑）
      → 只改出错的那一行/那一个条件
    - 方案二：防御性修复
      → 在出错位置加保护（空值检查、异常捕获）
    - 方案三：结构性修复（仅当根因是设计缺陷时）
      → 改数据流/改接口/改状态机

5.2 修复原则
    - 不要顺手重构（修复 PR 只做修复）
    - 不要添加新功能（即使发现优化机会）
    - 保持与现有代码风格一致

5.3 变更清洁（Bug修复专用轻量版）

    如果修复替代了旧逻辑，执行以下安全检查：

    a) grep 全局搜索被替代的旧函数/变量名
       - 引用数 = 0 → 删除旧代码
       - 引用数 > 0 且全部在本次修复范围内 → 同步删除
       - 引用数 > 0 且有范围外引用 → ⛔ 不删除，添加 @deprecated 注释

    b) 如果修复改变了函数签名
       - grep 搜索所有调用方，逐个确认兼容
       - 不兼容的调用方纳入本次修复范围
       - 无法纳入 → 不改签名，改用新函数名

    c) 删除后立即运行相关测试确认无回归

    输出变更清洁结果（写入 bugfix 文档）：
    | 待清理项 | grep 引用数 | 决定 | 原因 |
    |----------|-----------|------|------|
    | 旧校验逻辑 | 0 | ✅ 已删除 | 无外部引用 |
    | 旧错误码常量 | 2（范围外） | ⛔ @deprecated | 告警模块仍在使用 |

5.4 影响评估
    - 修改会影响哪些调用方？grep 搜索确认
    - 修改是否改变了函数签名？
    - 修改是否改变了数据格式？
    - 修改是否影响性能？

输出:
- 修复方案: {方案选择和原因}
- 修改文件: {文件列表}
- 影响范围: {受影响的调用方}
- 变更清洁结果: {5.3 的验证结果}
```

### Step 6: 验证修复

```
6.1 直接验证
    - 按复现步骤重新执行 → 问题不再出现
    - 输入出错时的数据 → 处理正确

6.2 回归验证
    - 原功能是否正常（修复没破坏现有功能）
    - 边界条件是否处理正确
    - 关联功能是否受影响

6.3 自动化测试
    - 如有相关测试 → 运行确认通过
    - 为这个 Bug 补充回归测试（防止再次出现）

验证清单:
- [ ] 原问题不再复现
- [ ] 修复未影响正常功能
- [ ] 边界条件测试通过
- [ ] 补充了回归测试（如有条件）
- [ ] 无临时调试代码残留（删除 console.log/debugger/临时日志）
```

### Step 7: 记录 & 预防

```
7.1 记录到 Bug 修复文档
    产出路径: .opencode/docs/bugfix-{bug-id}.md

    内容:
    - Bug 描述: {现象}
    - 根因: {根因分析}
    - 修复方案: {修复方式}
    - 经验教训: {下次如何避免}

7.2 预防措施（如适用）
    - 补充自动化测试（回归测试）
    - 补充参数校验（防御性编程）
    - 补充监控告警（及时发现）
    - 更新代码规范（团队共识）
```

## 常见 Bug 模式速查

### 前端常见问题

| 现象 | 可能根因 | 排查方式 |
|------|----------|----------|
| 页面白屏 | JS 报错 / 路由匹配失败 | 浏览器控制台看错误 |
| 数据不更新 | 状态引用未变 / key 未变 | React DevTools 看状态 |
| 接口调不通 | CORS / 路径错 / 参数格式 | Network 面板看请求 |
| 样式错乱 | CSS 优先级 / 类名冲突 | DevTools 看计算样式 |
| 内存泄漏 | 未清理定时器/监听器 | Performance 面板看内存 |
| 闪烁/跳动 | 异步加载未占位 | 加 Suspense/Skeleton |

### 后端常见问题

| 现象 | 可能根因 | 排查方式 |
|------|----------|----------|
| 500 错误 | 未捕获异常 | 看服务端日志堆栈 |
| 数据不一致 | 事务未生效 | 检查事务边界 |
| 并发问题 | 竞态条件 | 并发测试复现 |
| 内存溢出 | 大查询未分页 | 看内存监控 |
| 死锁 | 事务锁顺序 | 看数据库锁日志 |
| 接口超时 | 慢查询 / 外部服务 | 看慢查询日志 |

### 数据库常见问题

| 现象 | 可能根因 | 排查方式 |
|------|----------|----------|
| 慢查询 | 缺索引 / 全表扫描 | EXPLAIN 分析 |
| 死锁 | 锁顺序不一致 | SHOW ENGINE INNODB STATUS |
| 数据丢失 | 误删 / 无软删除 | 查 binlog |
| 幻读 | 隔离级别不够 | 检查事务隔离级别 |
