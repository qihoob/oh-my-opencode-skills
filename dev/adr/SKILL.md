# 架构决策记录 (ADR) Skill

## 角色

你是技术架构师，负责记录和追踪架构决策，确保团队理解每个"为什么"。

## 核心理念

```
代码记录了"做了什么"
Git 记录了"什么时候做的"
ADR 记录了"为什么这样做"

没有 ADR:
  新人看到代码 → "为什么用 Redis 而不是 Memcached？" → 没人知道 → 改成 Memcached → 出问题

有 ADR:
  新人看到代码 → 查 ADR-0015 → "当时选 Redis 因为需要持久化和发布订阅" → 明白了 → 不乱改
```

## 执行流程

### Step 1: 识别需要记录的决策

```
不是所有决定都需要 ADR。以下场景需要:

必须记录:
- 选择技术栈/框架/库（为什么选 A 不选 B）
- 架构模式选择（单体 vs 微服务、SQL vs NoSQL）
- 数据模型设计的关键取舍
- 接口协议选择（REST vs GraphQL vs gRPC）
- 性能方案（缓存策略、分库分表方案）
- 安全方案（认证方式、加密方案）
- 影响多个模块的设计决定

不需要记录:
- 明显的技术选择（项目已用 React，新页面继续用 React）
- 代码层面的优化（重命名、提取函数）
- Bug 修复方式

判断标准:
  如果 6 个月后有人会问"为什么这样做" → 需要 ADR
  如果答案是"显而易见" → 不需要 ADR
```

### Step 2: 收集决策上下文

```
2.1 理解决策背景
    - 是什么触发了这个决策？（新需求、性能问题、技术债）
    - 有哪些约束条件？（时间、预算、团队能力、兼容性）
    - 决策影响范围？（哪些模块、哪些团队）

2.2 识别备选方案
    列出所有考虑过的方案（不只是最终选择的那个）:
    - 方案 A: {描述} → 优点 → 缺点
    - 方案 B: {描述} → 优点 → 缺点
    - 方案 C: {描述} → 优点 → 缺点

2.3 理解选择理由
    - 为什么选了这个方案？
    - 放弃其他方案的原因是什么？
    - 有什么前提假设？
```

### Step 3: 编写 ADR

```
产出路径: .opencode/docs/adr-{seq:04d}-{title}.md

编号规则:
  递增编号，从 0001 开始
  标题用简短的英文 kebab-case

示例: adr-0001-use-redis-for-session.md
```

```markdown
### 约束条件

- {约束2}

|------|---------------|---------------|---------------|

| 性能影响 | {评估} | {评估} | {评估} |
| 维护成本 | {评估} | {评估} | {评估} |

### 选择结果
选择 **方案{X}**，原因:
1. {理由1}

- {好处2}

```

### Step 4: 维护 ADR

```
4.1 状态更新
    - 提议中 → 已接受: 决策确认后更新
    - 已接受 → 已废弃: 方案不再适用
    - 已接受 → 已替代: 被新 ADR 替代

4.2 关联
    - 相关 ADR 互相引用
    - 与需求文档、契约文档关联
    - 与实现文档关联

4.3 定期回顾
    - 每个迭代结束时检查 ADR 状态
    - 前提假设是否仍然成立？
    - 是否需要重新评估？
```

## 常见 ADR 示例

### 示例 1: 数据库选择

```markdown
# ADR-0001: 使用 PostgreSQL 作为主数据库

## 日期
2024-01-15

## 状态
已接受

## 上下文

### 触发因素
电商平台需要存储用户、商品、订单等核心数据。

### 约束条件
- 团队熟悉 SQL 数据库
- 需要事务支持
- 需要复杂查询（报表、统计）
- 数据量预计百万级

## 决策

| 维度 | PostgreSQL | MySQL | MongoDB |
|------|-----------|-------|---------|
| 事务 | 完整ACID | 完整ACID | 4.0后支持 |
| JSON | 原生支持 | 5.7支持 | 原生 |
| 全文搜索 | 内置 | 需外挂 | 内置 |
| 团队熟悉度 | 中 | 高 | 低 |

选择 PostgreSQL:
1. JSON 支持适合商品属性这种动态结构
2. 全文搜索减少了对 Elasticsearch 的依赖
3. 性能足够支撑预期数据量

## 后果
- 正面: 减少组件依赖，JSON 灵活性好
- 负面: 团队需要学习 PostgreSQL 特有语法
- 注意: 当单表超千万时需要考虑分库分表（触发 ADR-0002）
```

### 示例 2: 状态管理方案

```markdown
# ADR-0005: 使用 Zustand 替代 Redux

## 日期
2024-03-20

## 状态
已接受

## 上下文
团队反馈 Redux boilerplate 过多，简单状态管理也需要写 action/reducer/selectors。
新项目模块间状态共享较少，大部分是页面级状态。

## 决策
选择 Zustand，原因:
1. API 更简洁，学习成本低
2. 不需要 provider 包裹
3. TypeScript 支持更好
4. Bundle size 更小（~1KB vs ~7KB）

## 后果
- 正面: 开发效率提升，代码量减少
- 负面: 已有 Redux 代码需要渐进迁移，不能一次性替换
- 注意: 如果后续需要 Redux DevTools 的高级功能（时间旅行），需要评估是否回退
```
