# Skill: global-project-analysis

**角色**: 产品 (Product)
**功能**: 项目全局分析 + 项目健康检查
**触发关键词**: 项目分析、模块划分、全局梳理、了解项目结构、项目体检、健康检查、项目偏移
**版本**: 2.0 (新增健康检查模式)

## 两种使用模式

| 模式 | 触发场景 | 目的 |
|------|----------|------|
| **初始分析** | 新项目、新迭代开始 | 建立项目认知地图 |
| **健康检查** | 提测前、验收前、迭代结束、定期检查 | 检测项目偏移度 |

---

## 模式一：初始分析

### 产出文档
**路径**: `.opencode/docs/project-overview.md`

### 核心能力

1. **项目扫描**
   - 识别项目类型 (Web/App/API/CLI)
   - 分析技术栈 (框架、语言、数据库)
   - 发现目录结构和组织方式

2. **模块识别**
   - 按功能划分模块边界
   - 识别核心模块和辅助模块
   - 分析模块间依赖关系

3. **关键文件定位**
   - 入口文件
   - 配置文件
   - 核心业务逻辑文件

### 输入要求

用户需提供：
- 项目名称/类型（可选）
- 分析重点（可选，如：只关注后端）

### 输出格式

```markdown
# 项目概览：{项目名称}

## 技术栈

| 层级 | 技术 | 版本 |
|------|------|------|
| 前端 | React/Vue/Angular | x.x |
| 后端 | Node.js/Python/Go | x.x |
| 数据库 | MySQL/MongoDB | x.x |

## 目录结构

```
project/
├── src/           # 源代码
│   ├── modules/   # 业务模块
│   ├── utils/     # 工具函数
│   └── main.ts    # 入口
├── tests/         # 测试代码
├── docs/          # 文档
└── config/        # 配置文件
```

## 模块划分

### 模块 A: {名称}
- **职责**: {功能描述}
- **位置**: `src/modules/a/`
- **核心文件**: `service.ts`, `controller.ts`

### 模块 B: {名称}
- **职责**: {功能描述}
- **位置**: `src/modules/b/`

## 关键文件

| 文件 | 作用 |
|------|------|
| `src/main.ts` | 应用入口 |
| `src/config.ts` | 配置管理 |

## 后续技能引用
<!-- 需求分析技能请参考此文档了解项目背景 -->
```

## 模式二：项目健康检查

### 触发时机

```
自动触发点:
- 提测前（collab-dev-to-qa 前置检查）
- 验收前（collab-acceptance-review 前置检查）
- 迭代结束（iteration-closure 前置检查）

手动触发:
- "项目体检" "健康检查" "看看项目偏移" "项目整体状态"
```

### 产出文档
**路径**: `.opencode/docs/health-check-{date}.md`

### 执行步骤

#### Step 1: 文档链条完整性

```
扫描 .opencode/docs/ 下所有文档，建立文档关系图：

  requirement-*.md → contract-*.md → implementation-*.md → test-cases-*.md → test-report-*.md

检查项:
- [ ] 每个需求是否有对应的实现文档？
- [ ] 每个实现文档是否引用了需求文档？
- [ ] 涉及跨层的功能是否有契约文档？
- [ ] 每个验收标准是否有对应的测试用例？
- [ ] 文档链条是否连续（无断裂）？

输出: 断链清单
| 需求文档 | 契约 | 实现 | 测试用例 | 测试报告 | 状态 |
|----------|------|------|----------|----------|------|
| login ✅ | - | ✅ | ✅ | ✅ | 完整 |
| order ✅ | ✅ | ✅ | ✅ | ❌ | 🔴 报告缺失 |
| pay ✅ | ❌ | ✅ | ❌ | ❌ | 🔴 跨层无契约 |
```

#### Step 2: 需求-实现对齐度

```
对每个已实现功能，对比需求文档和实现文档：

检查项:
- [ ] 验收标准全部有对应的实现说明？
- [ ] 需求中的优先级与实现优先级是否一致？
- [ ] 需求变更后实现是否同步更新？
- [ ] 是否有实现了但需求中没有的功能（镀金）？
- [ ] 是否有需求中有但未实现的功能（遗漏）？

输出: 偏移清单
| 功能 | 需求AC数 | 已实现 | 遗漏 | 镀金 | 偏移度 |
|------|----------|--------|------|------|--------|
| 登录 | 8 | 7 | 1 | 0 | 低 |
| 订单 | 12 | 8 | 2 | 2 | 🔴 高 |
```

#### Step 3: 契约一致性（如有契约文档）

```
对每个 contract-*.md，检查三层对齐：

| 契约 | 数据库 | 后端 | 前端 | 一致性 |
|------|--------|------|------|--------|
| order | ✅ | ⚠️ 缺2个错误码 | ❌ 字段名不一致 | 🔴 |
| user | ✅ | ✅ | ✅ | 🟢 |

具体检查:
- [ ] API 路径/方法三层一致？
- [ ] 数据模型字段数据库全覆盖？
- [ ] 前端类型定义与 API 响应一致？
- [ ] 错误码前后端共用？
- [ ] 状态流转实现与契约一致？
```

#### Step 4: 代码健康度扫描

```
快速扫描代码层面的健康指标：

| 指标 | 检查方式 | 健康标准 |
|------|----------|----------|
| TODO 数量 | grep "TODO" | < 10 / 无 P0 TODO |
| 重复代码 | 同名函数/相似逻辑 | 无跨模块重复 |
| 废弃代码 | 未被引用的函数/类 | 无死代码 |
| any 使用 | grep ": any" | < 5 处 |
| console.log | grep "console.log" | 0 处 |
| 超长函数 | 函数行数 > 100 | 0 个 |
| 超深嵌套 | 嵌套 > 4 层 | 0 个 |
| 未使用导入 | lint 检查 | 0 处 |
```

#### Step 5: 架构偏移检测

```
对比项目概览文档与当前实际结构：

- [ ] 模块边界是否被破坏？（模块A的代码出现在模块B的目录中）
- [ ] 依赖方向是否正确？（无底层模块依赖高层模块）
- [ ] 共享代码是否都在 shared/ 中？（无模块间直接引用）
- [ ] 新增文件是否在预期的目录结构中？

输出: 架构偏移报告
| 偏移类型 | 位置 | 说明 |
|----------|------|------|
| 边界违反 | module-a 引用 module-b 内部文件 | 应通过接口调用 |
| 位置异常 | utils/ 下出现业务逻辑 | 应移入对应模块 |
```

#### Step 6: 输出项目健康报告

### 健康报告模板

```markdown
# 项目健康报告

**检查日期**: {YYYY-MM-DD}
**检查范围**: 全项目
**基准文档**: project-overview.md

## 健康评分

| 维度 | 得分 | 状态 |
|------|------|------|
| 文档链完整性 | {N}/100 | 🟢/🟡/🔴 |
| 需求实现度 | {N}/100 | 🟢/🟡/🔴 |
| 契约一致性 | {N}/100 | 🟢/🟡/🔴 |
| 代码健康度 | {N}/100 | 🟢/🟡/🔴 |
| 架构一致性 | {N}/100 | 🟢/🟡/🔴 |
| **综合健康度** | **{N}/100** | **🟢/🟡/🔴** |

> 🟢 ≥ 80  🟡 60-79  🔴 < 60

## 关键问题（Top 5）

| # | 问题 | 维度 | 严重程度 | 建议动作 |
|---|------|------|----------|----------|
| 1 | 订单模块无契约文档，前后端字段不一致 | 契约一致性 | 🔴 严重 | 立即补充 contract-order.md |
| 2 | 支付功能 2 个 AC 未实现 | 需求实现度 | 🔴 严重 | 确认是否需求变更或遗漏 |
| 3 | module-a 直接引用 module-b 内部 service | 架构一致性 | 🟡 中等 | 重构为接口调用 |
| 4 | 12 处 TODO，其中 3 处标记 P0 | 代码健康度 | 🟡 中等 | 排期清理 |
| 5 | test-cases-order.md 过期 30 天 | 文档链完整性 | 🟡 中等 | 更新测试用例 |

## 趋势（如有历史报告）

| 维度 | 上次 | 本次 | 变化 |

## 行动建议（按优先级）

1. {动作} → 负责人: {name} → 截止: {date}

### 本迭代内修复
2. {动作} → 负责人: {name} → 截止: {date}

```

## 最小上下文原则

1. **增量扫描** - 先扫描根目录，再按需深入
2. **按需读取** - 只读取关键文件，不加载全部代码
3. **结构化输出** - 便于后续技能快速定位
4. **历史对比** - 如有历史健康报告，对比变化趋势

## 工具可用
- glob: 扫描文件结构
- read: 读取关键文件
- grep: 搜索代码模式（TODO、any、console.log等）
- write: 写入健康报告
