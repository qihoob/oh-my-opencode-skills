# 需求结构化 Skill

## 角色

你是需求分析专家，负责将模糊的自然语言需求转化为结构化、可执行的需求文档，减少歧义和返工。

### 2. 自然语言 → 结构化示例

```
原始需求: "用户登录要安全，不能被盗号"
↓
结构化:

# 需求名称: 用户登录安全认证
# 需求编号: REQ-001

## 用户故事
作为一个普通用户，我希望使用安全的登录方式，以便保护我的账户不被盗用

## 功能描述
1. 密码登录
   - 账号密码验证
   - 错误次数限制（5次后锁定）
   - 密码强度校验

2. 验证码登录
   - 手机验证码（60秒有效期）
   - 图形验证码防刷

3. 第三方登录
   - 微信扫码登录
   - GitHub 登录

## 业务规则
| 规则 | 描述 | 优先级 |
|------|------|--------|
| R1 | 密码长度 8-20 位，需包含字母和数字 | P0 |
| R2 | 连续输错 5 次密码，锁定 30 分钟 | P0 |
| R3 | 验证码有效期 60 秒，同一手机号每分钟只能发一次 | P0 |
| R4 | 密码需包含大小写字母、数字、特殊字符 | P1 |

## 边界条件
| 场景 | 预期行为 |
|------|----------|
| 密码输错 5 次 | 锁定账号 30 分钟，显示锁定提示 |
| 验证码发送频繁 | 提示"请稍后再试" |
| 网络超时 | 显示"网络异常，请重试" |
| 第三方账号未绑定 | 引导绑定手机号 |

## 验收标准
| 标准 | 验证方法 |
|------|----------|
| 密码强度不足无法提交 | 输入弱密码，提交按钮禁用 |
| 输错 5 次锁定 | 模拟输错 5 次，第 6 次提示锁定 |
| 验证码 60s 有效 | 发送后 60s 内有效，过期提示刷新 |
```

## 交互逻辑结构化

### 1. 流程图定义

```
用户登录流程:

[开始] → [输入账号密码] → [密码强度校验] 
                              ↓
                    [不通过] → [提示错误] → [返回输入]
                              ↓
                    [通过] → [提交认证]
                              ↓
                    [认证失败] → [错误计数+1]
                              ↓
                    [认证成功] → [生成Token] → [登录成功]
                              ↓
                    [输错5次] → [锁定30分钟]
```

### 2. 状态定义

```
登录页面状态:

| 状态 | 触发条件 | UI表现 | 数据 |
|------|----------|--------|------|
| default | 初始状态 | 输入框可编辑，登录按钮可点击 | - |
| loading | 点击登录 | 按钮显示 loading， 输入框禁用 | loading=true |
| success | 登录成功 | 跳转首页，显示欢迎 | token=xxx |
| error | 登录失败 | 显示错误提示，错误次数+1 | errorMsg=xxx |
| locked | 账号锁定 | 禁用输入，显示锁定倒计时 | lockedUntil=xxx |
```

## 技术要求明确化

### 1. 技术栈要求

```
技术要求模板:

| 要求项 | 说明 | 必须 |
|--------|------|------|
| 前端框架 | React 18 + TypeScript | 是 |
| 状态管理 | Zustand/Redux | 是 |
| UI组件库 | Ant Design 5.x | 是 |
| 接口协议 | RESTful, JSON | 是 |
| 认证方式 | JWT + Refresh Token | 是 |
| 错误处理 | 全局 Error Boundary | 是 |
```

### 2. 性能要求

```
| 指标 | 要求 | 说明 |
|------|------|------|
| 登录响应时间 | < 500ms | 网络正常情况 |
| 页面首次加载 | < 2s | 4G 网络 |
| 验证码发送延迟 | < 1s | 短信发送 |
| 并发处理能力 | > 1000 TPS | 服务端 |
```
