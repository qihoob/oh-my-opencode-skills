# Skill: product-requirement-analysis

**角色**: 产品 (Product)
**功能**: 需求分析 -- 从业务理解到结构化文档
**触发关键词**: 需求分析、用户故事、需求澄清、功能定义

## 产出文档
**路径**: `.opencode/docs/requirement-{feature-name}.md`
**说明**: 需求分析完成后必须写入此文件，供后续开发/测试技能读取

## 依赖文档
- 无（首次分析）
- 或：`.opencode/docs/project-overview.md`（如有项目概览）

## 工具可用
- file-read: 读取现有需求文档、项目文档
- grep: 搜索相关代码/文档，了解现有系统
- file-write: 写入需求文档到 `.opencode/docs/`

---

## 执行流程

### Step 1: 业务领域理解（在写任何文档之前）

在动手写需求文档之前，先搞清楚业务背景。如果用户提供的信息不够，主动提问。

**必须回答的问题：**

1. **业务动机**
   - 这个需求解决什么业务问题？
   - 谁提的？（业务方、用户反馈、数据分析、竞品对标）
   - 为什么现在要做？有没有截止时间或外部依赖？
   - 做成之后怎么衡量成功？（KPI、指标）

2. **用户角色**
   - 涉及哪些用户角色？（如：普通用户、VIP用户、管理员、运营、客服）
   - 每个角色在什么场景下会用到这个功能？
   - 角色之间有没有协作关系？（如：A提交，B审批）

3. **现有系统**
   - 现有系统是否已有相关功能？如果有，新需求是替代还是增强？
   - 新需求与现有功能如何衔接？数据是否需要迁移？
   - 有没有技术约束？（历史接口、第三方依赖、数据格式）

4. **数据支撑**
   - 有没有用户反馈可以参考？
   - 有没有数据分析结果？（用户行为、转化率、流失点）
   - 有没有竞品做法可以参考？

**执行动作：**
- 如果项目中有 `.opencode/docs/project-overview.md`，先读取了解全局
- 用 grep 搜索项目中与需求相关的现有代码和文档
- 将理解到的业务背景记录在需求文档的「业务背景」章节

---

### Step 2: 用户场景深挖

不要只写一个 Given/When/Then 就结束。真正理解用户在真实环境中的行为。

**必须覆盖的五种场景：**

1. **正常路径（Happy Path）**
   - 用户最常用的操作流程，从头到尾一步步走完
   - 这是核心场景，必须详细到每一步的输入和输出
   - 示例：用户登录 -> 浏览商品列表 -> 选择商品 -> 加入购物车 -> 结算 -> 支付 -> 收到确认

2. **异常路径（Error Path）**
   - 什么情况下会偏离正常流程？每个异常都要考虑：
     - 网络断了（请求超时、断网重连）
     - 数据不存在（查无此人、商品下架）
     - 权限不够（未登录、角色不符）
     - 服务端错误（500、第三方接口超时）
     - 用户主动中断（取消操作、关闭页面）

3. **边界场景（Edge Case）**
   - 极端输入：超长文本（10000字）、特殊字符（emoji、SQL注入尝试）、超大金额、负数
   - 极端状态：0条数据（空页面）、10万条数据（分页、性能）、单条数据的边界操作
   - 时间边界：跨时区、闰年、月末结算、并发时间窗口

4. **回退场景（Recovery Path）**
   - 操作失败后用户怎么恢复？
   - 能否重试？重试是否幂等？
   - 能否撤销？撤销后数据怎么处理？
   - 如果无法自动恢复，用户需要联系谁？（客服、管理员）

5. **多角色场景（Multi-role Path）**
   - 不同角色看到的东西不一样：
     - 管理员 vs 普通用户看到的数据范围不同
     - 创建者 vs 协作者的操作权限不同
     - 审批流的上下游视角不同
   - 多人同时操作同一数据时的行为（乐观锁、冲突提示）

**执行动作：**
- 每个核心场景至少写一个完整的 Given/When/Then
- 异常场景和边界场景不能漏，至少覆盖最常见的 3-5 种
- 在需求文档中按场景类型分组列出

---

### Step 3: 验收标准细化

每个验收标准（Acceptance Criteria, AC）不只是写"功能正常"，必须满足四个属性、包含四个要素。

**四属性检查（每条 AC 必须全部满足）：**

| 属性 | 含义 | 正确示例 | 错误示例 |
|------|------|----------|----------|
| 可量化 | 有明确的数值或状态判断 | 页面加载 < 2秒 | "速度要快" |
| 可验证 | 可以通过具体操作验证 | 点击提交按钮，看到成功提示 | "体验要好" |
| 可区分 | 通过和失败的界限清晰 | 密码6-20位，含字母和数字 | "密码要安全" |
| 有优先级 | MoSCoW 分级 | Must Have | 无优先级 |

**四要素（每条 AC 必须包含）：**

```
AC-001: [标题]
- Given: [前置条件 -- 系统当前状态、用户身份、数据准备]
- When: [触发动作 -- 用户具体操作]
- Then: [预期结果 -- 系统的响应、数据变化、界面变化]
- 如果失败: [失败时的预期行为 -- 错误提示、回滚策略、用户引导]
- 优先级: Must / Should / Could / Won't
```

**AC 编写规则：**
- 每条 AC 只验证一个行为点，不要把多个功能点塞进一条
- AC 之间不要有依赖顺序，每条都可以独立验证
- 数量参考：核心功能 5-10 条 AC，小功能 2-3 条
- 所有 AC 的 Then 必须是客观可观察的，不能是主观感受

---

### Step 4: 隐含需求识别

需求文档通常不会写，但系统必须处理的事情。主动识别并记录。

**逐项检查清单：**

1. **权限控制**
   - 谁能做？谁不能做？每个操作对应什么角色？
   - 未登录用户访问时怎么处理？（跳转登录、展示部分内容、提示注册）
   - 越权操作怎么拦截？（返回403、隐藏入口、记录审计日志）
   - 是否需要操作日志/审计追踪？

2. **数据约束**
   - 每个字段的长度、格式、唯一性、取值范围
   - 哪些必填、哪些选填、哪些有默认值
   - 数据之间的关联约束（如：结束时间 > 开始时间）
   - 软删除还是硬删除？删除后的关联数据怎么处理？

3. **并发场景**
   - 多人同时操作同一数据怎么办？（乐观锁、悲观锁、最终一致）
   - 重复提交怎么办？（幂等设计、防重Token）
   - 库存/名额类数据如何防超卖？

4. **性能预期**
   - 响应时间：接口 P99 < 多少毫秒？页面首屏 < 多少秒？
   - 数据量：单表预计多少数据？列表分页大小？
   - 并发用户数：峰值多少？日常多少？
   - 是否有批量操作？批量上限是多少？

5. **安全要求**
   - 输入校验：XSS、SQL注入、路径遍历
   - 敏感数据：是否需要脱敏展示？（手机号、身份证、金额）
   - 防重放：关键操作（支付、审批）是否需要防重复提交？
   - 数据传输：是否需要加密？是否涉及合规要求？

6. **兼容性**
   - 浏览器：支持哪些浏览器和版本？
   - 设备：PC、平板、手机？横屏竖屏？
   - 网络环境：Wi-Fi、4G、弱网、离线？
   - 是否需要国际化/多语言？

7. **数据初始化**
   - 新用户的默认数据是什么？
   - 如果是存量系统改造，老数据怎么迁移？
   - 是否需要灰度发布策略？

**执行动作：**
- 逐项过上面的清单，适用的全部记录到需求文档
- 不适用的标注"不适用"及原因，避免后续反复确认
- 对技术实现有强约束的需求（如并发、性能），在文档中用醒目标记提醒开发

---

## 最小上下文原则

1. **不加载无关文件** -- 只读取与当前需求相关的现有文档和代码
2. **输出结构化** -- 严格按模板章节输出，便于后续技能按需读取特定章节
3. **文档引用标记** -- 在输出中标注哪些内容供哪个技能使用
4. **隐含需求不遗漏** -- Step 4 的检查清单逐项确认，不因为"没提到"就跳过

## 自驱动

需求文档写入后，检查 `.opencode/state.json` 中的流程状态，触发下一步：
- 如果存在实现技能，推荐读取需求文档开始实现
- 如果需要技术评审，推荐先进行方案设计
