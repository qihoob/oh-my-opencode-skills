# Skill: release-management

**角色**: DevOps
**功能**: 发布管理
**触发关键词**: 发布管理、版本发布、灰度发布、发布计划、上线计划、发布流程、回滚策略

## 产出文档
- **发布计划文档**: `.opencode/docs/release-plan-{version}.md`

## 核心能力

1. **发布清单**
   - Pre-flight check（发布前检查）
   - 发布物清单（代码、配置、数据库、静态资源）
   - 发布顺序规划

2. **灰度策略**
   - 功能开关灰度
   - 流量切换灰度
   - 蓝绿部署
   - 金丝雀发布

3. **回滚判定**
   - 回滚触发条件
   - 回滚步骤
   - 回滚验证

4. **发布窗口**
   - 发布时间选择
   - 发布窗口约束
   - 发布冻结期

5. **监控阈值**
   - 发布后监控指标
   - 告警阈值设定
   - 异常判定标准

6. **发布通知**
   - 发布前通知模板
   - 发布中进度通知
   - 发布后结果通知

## 依赖文档

按优先级检查：
1. `.opencode/docs/code-review-*.md` — 代码审查通过
2. `.opencode/docs/test-report-*.md` — 测试通过
3. `.opencode/docs/acceptance-report.md` — 验收通过（可选）
4. `.opencode/docs/db-change-impact-*.md` — 数据库变更影响分析（如有）
5. `.opencode/docs/branch-strategy-*.md` — 分支策略（版本号来源）

## 执行流程

### Step 1: 发布物收集

```
目的: 收集所有需要发布的变更物

1.1 代码变更
    - bash: git diff main~10..main（查看最近变更）
    - bash: git log --oneline main~10..main（查看最近提交）
    - 确认变更范围

1.2 数据库变更
    - 检查是否有 db-migration-*.sql
    - 检查 db-change-impact-*.md
    - 确认数据库变更类型（新增表/修改字段/索引）

1.3 配置变更
    - 检查 config/*.yaml, config/*.json
    - 检查环境变量变更
    - 检查 feature flag 配置

1.4 静态资源变更
    - 检查前端构建产物
    - 检查 CDN 资源更新

1.5 依赖变更
    - bash: npm ls --depth=0（查看依赖）
    - 检查 package.json 依赖版本变化

输出:
- 代码变更: {变更范围}
- 数据库变更: {变更内容}
- 配置变更: {变更项}
- 静态资源: {变更项}
- 依赖变更: {变更项}
```

### Step 2: Pre-flight Check

```
目的: 发布前强制检查，确保发布条件满足

2.1 检查清单
    | 检查项 | 检查方式 | 预期结果 |
    |--------|----------|----------|
    | 代码审查通过 | code-review-*.md 存在且 result=pass | ✅ |
    | 测试通过 | test-report-*.md 存在且 passRate=100% | ✅ |
    | 验收通过 | acceptance-report.md 存在且 result=pass（可选） | ✅ |
    | 无未合并 PR | bash: git branch --merged main | 无遗留 |
    | 无阻塞 Bug | bash: grep "P0" bug-*.md（如有） | 无 P0 Bug |
    | 数据库迁移脚本就绪 | db-migration-*.sql 存在（如有数据库变更） | ✅ |
    | 配置文件就绪 | config/*.yaml 验证 | ✅ |
    | CI 通过 | bash: gh run list --limit 5（GitHub） | 最新 CI 成功 |

2.2 检查执行
    - 对每项执行检查
    - 不满足项 → 阻止发布，标注原因

2.3 检查结果
    - ✅ 全部通过 → 可发布
    - ❌ 有不满足项 → 不可发布，需先解决

输出:
- Pre-flight Check 结果: ✅ 可发布 / ❌ 不可发布
- 不满足项清单: [不满足项列表]
```

### Step 3: 灰度策略选择

```
目的: 根据变更类型选择合适的灰度策略

3.1 灰度策略矩阵
    ┌───────────────────┬───────────────────┬───────────────────┬───────────────────┐
    │ 变更类型          │ 功能开关灰度      │ 流量切换灰度      │ 蓝绿部署          │
    ├───────────────────┼───────────────────┼───────────────────┼───────────────────┤
    │ 适用场景          │ 新功能上线        │ API 变更          │ 全量重构          │
    │ 灰度比例          │ 0% → 10% → 50% → 100% │ 0% → 5% → 20% → 100% │ 全量切换 │
    │ 回滚难度          │ 低（关闭开关）    │ 中（调整流量）    │ 高（切换回蓝）    │
    │ 监控粒度          │ 功能级            │ API级             │ 服务级            │
    │ 数据库影响        │ 无                │ 可能              │ 可能              │
    └───────────────────┴───────────────────┴───────────────────┴───────────────────┘

3.2 功能开关灰度（推荐用于新功能）
    步骤:
    1. 功能开关初始关闭（0% 用户）
    2. 灰度 10% 用户（白名单/随机）
    3. 观察 24 小时，监控关键指标
    4. 灰度 50% 用户
    5. 观察 24 小时
    6. 全量开放（100%）

    配置示例:
    // feature-flag.yaml
    features:
      new_login:
        enabled: true
        rollout:
          percentage: 10  # 逐步调整: 10 → 50 → 100
          whitelist: ["user_123", "user_456"]

3.3 流量切换灰度（推荐用于 API 变更）
    步骤:
    1. 部署新版本到部分实例（如 5%）
    2. 通过负载均衡切换 5% 流量到新实例
    3. 监控错误率、响应时间
    4. 逐步扩大流量比例（20% → 50% → 100%）

    配置示例:
    // nginx.conf
    upstream backend {
      server old-version:8080 weight=95;
      server new-version:8080 weight=5;  # 逐步调整: 5 → 20 → 50 → 100
    }

3.4 蓝绿部署（推荐用于全量重构）
    步骤:
    1. 部署新版本到 Green 环境（流量 0%）
    2. 运行冒烟测试验证 Green
    3. 切换流量从 Blue 到 Green（一次性切换）
    4. 监控关键指标
    5. 如异常，快速切换回 Blue

    配置示例:
    // k8s service
    apiVersion: v1
    kind: Service
    metadata:
      name: backend
    spec:
      selector:
        version: blue  # 切换时改为 green

3.5 策略选择建议
    | 变更类型 | 建议策略 | 灰度比例建议 |
    |----------|----------|--------------|
    | 新功能（无数据变更） | 功能开关 | 10% → 50% → 100% |
    | API 变更（兼容） | 流量切换 | 5% → 20% → 50% → 100% |
    | API 变更（不兼容） | 功能开关 + 流量切换 | 先开关灰度新 API，再切换 |
    | 数据库变更 | 功能开关 + 数据迁移 | 先迁移数据，再灰度功能 |
    | 全量重构 | 蓝绿部署 | 验证后全量切换 |

输出:
- 灰度策略: {功能开关/流量切换/蓝绿部署}
- 灰度比例: {比例变化序列}
- 灰度配置: {配置示例}
```

### Step 4: 回滚策略定义

```
目的: 定义回滚触发条件和回滚步骤

4.1 回滚触发条件
    | 触发条件 | 阈值 | 检测方式 |
    |----------|------|----------|
    | 错误率飙升 | 错误率 > 5%（原 < 1%） | 监控报警 |
    | 响应时间飙升 | P99 > 3s（原 < 1s） | 监控报警 |
    | 服务不可用 | 可用性 < 95% | 健康检查失败 |
    | 业务指标异常 | 转化率下降 > 20% | 业务监控 |
    | 用户投诉激增 | 投诉量 > 正常 3 倍 | 客服反馈 |
    | 数据异常 | 数据丢失/不一致 | 数据检查 |

4.2 回滚步骤（功能开关）
    步骤:
    1. 关闭功能开关（立即生效）
    2. 通知相关人员
    3. 监控指标恢复
    4. 记录回滚原因
    5. 分析根因

    命令:
    // 关闭功能开关
    curl -X POST http://config-server/features/new_login/disable

4.3 回滚步骤（流量切换）
    步骤:
    1. 调整负载均衡，流量全部切回旧版本
    2. 通知相关人员
    3. 监控指标恢复
    4. 记录回滚原因
    5. 分析根因

    命令:
    // 调整 nginx upstream
    upstream backend {
      server old-version:8080 weight=100;
      server new-version:8080 weight=0;
    }
    nginx -s reload

4.4 回滚步骤（蓝绿部署）
    步骤:
    1. 切换 Service selector 到 Blue
    2. 通知相关人员
    3. 监控指标恢复
    4. 记录回滚原因
    5. 分析根因

    命令:
    // k8s 回滚
    kubectl patch service backend -p '{"spec":{"selector":{"version":"blue"}}}'

4.5 数据库回滚（如有数据库变更）
    - 数据库变更通常不能自动回滚
    - 需要准备回滚脚本（逆操作）
    - 示例: 新增字段 → 删除字段

    回滚脚本: rollback-{version}.sql
    -- 回滚 v1.2.0 数据库变更
    ALTER TABLE users DROP COLUMN new_field;

输出:
- 回滚触发条件: {条件阈值表}
- 回滚步骤: {步骤序列}
- 回滚命令: {命令示例}
- 数据库回滚脚本: {脚本内容（如有）}
```

### Step 5: 发布窗口选择

```
目的: 选择合适的发布时间窗口

5.1 发布窗口约束
    | 约束类型 | 说明 |
    |----------|------|
    | 业务高峰期 | 避免业务高峰（如电商大促期间） |
    | 周末/节假日 | 避免周末发布（除非紧急） |
    | 发布冻结期 | 公司规定的发布冻结期禁止发布 |
    | 团队可用 | 确保发布期间有值班人员 |

5.2 推荐发布窗口
    - 工作日 10:00-16:00（有足够时间处理问题）
    - 避免周五下午（周末无人值班）
    - 避开业务高峰时段

5.3 发布时间表
    | 时间 | 动作 |
    |------|------|
    | 发布前 1 小时 | 发送发布通知 |
    | 发布开始 | 执行发布 |
    | 发布后 30 分钟 | 观察监控指标 |
    | 发布后 2 小时 | 确认稳定，通知发布成功 |
    | 发布后 24 小时 | 全量灰度（如灰度发布） |

输出:
- 发布窗口: {发布日期和时间}
- 发布时间表: {时间动作表}
```

### Step 6: 监控阈值设定

```
目的: 设定发布后监控指标和告警阈值

6.1 监控指标
    | 指标类型 | 指标名称 | 监控方式 |
    |----------|----------|----------|
    | 服务可用性 | HTTP 200 比例 | Prometheus + Grafana |
    | 响应时间 | P50/P99 响应时间 | APM（如 Jaeger） |
    | 错误率 | HTTP 5xx 比例 | Prometheus |
    | QPS | 每秒请求量 | Prometheus |
    | 业务指标 | 转化率/订单量 | 业务监控 |
    | 数据库 | 连接数/慢查询 | DB 监控 |
    | 资源 | CPU/内存/磁盘 | Prometheus |

6.2 告警阈值
    | 指标 | 正常值 | 告警阈值 | 告警级别 |
    |------|--------|----------|----------|
    | HTTP 5xx | < 1% | > 5% | P1 |
    | P99 响应时间 | < 1s | > 3s | P1 |
    | 服务可用性 | > 99% | < 95% | P0 |
    | CPU | < 70% | > 90% | P2 |
    | 内存 | < 80% | > 95% | P1 |
    | 数据库连接数 | < 80 | > 150 | P1 |

6.3 监控 Dashboard
    - 发布前准备监控 Dashboard
    - 发布后实时观察关键指标
    - 异常时立即触发告警

输出:
- 监控指标: {指标列表}
- 告警阈值: {阈值表}
- 监控 Dashboard: {Dashboard 链接}
```
