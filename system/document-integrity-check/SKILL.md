# Skill: document-integrity-check

**角色**: 系统 (System) / 质量保证
**功能**: 文档完整性与一致性检查（含内容级验证）
**触发关键词**: 文档检查、完整性检查、文档审计、文档质量
**版本**: 2.0 (新增内容一致性检查)

## 产出文档
**路径**: `.opencode/docs/DOC_AUDIT.md`
**说明**: 文档审计报告，记录完整性、一致性、内容偏差的详细分析

## 依赖文档
**扫描范围**: `.opencode/docs/` 全部内容

## 两种检查深度

| 深度 | 触发场景 | 检查范围 |
|------|----------|----------|
| **快速检查** | 日常、提测前 | 文档存在性 + 格式规范 |
| **深度检查** | 迭代结束、验收前 | 快速检查 + 内容一致性 + 双向验证 |

## 核心能力

1. **全局扫描**
   - 扫描所有需求文档
   - 扫描所有实现文档
   - 扫描所有测试文档
   - 扫描所有契约文档
   - 建立文档关系图

2. **完整性检查清单**
   ```
   □ 每个需求是否有验收标准？
   □ 每个功能是否有实现文档？
   □ 每个验收标准是否有测试用例？
   □ 涉及跨层的功能是否有契约文档？
   □ 文档是否有最后更新时间？
   □ 文档是否有明确的负责人？
   ```

3. **一致性检查**
   - 需求变更是否有对应实现更新？
   - 实现修改是否有对应测试更新？
   - 测试失败是否有对应问题记录？

4. **内容一致性检查（深度模式）**

   ```
   4.1 需求 ↔ 实现 对比
   - 逐条对比验收标准与实现说明
   - 检测遗漏（需求有但实现没有）
   - 检测镀金（实现了但需求没要求）
   - 检测偏差（实现方式与需求描述不符）

   4.2 契约 ↔ 三层 对比
   - 对比 contract-*.md 中的数据模型与实现文档中的表结构
   - 对比 contract-*.md 中的 API 定义与实现文档中的接口设计
   - 对比 contract-*.md 中的状态流转与实现文档中的业务逻辑
   - 对比 contract-*.md 中的错误码与实际错误处理

   4.3 文档 ↔ 代码 双向验证
   - 实现文档中列出的文件是否存在？
   - 实现文档中描述的接口是否在代码中存在？
   - 代码中的接口是否都有对应的文档？
   - 数据库表是否有对应的文档说明？
   ```

5. **质量评分**
   - 文档覆盖率 = 有文档的功能数 / 总功能数
   - 测试覆盖率 = 有测试的功能数 / 总功能数
   - 一致性得分 = 一致的文档对数 / 总文档对数
   - 内容准确率 = 内容匹配的检查项 / 总检查项

## 执行流程

### Step 1: 建立文档关系图

```
扫描 .opencode/docs/ 建立文档依赖图：

requirement-*.md ──→ contract-*.md ──→ implementation-*.md ──→ test-cases-*.md ──→ test-report-*.md
      │                    │                    │                     │
      │                    └→ (跨层契约)        └→ (代码变更记录)     └→ (Bug记录)
      │
      └→ (验收标准) ──→ collab-acceptance-review

输出: 文档依赖矩阵
```

### Step 2: 存在性检查

```
逐条检查文档链条：

| 功能 | 需求 | 契约 | 实现 | 测试用例 | 测试报告 | 完整性 |
|------|------|------|------|----------|----------|--------|
| 登录 | ✅ | - | ✅ | ✅ | ✅ | 🟢 完整 |
| 订单 | ✅ | ✅ | ✅ | ✅ | ❌ | 🔴 缺报告 |
| 支付 | ✅ | ❌ | ✅ | ❌ | ❌ | 🔴 断链 |
```

### Step 3: 内容一致性检查（深度模式）

```
对每条文档链，抽取关键字段进行对比：

3.1 需求 → 实现对齐
| 需求AC | 实现说明 | 一致性 |
|--------|----------|--------|
| AC1: 用户可用邮箱登录 | 实现了邮箱+手机登录 | ⚠️ 超出范围 |
| AC2: 密码错误3次锁定 | 实现了5次锁定 | ❌ 参数不一致 |
| AC3: 记住我功能 | 无实现 | 🔴 遗漏 |

3.2 契约 → 实现对齐
| 契约定义 | 数据库 | 后端 | 前端 | 一致性 |
|----------|--------|------|------|--------|
| POST /api/orders | ✅ | ✅ | ✅ | 🟢 |
| status: pending/paid | ✅ | ✅ | ❌ 前端无 cancelled | 🔴 |
| 错误码 ORDER_EXPIRED | - | ✅ | ❌ 前端未处理 | 🟡 |

3.3 文档 → 代码双向
| 文档描述 | 代码实际 | 一致性 |
|----------|----------|--------|
| 文档: GET /api/users | 代码: GET /api/v2/users | ❌ 路径不一致 |
| 文档: 返回 {list: []} | 代码: 返回 {items: []} | ❌ 字段不一致 |
| 文档: 参数 pageSize | 代码: 参数 size | ❌ 参数名不一致 |
```

### Step 4: 产出审计报告

## 检查规则

### 需求文档规则
- 必须包含用户故事
- 必须包含验收标准（至少 1 个场景）
- 必须包含优先级

### 实现文档规则
- 必须引用对应的需求文档
- 必须说明技术实现方案
- 必须说明已知限制
- 必须包含变更影响分析（如为变更）

### 契约文档规则
- 必须包含数据模型定义
- 必须包含 API 接口定义
- 必须包含状态流转定义（如有状态）
- 必须包含错误码定义

### 测试文档规则
- 必须引用对应的需求文档
- 必须覆盖所有验收标准
- 必须包含测试步骤和预期结果
