# oh-my-opencode-skills 完整任务清单

基于 53 个技能的完整 SDLC 任务清单，覆盖产品→协同→开发→测试→系统→项目→DevOps→视觉全流程

---

## 按阶段划分的任务清单

### 阶段 1: 产品规划与需求分析 (10 skills)

#### 1.1 需求分析 (`product/requirement/product-requirement-analysis`)
**触发词**: 需求分析、用户故事、需求澄清、功能定义

- [ ] 收集原始需求描述
- [ ] 识别目标用户画像
- [ ] 明确业务背景和价值
- [ ] 编写用户故事 (As a... I want to... So that...)
- [ ] 定义验收标准 (Gherkin 格式)
  - [ ] 正常流程场景
  - [ ] 异常处理场景
  - [ ] 边界条件场景
- [ ] 优先级排序 (MoSCoW 方法)
  - [ ] Must Have (必须有)
  - [ ] Should Have (应该有)
  - [ ] Could Have (可以有)
  - [ ] Won't Have (本次不做)
- [ ] 产出需求文档 `.opencode/docs/requirement-{feature}.md`

#### 1.2 需求整理 (`product/requirement/product-requirement-structuring`)
**触发词**: 需求整理、结构化需求

- [ ] 整合多个相关需求
- [ ] 建立需求层次结构
- [ ] 识别需求依赖关系
- [ ] 制定需求实现顺序
- [ ] 输出结构化需求文档

#### 1.3 需求协同优化 (`product/requirement/product-collaborative-requirement-optimization`)
**触发词**: 需求评审、三方评审

- [ ] 组织需求三方评审会议 (产品+开发+测试)
- [ ] 收集各方反馈意见
- [ ] 识别需求冲突与矛盾
- [ ] 协调达成一致方案
- [ ] 更新优化需求文档
- [ ] 输出需求评审记录

#### 1.4 项目全局分析 (`product/analysis/global-project-analysis`)
**触发词**: 项目分析、模块划分、全局梳理

- [ ] 梳理项目整体架构
- [ ] 识别功能模块边界
- [ ] 分析模块间依赖关系
- [ ] 评估技术栈选型
- [ ] 输出项目架构文档

#### 1.5 数据分析 (`product/analysis/data-analysis`)
**触发词**: 数据分析、埋点、指标

- [ ] 定义业务指标体系
- [ ] 设计数据埋点方案
- [ ] 规划数据看板
- [ ] 制定数据验证方法
- [ ] 输出数据分析文档

#### 1.6 技术评估 (`product/analysis/product-technical-assessment`)
**触发词**: 技术评估、技术可行性

- [ ] 评估技术实现难度
- [ ] 识别技术风险点
- [ ] 分析技术选型优劣
- [ ] 估算开发工作量
- [ ] 输出技术评估报告

#### 1.7 模块需求细化 (`product/module/module-product-requirement`)
**触发词**: 模块需求、详细需求

- [ ] 细化模块级需求
- [ ] 明确模块接口定义
- [ ] 定义模块间交互流程
- [ ] 输出模块需求文档

#### 1.8 模块文档管理 (`product/module/module-document-keeper`)
**触发词**: 模块文档、文档整理、文档管理、把控全局

- [ ] 扫描 `.opencode/docs/` 目录
- [ ] 按模块归类文档
- [ ] 生成模块文档索引 `INDEX.md`
- [ ] 检查文档完整性
- [ ] 标识缺失文档
- [ ] 生成文档审计报告 `DOC_AUDIT.md`

#### 1.9 用户反馈分析 (`product/feedback/analysis`)
**触发词**: 用户反馈、反馈分析、NPS、A/B 测试

- [ ] 收集用户反馈数据
- [ ] 分析 NPS 评分
- [ ] 设计 A/B 测试方案
- [ ] 提炼改进建议
- [ ] 输出反馈分析报告

#### 1.10 页面功能最佳实践 (`product/module/product-page-feature-best-practices`)
**触发词**: 页面最佳实践、交互规范

- [ ] 分析页面功能需求
- [ ] 匹配业界最佳实践方案
- [ ] 制定交互设计规范
- [ ] 定义页面性能指标
- [ ] 建立组件复用建议
- [ ] 输出页面功能最佳实践文档

### 阶段 2: 协同交接 (9 skills)

#### 2.1 产品→开发需求交接 (`collaboration/handoff/collab-product-to-dev`)
**触发词**: 需求交接、需求评审、开发确认

- [ ] 组织需求评审会议
- [ ] 详细说明功能需求
- [ ] 解答开发疑问
- [ ] 确认技术实现方案
- [ ] 识别技术风险
- [ ] 明确完成标准
- [ ] 约定沟通机制
- [ ] 记录决策要点
- [ ] 产出需求交接单

#### 2.2 开发→测试提测交接 (`collaboration/handoff/collab-dev-to-qa`)
**触发词**: 提测、测试交接、开发完成、申请测试

- [ ] 确认功能开发完成
- [ ] 完成单元测试
- [ ] 完成代码审查
- [ ] 开发自测通过
- [ ] 提供测试账号/数据
- [ ] 标注风险点
- [ ] 说明测试重点范围
- [ ] 建议回归测试模块
- [ ] 确认测试环境准备就绪
- [ ] 产出提测申请单

#### 2.3 验收评审 (`collaboration/review/collab-acceptance-review`)
**触发词**: 验收、UAT、验收评审、产品验收、上线确认

- [ ] 准备验收测试用例
- [ ] 组织验收会议
- [ ] 逐项验证验收标准
- [ ] 记录验收问题
- [ ] 评估问题严重程度
- [ ] 做出验收决策
  - [ ] 通过 - 可上线
  - [ ] 有条件通过 - 修复后可上线
  - [ ] 需要复测 - 修复后需重新验收
  - [ ] 不通过 - 延期上线
- [ ] 制定问题修复计划
- [ ] 确认上线时间
- [ ] 产出验收评审报告

#### 2.4 迭代复盘 (`collaboration/process/collab-retrospective`)
**触发词**: 复盘、回顾、总结、retro、迭代总结

- [ ] 收集迭代数据
  - [ ] 计划 vs 实际完成情况
  - [ ] Bug 统计与趋势
  - [ ] 代码审查问题统计
  - [ ] 测试覆盖率
- [ ] 识别做得好的 (Keep)
- [ ] 识别需要改进的 (Improve)
- [ ] 识别停止做的 (Stop)
- [ ] 制定改进行动计划
  - [ ] 明确行动项
  - [ ] 分配负责人
  - [ ] 设定时间节点
- [ ] 规划下迭代目标
- [ ] 产出迭代复盘报告

#### 2.5 Bug协调器 (`collaboration/process/bug-coordinator`)
**触发词**: Bug协调、Bug追踪、Bug修复

- [ ] 收集和汇总 Bug 列表
- [ ] 对 Bug 进行分类分级 (P0/P1/P2/P3)
- [ ] 分配 Bug 修复责任人
- [ ] 跟踪 Bug 修复进度
- [ ] 协调跨模块 Bug 修复
- [ ] 验证 Bug 修复结果
- [ ] 产出 Bug 协调报告

#### 2.6 迭代闭环 (`collaboration/process/iteration-closure`)
**触发词**: 迭代闭环、下一迭代、复盘后

- [ ] 确认迭代目标完成情况
- [ ] 归档迭代文档和产出物
- [ ] 总结本迭代经验教训
- [ ] 过滤未完成事项至下迭代
- [ ] 规划下迭代目标和范围
- [ ] 同步迭代状态给干系人
- [ ] 产出迭代闭环报告

#### 2.7 文档对齐 (`collaboration/sync/document-alignment`)
**触发词**: 文档对齐、文档同步、三方对齐

- [ ] 比对需求文档与实现文档
- [ ] 比对实现文档与测试用例
- [ ] 识别文档不一致处
- [ ] 协调相关方更新文档
- [ ] 生成文档对齐报告
- [ ] 记录待对齐事项

#### 2.8 上下文对齐 (`collaboration/sync/context-alignment`)
**触发词**: 同步、对齐、上下文对齐

- [ ] 识别需要同步的上下文信息
- [ ] 收集各方当前工作状态
- [ ] 比对各方理解差异
- [ ] 组织对齐沟通
- [ ] 达成共识并记录
- [ ] 分发对齐后的上下文
- [ ] 产出上下文对齐报告

#### 2.9 应急响应 (`collaboration/sync/incident`)
**触发词**: 线上故障、紧急、P0、应急

- [ ] 接收故障报告
- [ ] 快速评估影响范围和严重程度
- [ ] 组建应急响应团队
- [ ] 制定应急处理方案
  - [ ] 止损方案 (回滚/降级/限流)
  - [ ] 修复方案
  - [ ] 验证方案
- [ ] 执行应急处理
- [ ] 持续监控恢复状态
- [ ] 组织故障复盘
- [ ] 产出故障报告和改进措施

#### 3.6 代码审查 (`dev/code-review`)
**触发词**: 代码审查、Code Review、CR、代码检查、代码评审、代码质量

**通用审查项**:
- [ ] 功能正确性
  - [ ] 代码实现符合需求文档
  - [ ] 边界条件处理完整
  - [ ] 异常场景覆盖
  - [ ] 并发/竞态条件考虑
- [ ] 代码质量
  - [ ] 遵循项目代码风格
  - [ ] 命名清晰有意义
  - [ ] 函数单一职责
  - [ ] 无重复代码 (DRY)
  - [ ] 圈复杂度 < 10
- [ ] 错误处理
  - [ ] 所有异常被捕获处理
  - [ ] 错误信息清晰有用
  - [ ] 无空 catch 块
  - [ ] 资源正确释放
- [ ] 性能
  - [ ] 无 N+1 查询问题
  - [ ] 循环内无低效操作
  - [ ] 适当使用缓存
  - [ ] 大数据量处理优化
- [ ] 安全
  - [ ] 输入参数校验
  - [ ] SQL 注入防护
  - [ ] XSS 防护
  - [ ] 敏感信息不硬编码
  - [ ] 权限校验完整
- [ ] 可维护性
  - [ ] 代码注释充分
  - [ ] 复杂逻辑有说明
  - [ ] TODO/FIXME 标注
  - [ ] 文档同步更新
- [ ] 测试
  - [ ] 单元测试覆盖率 > 80%
  - [ ] 关键逻辑有测试
  - [ ] 边界条件测试
  - [ ] 测试用例清晰独立

**前端特定审查项** (如适用):

- [ ] Props 类型定义完整
- [ ] 状态管理合理

- [ ] 避免内联函数/对象
- [ ] 图片懒加载

- [ ] 避免不必要的重渲染
- [ ] 样式复用性
- [ ] 响应式适配

**后端特定审查项** (如适用):

- [ ] 参数校验 (DTO/Schema)
- [ ] 响应格式统一
- [ ] 状态码正确
- [ ] 数据库索引合理使用
- [ ] 无慢查询风险
- [ ] 事务边界正确
