# Skill: sprint-planning

**角色**: 项目 (Project)
**功能**: Sprint/迭代规划
**触发关键词**: Sprint规划、迭代规划、排期、规划迭代、Sprint启动、迭代开局

## 产出文档
- **Sprint规划文档**: `.opencode/docs/sprint-plan-{id}.md`

## 核心能力

1. **需求池管理**
   - 需求优先级排序（MoSCoW / WSJF）
   - 需求依赖关系分析
   - 需求状态追踪

2. **工作量估算**
   - Story Point 估算
   - T恤尺码估算（S/M/L/XL）
   - 三点估算（乐观/悲观/最可能）
   - 历史数据校准

3. **资源分配**
   - 团队成员技能匹配
   - 任务分配（谁做什么）
   - 工作量均衡检查

4. **Sprint目标定义**
   - Sprint目标（一句话）
   - Sprint范围确认
   - 风险预留（缓冲时间）

## 依赖文档

按优先级检查：
1. `.opencode/docs/requirement-*.md` — 已完成的需求文档
2. `.opencode/docs/health-check-*.md` — 上一迭代遗留问题（可选）
3. `.opencode/docs/retrospective-report.md` — 复盘建议（可选）

## 执行流程

### Step 1: 收集输入

```
目的: 收集所有需要纳入规划的需求和约束

1.1 读取需求池
    - glob 找到所有 requirement-*.md
    - 筛选状态为 "待排期" 的需求

1.2 读取上一迭代遗留
    - health-check-*.md 中的偏移项
    - retrospective-report.md 中的改进任务

1.3 确认资源约束
    - 团队可用人力（人天）
    - Sprint 长度（通常 2 周）
    - 固定事件（休假、培训）

输出:
- 可排需求: [需求列表]
- 遗留任务: [遗留列表]
- 可用资源: X 人天
```

### Step 2: 优先级排序

```
目的: 确定需求优先级，高优先级优先纳入

2.1 选择排序方法
    - MoSCoW: Must/Should/Could/Won't
    - WSJF: (业务价值 + 时间紧迫性 + 风险降低) / 工作量

2.2 应用排序规则
    - 业务优先级（来自需求文档）
    - 依赖关系（前置需求必须先做）
    - 技术风险（高风险项尽早做）

2.3 输出优先级队列
    | ID | 需求名称 | 业务优先级 | 依赖 | 技术风险 | 综合评分 |
    |----|----------|------------|------|----------|----------|
    | R1 | 用户登录 | P0 | 无 | 低 | 100 |
    | R2 | 支付功能 | P0 | R1 | 高 | 95 |
    | R3 | 商品列表 | P1 | 无 | 低 | 80 |

输出:
- 优先级队列: [排序后的需求列表]
```

### Step 3: 工作量估算

```
目的: 估算每个需求的工作量，用于容量规划

3.1 选择估算方法
    - Story Point（推荐）: 1/2/3/5/8/13/21
    - T恤尺码: S(1天)/M(2-3天)/L(5天)/XL(10天+)
    - 人天估算: 直接估算天数

3.2 执行估算
    - 读取每个需求的 AC 数量、复杂度
    - 参考历史类似需求的工作量
    - 考虑技术复杂度（后端/前端/数据库变更）

3.3 估算结果
    | ID | 需求名称 | Story Point | 人天估算 | 前端 | 后端 | 测试 |
    |----|----------|-------------|----------|------|------|------|
    | R1 | 用户登录 | 5 SP | 3 天 | 1天 | 1天 | 1天 |
    | R2 | 支付功能 | 13 SP | 8 天 | 3天 | 3天 | 2天 |
    | R3 | 商品列表 | 3 SP | 2 天 | 1天 | 0.5天 | 0.5天 |

3.4 校准检查
    - 总估算是否超过可用容量？
    - 是否需要调整范围？

输出:
- 估算表: [需求工作量明细]
- 总工作量: X 人天
```

### Step 4: 容量匹配与范围确认

```
目的: 确认本次 Sprint 能做多少需求

4.1 计算可用容量
    - 团队人数 × Sprint 天数 × 效率系数(通常 0.7-0.8)
    - 扣除固定事件（休假、培训）
    - 示例: 5人 × 10天 × 0.75 = 37.5 人天

4.2 匹配需求
    - 按优先级从高到低纳入
    - 总工作量 ≤ 可用容量 - 预留(通常 10-20%)
    - 预留用于应对突发情况

4.3 范围确认
    - 确认纳入的需求列表
    - 确认不纳入的需求（推迟到下一 Sprint）
    - 确认风险预留比例

输出:
- Sprint 范围: [纳入的需求列表]
- 总工作量: X 人天
- 预留: Y 人天 (Z%)
```

### Step 5: 任务分配

```
目的: 将需求拆分成任务，分配给具体人员

5.1 需求拆解为任务
    - 每个需求拆成: 前端任务、后端任务、测试任务
    - 标注依赖关系（后端先做，前端依赖接口）

5.2 技能匹配
    - 根据团队成员技能分配
    - 前端开发 → 前端任务
    - 后端开发 → 后端任务
    - 测试工程师 → 测试任务

5.3 工作量均衡
    - 检查每人工作量是否均衡
    - 避免某人过载或空闲

5.4 输出任务分配表
    | 需求 | 任务 | 负责人 | 工作量 | 开始日期 | 结束日期 |
    |------|------|--------|--------|----------|----------|
    | R1-登录 | 后端API | 张三 | 1天 | 3/1 | 3/1 |
    | R1-登录 | 前端页面 | 李四 | 1天 | 3/2 | 3/2 |
    | R1-登录 | 测试 | 王五 | 1天 | 3/3 | 3/3 |

输出:
- 任务分配表: [详细任务分配]
- 均衡检查: ✅ 均衡 / ⚠️ 不均衡需调整
```

### Step 6: Sprint 目标与交付物

```
目的: 定义 Sprint 目标和预期交付物

6.1 Sprint 目标定义
    - 一句话描述本次 Sprint 要达成什么
    - 示例: "完成用户登录和支付功能，支持用户完成下单流程"

6.2 交付物清单
    - 功能交付物: [需求对应的功能]
    - 文档交付物: [需求文档、设计文档、测试报告]
    - 其他交付物: [数据库变更、配置变更]

6.3 验收标准
    - 所有纳入需求完成开发、测试
    - 代码审查通过
    - 功能演示通过

输出:
- Sprint 目标: {一句话描述}
- 交付物清单: [交付物列表]
- 验收标准: [验收条件]
```
