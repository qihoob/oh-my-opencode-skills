## 阶段1: 需求分析

### 1.1 收集需求

```
输入: 用户/产品描述的核心需求
输出: 结构化需求清单
```

**需求采集问题**:
- 核心用户痛点是什么？
- 目标用户群体？
- 主要功能优先级？
- 性能/安全要求？
- 集成需求？

### 1.2 产出文档

#### 产品需求文档 (PRD)

```markdown
# [项目名称] 产品需求文档

## 1. 概述
- 项目背景
- 目标用户
- 核心价值

## 2. 功能需求

### 2.1 功能模块A
| 功能 | 描述 | 优先级 | 验收标准 |
|------|------|--------|----------|
| 功能A.1 | 描述 | P0 | 验收条件1 |
| 功能A.2 | 描述 | P1 | 验收条件2 |

### 2.2 功能模块B
...

## 3. 非功能需求
- 性能要求
- 安全要求
- 兼容性

## 4. 验收标准
- [ ] 验收项1
- [ ] 验收项2
```

#### 架构设计文档

```markdown
# [项目名称] 架构设计

## 1. 系统架构
- 整体架构图
- 技术选型

## 2. 模块划分
| 模块 | 职责 | 技术栈 |
|------|------|--------|
| 模块A | 核心业务 | React + Node.js |
| 模块B | 数据服务 | Python |

## 3. 数据模型
- 核心实体
- 关系图

## 4. API 接口设计
- 接口规范
- 认证机制
```

## 阶段2: 接口定义

### 2.1 API 契约

```markdown
# API 接口定义

## 模块A

### 获取列表
- **URL**: `GET /api/v1/module-a`
- **请求参数**:
  | 参数 | 类型 | 必填 | 说明 |
  |------|------|------|------|
  | page | number | 否 | 页码 |
  | size | number | 否 | 每页数量 |

- **响应**:
```json
{
  "code": 0,
  "data": {
    "items": [],
    "total": 0
  }
}
```

### 创建资源
- **URL**: `POST /api/v1/module-a`
- **请求体**:
```json
{
  "name": "string",
  "description": "string"
}
```
```

### 2.2 模块接口依赖表

| 调用方 | 提供方 | 接口 | 依赖类型 |
|--------|--------|------|----------|
| 前端A | 后端A | /api/xxx | 强依赖 |
| 模块B | 模块C | /api/yyy | 弱依赖 |

## 阶段4: 测试执行

### 4.1 测试策略

| 测试类型 | 覆盖范围 | 工具 |
|----------|----------|------|
| 单元测试 | 核心逻辑 | Jest/Vitest |
| 集成测试 | API 接口 | Supertest |
| E2E测试 | 关键流程 | Playwright |

### 4.2 测试用例模板

```markdown
# [模块名] 测试用例

## 功能点A

### 用例1: 正常场景
- **前置条件**: 用户已登录
- **输入**: 点击按钮A
- **预期**: 弹出对话框

### 用例2: 异常场景
- **前置条件**: 用户未登录
- **输入**: 点击按钮A
- **预期**: 跳转登录页
```

### 4.3 Bug 报告模板

```markdown

## 基本信息

## 问题描述
[详细描述]

## 复现步骤
1. xxx

## 预期结果
[描述]

## 实际结果
[描述]

## 根因分析
[分析]

## 修复方案
[方案]

## 关联模块
- 调用方: 模块A
- 提供方: 模块B
```

## 阶段5: Bug 协同修复

### 5.1 Bug 分类

| 类型 | 处理方式 |
|------|----------|
| 本模块 Bug | 模块负责人修复 |
| 跨模块 Bug | 协调相关模块修复 |
| 公共组件 Bug | 组件负责人修复 |

### 5.2 协同流程

```
Bug发现 -> 责任认定 -> 修复排期 -> 开发修复 -> 回归测试 -> 关闭
```
