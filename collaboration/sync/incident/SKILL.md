# 应急响应 Skill

**角色**: 协同 (Collaboration)
**功能**: 线上故障应急响应
**触发关键词**: 线上故障、紧急、P0、应急、服务挂了

## 产出文档

- **应急处理报告**: `.opencode/docs/incident-report-{id}.md`

## 依赖文档

- 无（立即响应，不等待前置文档）

## 核心能力

### 1. 应急响应
- 问题定位
- 快速止血（降级/回滚/限流）
- 服务恢复

### 2. 沟通协调
- 进度同步
- 影响评估
- 用户通知

### 3. 故障复盘
- 根因分析
- 改进措施
- 预防计划

## 严重程度分级

| 级别 | 影响 | 响应时间 | 解决目标 |
|------|------|---------|---------|
| P0 | 全站不可用 | 5min | 30min |
| P1 | 核心功能不可用 | 15min | 2h |
| P2 | 部分功能异常 | 1h | 8h |
| P3 | 体验问题 | 4h | 24h |

## 执行步骤

### Step 1: 确认问题与止血

```
1.1 确认故障现象
    - 报警信息：监控/日志/用户反馈
    - 影响范围：哪些用户/哪些功能
    - 严重程度：P0/P1/P2/P3

1.2 快速止血
    - 降级：关闭非核心功能
    - 回滚：回退到上一个稳定版本
    - 限流：限制流量防止级联故障
    - 熔断：切断故障链路

1.3 通知相关方
    - 开发团队：开始排查
    - 产品团队：评估业务影响
    - 用户：公告说明（如需要）
```

### Step 2: 根因定位

```
2.1 收集线索
    - 错误日志和时间线
    - 最近的变更（部署/配置/代码）
    - 资源状态（CPU/内存/磁盘/网络）

2.2 缩小范围
    - 是代码问题还是基础设施问题？
    - 是单个服务还是多个服务？
    - 是特定数据触发还是普遍问题？

2.3 定位根因
    - 找到具体的代码行/配置项/资源瓶颈
    - 确认为什么在这个时间点触发
```

### Step 3: 修复验证

```
3.1 实施修复
    - 最小修复（止血优先，不是重构）
    - 灰度发布或全量发布

3.2 验证恢复
    - 监控指标恢复正常
    - 核心功能验证通过
    - 确认无次生故障
```

### Step 4: 产出应急报告

将完整处理过程写入 `.opencode/docs/incident-report-{id}.md`。

### Step 5: 后续跟进

```
- 如有残留 Bug → 触发 bug-coordinator 分配修复
- 如需深入复盘 → 触发 collab-retrospective
```

## 配合 Skills

| 配合技能 | 关系 | 说明 |
|----------|------|------|
| `dev/context/dev-context-first` | 后续 | 止血后获取上下文，定位根因 |
| `dev/implementation/dev-implementation` | 后续 | 根因修复实现 |
| `collaboration/process/bug-coordinator` | 协同 | 残留 Bug 分配和追踪 |
| `qa/execution/test-executor` | 后续 | 修复后执行验证测试 |
| `collaboration/process/collab-retrospective` | 后续 | 故障复盘 |
| `system/security/compliance` | 条件 | 如涉及安全事件，触发合规检查 |

## 工具可用
- read: 读取相关代码、配置、日志
- grep: 搜索错误日志、定位异常代码
- bash: 执行诊断命令、查看服务状态
- write: 写入应急处理报告
