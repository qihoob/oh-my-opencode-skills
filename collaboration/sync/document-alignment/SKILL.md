# Skill: document-alignment

**角色**: 协同 (Collaboration) / 技术负责人
**功能**: 跨角色文档对齐与同步
**触发关键词**: 文档对齐、文档同步、三方对齐、文档一致性

## 产出文档
**路径**: `.opencode/docs/alignment-{module}.md`
**说明**: 文档对齐报告，记录三方文档的一致性和同步状态

## 依赖文档
**必须读取**:
- `.opencode/docs/requirement-*.md` - 所有需求文档
- `.opencode/docs/implementation-*.md` - 所有实现文档
- `.opencode/docs/test-cases-*.md` - 所有测试用例

## 核心能力

1. **三方文档比对**
   ```
   需求文档 ←→ 实现文档 ←→ 测试文档
       ↑           ↑           ↑
    验收标准    技术方案    测试用例
   ```

2. **对齐检查点**
   - 需求的每个验收标准是否有实现？
   - 实现的每个功能点是否有测试？
   - 测试的每个用例是否对应验收标准？

3. **变更同步检查**
   - 需求变更后，实现文档是否更新？
   - 实现修改后，测试用例是否更新？
   - 测试失败后，是否有问题追踪？

4. **对齐会议支持**
   - 生成对齐会议材料
   - 标识需要对齐的条目
   - 记录对齐决策

## 输出格式

```markdown
# 文档对齐报告：{模块名称}

**对齐时间**: 2026-03-09
**参与角色**: 产品、开发、测试

## 对齐状态概览
| 功能 | 需求 - 实现对齐 | 实现 - 测试对齐 | 整体状态 |
|------|----------------|----------------|----------|
| 登录 | ✅ 一致 | ✅ 一致 | 🟢 通过 |
| 注册 | ✅ 一致 | ⚠️ 部分覆盖 | 🟡 需注意 |
| 支付 | ⚠️ 有差异 | ❌ 缺失 | 🔴 需对齐 |

## 详细对齐检查

### 登录功能
**需求验收标准**:
- AC1: 用户名密码验证 ✅ 已实现 ✅ 已测试
- AC2: 记住登录状态 ✅ 已实现 ✅ 已测试
- AC3: 失败错误提示 ✅ 已实现 ✅ 已测试

**对齐结论**: 完全对齐

### 支付功能
**需求验收标准**:
- AC1: 支持支付宝 ✅ 已实现 ✅ 已测试
- AC2: 支持微信支付 ✅ 已实现 ❌ 未测试
- AC3: 支付超时处理 ⚠️ 实现不完整 ❌ 未测试

**对齐问题**:
1. AC2 缺少测试用例
2. AC3 实现逻辑与需求不符

**待对齐事项**:
- [ ] 产品确认 AC3 需求细节
- [ ] 开发补充 AC3 实现
- [ ] 测试补充 AC2、AC3 用例

## 变更同步记录
| 时间 | 变更内容 | 是否同步 | 状态 |
|------|----------|----------|------|
| 03-01 | 需求 AC3 修改 | ❌ 未同步实现 | 待处理 |
| 03-05 | 实现方案调整 | ✅ 已同步测试 | 完成 |

## 对齐会议记录
**时间**: 2026-03-09
**参与**: 产品 A、开发 B、测试 C
**决策**:
1. AC3 需求维持原样，开发周三前完成
2. 测试周五前补充用例
```

## 对齐工作流

1. **会前准备** - 生成对齐报告初稿
2. **对齐会议** - 讨论差异点并记录决策
3. **会后跟进** - 跟踪待办事项完成
