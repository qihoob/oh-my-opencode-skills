# Skill: collab-acceptance-review

**角色**: 协同 (Collaboration)
**功能**: 验收评审（含 AC 代码对照验证、测试覆盖验证、项目级健康总评）
**触发关键词**: 验收、UAT、验收评审、产品验收、上线确认
**版本**: 3.0 (新增实现与需求对照验证、增强验收决策判断标准)

## 核心能力

1. **验收准备**
   - 确认需求文档验收标准
   - 准备验收测试用例
   - 组织验收会议

2. **验收执行**
   - 逐项验证验收标准
   - 记录验收问题
   - 评估问题严重程度

3. **实现与需求对照验证**（核心步骤）
   - 逐条 AC 对照代码实现
   - 检查代码额外实现
   - 测试覆盖验证
   - Bug 修复验证

4. **验收决策**
   - 基于对照验证结果判断通过/有条件通过/不通过
   - 制定问题修复计划
   - 确认上线时间

## 执行流程

### Step 1: 验收准备

1. 读取 `.opencode/docs/requirement-*.md`，提取所有验收标准（AC）
2. 读取 `.opencode/docs/test-cases-*.md`，确认测试用例覆盖了所有 AC
3. 读取 `.opencode/docs/test-report-*.md`，获取最新测试结果
4. 读取 `.opencode/docs/implementation-*.md`，获取实现范围和涉及文件
5. 如有 Bug 记录，读取 `.opencode/docs/bug-*.md`
6. 确认验收环境已部署最新版本

### Step 2: 验收执行

逐项检查 AC checklist，确认每条验收标准的功能表现：

1. 按 AC 编号逐项检查
2. 对每个 AC 执行功能验证（手动或自动化结果确认）
3. 记录验证结果：通过 / 未通过 / 部分通过
4. 对未通过项记录问题详情、严重程度、修复建议

### Step 3: 实现与需求对照验证（核心步骤）

**目的**: 不是只核对 AC checklist，而是确认代码确实实现了需求描述的那些东西。

#### 3.1 逐条 AC 对照代码

对每条 AC：

1. 读取相关代码文件（根据 implementation-*.md 中记录的文件路径）
2. 确认代码中存在实现该 AC 的逻辑
3. 确认实现方式与 AC 描述一致（不是代码写了一个东西、需求说的是另一个东西）
4. 标注实现状态：**已实现** / **部分实现** / **未实现** / **实现与描述不一致**

输出格式：

| AC# | 验收标准 | 代码位置 | 实现状态 | 备注 |
|-----|----------|----------|----------|------|
| AC1 | 密码≥6位 | auth.service.ts:23 | 已实现 | 代码校验≥8位，比AC更严格 |
| AC2 | 超时取消 | order.service.ts:89 | 未实现 | 代码中无超时逻辑 |
| AC3 | 短信通知 | notify.ts:45 | 实现与描述不一致 | AC要求短信，代码实现的是邮件通知 |

#### 3.2 代码额外实现检查

对照需求和实现文档，检查代码中是否存在 AC 没提到的功能：

1. 扫描实现涉及的代码文件，识别所有功能入口和业务逻辑
2. 与 AC 清单逐一比对，找出额外实现
3. 对每个额外实现判断：
   - 是否为实现 AC 过程中的合理延伸（如必要的错误处理、日志记录）
   - 是否是未经产品确认的"悄悄加塞"功能
   - 是否应该纳入本次验收范围

输出格式：

| 功能 | 代码位置 | 是否应纳入验收 | 判断依据 |
|------|----------|---------------|----------|
| 操作限流 | api/rate-limit.ts | 是（合理延伸） | 防止接口滥用，属于安全基线 |
| 用户行为统计上报 | tracker.ts:12 | 需产品确认 | AC 中无此需求，需确认是否预期 |

#### 3.3 测试覆盖验证

1. 读取 `test-report-{feature}.md`，获取测试执行结果
2. 对照 `test-cases-{feature}.md`，确认所有用例都已执行
3. 检查以下项：
   - 失败的用例是否已修复并重新验证？
   - 有没有跳过的用例？为什么跳过？
   - 跳过的用例是否影响验收结论？
   - 用例覆盖是否遗漏了关键场景？

输出格式：

```
- 用例总数: {N}，已执行: {N}，通过: {N}，失败: {N}，跳过: {N}
- 跳过原因: {逐条说明}
- 失败用例修复状态: {逐条说明}
- 覆盖缺口: {说明是否有遗漏的关键场景}
```

#### 3.4 Bug 修复验证（如有 Bug）

如果测试过程中产生过 Bug：

1. 读取 `bug-{id}.md`，确认 Bug 的根因描述
2. 读取修复代码，确认修复确实解决了根因（不是表面打补丁）
3. 确认 Bug 的验证标准都通过了
4. 检查回归测试是否覆盖了关联模块
5. 判断修复是否会引入新问题

输出格式：

| Bug# | 标题 | 根因 | 修复方式 | 是否解决根因 | 回归覆盖 |
|------|------|------|----------|-------------|----------|
| BUG-001 | 登录超时无提示 | 未设置请求超时 | 添加 30s 超时拦截 | 是 | 已覆盖关联接口 |
| BUG-002 | 列表分页错误 | SQL offset 计算错误 | 修改计算公式 | 是 | 未覆盖排序场景 |

### Step 4: 验收决策

基于 Step 3 的对照验证结果，按以下标准判断验收结论：

**验收结论判断标准：**

| 条件 | 结论 | 后续动作 |
|------|------|----------|
| 所有 AC 已实现 + 测试全部通过 + 无未修复 Bug | 通过 | 进入上线准备 |
| 所有 AC 已实现 + 测试全部通过 + 有低级 Bug | 有条件通过 | 修复低级 Bug 后上线，无需复测 |
| 有 AC 未实现 或 测试有失败 或 有高级 Bug | 不通过 | 返回开发修复，需重新验收 |
| AC 实现与描述不一致 | 不通过 | 需重新确认需求，对齐后再验 |

**注意**: "实现与描述不一致"属于需求理解偏差，必须回到需求阶段重新对齐，不能当作通过。

### Step 5: 项目级健康总评

在验收结论确定后，对项目整体健康状态做一次快照评估：

| 维度 | 检查内容 | 判定标准 |
|------|----------|----------|
| 需求覆盖 | AC 是否全部验证 | 全部通过=绿，有遗漏=黄，大量遗漏=红 |
| 文档一致性 | 各文档之间是否一致 | 最新且一致=绿，有滞后=黄，矛盾=红 |
| 契约一致性 | contract-*.md 与实现是否对齐 | 完全对齐=绿，有偏差=黄，严重偏离=红 |
| 变更清洁 | 是否有残留旧代码 | 无残留=绿，有注释掉的旧代码=黄，新旧逻辑共存=红 |
| 跨层对齐 | 前后端/数据库是否一致 | 完全对齐=绿，有未处理项=黄，接口不匹配=红 |

列出需在上线前解决的问题，每项指定负责人。

## 自驱动规则

- 验收报告结论为 **通过** → 自动触发 `collab-retrospective`（复盘）
- 验收报告结论为 **不通过** → 自动生成 Bug 单（如无已有 Bug 单），通知开发修复
- 验收报告结论为 **有条件通过** → 自动创建修复任务，修复后无需重新验收，直接进入上线准备
