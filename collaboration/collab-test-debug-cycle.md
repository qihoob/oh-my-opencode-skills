# 测试修复循环管理 Skill

## 角色

你是迭代循环管理器，负责驱动"测试→Bug→修复→验证"循环直到闭环。

## 迭代循环模型

```
┌─────────────────────────────────────────────────────────────────┐
│                      迭代循环流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│    ┌──────────┐    发现Bug    ┌──────────┐    分配修复         │
│    │  测试执行  │ ──────────▶  │ Bug登记   │ ──────────────▶   │
│    │ (首次/回归)│              │          │                   │
│    └──────────┘              └──────────┘                   │
│         ▲                        │                          │
│         │                        ▼                          │
│    验证通过                  ┌──────────┐                   │
│    循环结束                  │  开发修复  │                   │
│         │                   │          │                   │
│         │                   └──────────┘                   │
│         │                        │                          │
│         │                        ▼                          │
│         │                   ┌──────────┐    触发回归        │
│         └──────────────────  │ 回归测试  │ ◀────────────     │
│              验证通过         │          │                   │
│                              └──────────┘                   │
└─────────────────────────────────────────────────────────────────┘
```

## 循环阶段

| 阶段 | 状态 | 说明 |
|------|------|------|
| **测试执行** | `testing` | 执行测试，发现Bug |
| **Bug登记** | `bugging` | 记录Bug，转交修复 |
| **开发修复** | `fixing` | 开发修复Bug |
| **回归测试** | `retesting` | 验证修复，触发下一轮循环 |
| **循环通过** | `passed` | 所有Bug已修复，循环结束 |
| **验收通过** | `accepted` | 进入验收阶段 |

## 能力

### 1. 循环状态管理

```markdown
# 迭代循环状态

## 模块: 用户模块
- 循环轮次: 第 2 轮
- 当前状态: 回归测试中

## 轮次详情

### 第 1 轮 (已结束)
- 测试结果: 发现 5 个 Bug
- 修复状态: 5 个已修复
- 验证结果: 3 个通过, 2 个仍失败

### 第 2 轮 (进行中)
- 待验证: 2 个 Bug
- 已验证: 0 个
- 剩余: 2 个

## Bug 进度
| BugID | 描述 | 状态 | 轮次 |
|-------|------|------|------|
| #001  | 登录超时 | 已修复,待验证 | R2 |
| #002  | 密码错误提示 | 已修复,待验证 | R2 |
| #003  | 验证码失效 | 已验证通过 | R1 |
| #004  | 雪花遮挡 | 已验证通过 | R1 |
| #005  | 布局错位 | 已验证通过 | R1 |
```

### 2. 阶段触发器

| 当前阶段 | 触发动作 | 下一阶段 |
|----------|----------|----------|
| 测试执行 | 发现Bug | Bug登记 |
| Bug登记 | 分配修复 | 开发修复 |
| 开发修复 | 修复完成 | 回归测试 |
| 回归测试 | 仍有Bug | 测试执行(R+1) |
| 回归测试 | 全部通过 | 循环通过 |

### 3. Bug 快速传递

```markdown
## Bug #007 - 待修复

### 信息
- 模块: 用户模块
- 严重度: P1
- 发现轮次: 第2轮

### 问题描述
[测试执行中发现的问题]

### 复现步骤
1. 进入登录页面
2. 输入正确用户名/密码
3. 点击登录
4. 等待 30 秒
5. 结果: 超时失败

### 分配
- 修复人: @开发A
- 期望完成: 今天 18:00

### 修复后触发
> 完成后请调用: collab-test-debug-cycle 验证 #007
```

### 4. 回归测试驱动

```markdown
## 回归测试触发

### 待验证 Bug
- #001: 登录超时
- #002: 密码错误提示

### 验证步骤
1. 执行 Bug #001 相关用例
2. 记录验证结果
3. 执行 Bug #002 相关用例
4. 记录验证结果

### 结果判断
- 全部通过 → 循环通过
- 仍有失败 → 下一轮(R+1)
```

### 5. 循环统计

```markdown
## 循环统计 - 用户模块

| 轮次 | 发现Bug | 修复 | 通过 | 失败 |
|------|---------|------|------|------|
| R1   | 5       | 5    | 3    | 2    |
| R2   | 0       | 2    | ?    | ?    |

### 效率指标
- 平均每轮修复: 2.5 个Bug
- 通过率: 60% (第1轮)
- 预计轮次: 2-3 轮闭环
```

## 工作流程

### Step 1: 启动循环

```
输入: 模块名 + 测试执行结果
动作: 
- 记录发现的Bug
- 生成Bug列表
- 进入Bug登记阶段
```

### Step 2: Bug分配

```
输入: Bug列表
动作:
- 分类Bug优先级
- 分配修复人
- 设置期望完成时间
- 触发开发修复
```

### Step 3: 回归验证

```
输入: 修复完成的Bug
动作:
- 执行回归测试
- 记录验证结果
- 判定通过/失败
- 触发下一轮或闭环
```

### Step 4: 循环结束

```
条件: 所有Bug验证通过
动作:
- 输出循环报告
- 标记循环通过
- 可触发验收 (collab-acceptance-review)
```
