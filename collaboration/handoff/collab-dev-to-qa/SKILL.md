# Skill: collab-dev-to-qa

**角色**: 协同 (Collaboration)
**功能**: 开发到测试的提测交接
**触发关键词**: 提测、测试交接、开发完成、申请测试、转测试
**版本**: 2.0 (增强协作)

## 依赖文档
- **必须读取**: `.opencode/docs/implementation-{feature}.md`
- **必须读取**: `.opencode/docs/code-review-{feature}.md`


## 产出文档
- **提测交接单**: `.opencode/docs/handoff-dev-to-qa.md`

## 核心能力

1. **提测申请**
   - 说明功能完成情况
   - 提供测试账号/数据
   - 标注风险点
   - 代码审查状态确认

2. **测试范围确认**
   - 影响模块说明
   - 回归测试建议
   - 特殊场景提示
   - 性能测试建议

3. **环境准备**
   - 测试环境确认
   - 数据准备说明
   - 部署版本标识
   - 回滚方案说明

## 前置依赖

**输入文档**:
- `.opencode/docs/implementation-*.md` (实现文档)
- `.opencode/docs/code-review-*.md` (代码审查报告)

**前置条件**:
- 代码审查通过 (passThreshold >= 80)
- 单元测试通过
- 自测完成

**输出文档**:
- `.opencode/docs/handoff-dev-to-qa.md` (提测单)

## 提测检查清单

### 开发完成确认
- [ ] 功能开发完成
- [ ] 单元测试通过 (覆盖率 >= 80%)
- [ ] 代码审查完成 (无严重问题)
- [ ] 自测通过
- [ ] 无阻塞性 Bug

### 文档完整性
- [ ] 实现文档已更新
- [ ] API 文档已更新
- [ ] 数据库变更记录
- [ ] 配置变更说明
- [ ] 契约文档已同步更新（如有跨层变更）

### 测试准备
- [ ] 测试账号准备
- [ ] 测试数据准备
- [ ] 测试环境部署
- [ ] 回归测试范围明确

### 项目级检查点（提测前必检）

```
目的：防止单功能提测导致全局偏移

检查项:
- [ ] 本次变更的涉及文件是否都在预期模块内（无越界修改）
- [ ] 是否有被替代的旧代码尚未清理（变更清洁）
- [ ] 契约文档（如有）是否与最新代码一致
- [ ] 涉及的数据库变更是否已准备迁移脚本
- [ ] 影响的其他模块是否已通知并确认回归范围

如发现异常 → 在提测单中标注，并建议先执行 health-check
```
