# Skill: qa-context-first

**角色**: 测试 (QA)
**功能**: 测试前获取最小上下文
**触发关键词**: 测试上下文、了解被测功能、测试范围、开始测试前

## 产出文档

- **测试上下文报告（可选持久化）**: `.opencode/docs/qa-context-report-{feature}.md`
- 如处于自驱动链路中（chain-executor 执行），必须持久化到文件，以触发下游技能
- 如为单次交互（用户直接说"测试上下文"），可即时输出不写文件

## 依赖文档

按优先级检查：
1. `.opencode/docs/requirement-{feature}.md` — 需求文档（必须）
2. `.opencode/docs/handoff-dev-to-qa.md` — 提测单（如有）
3. `.opencode/docs/implementation-{feature}.md` — 实现文档（如有）
4. `.opencode/docs/code-review-{feature}.md` — 代码审查报告（如有）

## 核心原则

**测试最小上下文原则**：只获取设计和执行测试所需的关键信息，不深入实现细节，聚焦验收标准和业务规则。

## 执行步骤

### Step 1: 读取需求文档

```
检查路径: .opencode/docs/requirement-{feature}.md

必须获取的信息：
- 用户故事（As a... I want to... So that...）
- 验收标准（Given/When/Then 场景列表）
- 优先级（MoSCoW 分级）
- 边界条件和异常流程

如不存在需求文档 → 提示用户先执行 product-requirement-analysis
```

### Step 2: 读取提测单（如有）

```
检查路径: .opencode/docs/handoff-dev-to-qa.md

获取信息：
- 功能完成情况
- 代码审查结果
- 测试重点和建议
- 已知问题和风险点
- 测试环境和测试数据
- 测试账号信息
```

### Step 3: 深入分析实现

```
目的：从代码中提取实际业务规则，识别代码与文档差异，发现测试风险

3.1 读取实现文档 implementation-{feature}.md（如有）
    获取：实现方案、变更文件列表、数据流

3.2 读取实际代码（必须执行）
    这是本技能最核心的步骤。必须读代码，不能只依赖文档。

    必须执行：
    - 读取 service 层代码：理解业务逻辑、校验规则、错误处理
    - 读取 controller 层代码：理解接口参数、状态码、权限检查
    - 读取 model/schema 代码：理解数据约束、字段类型、关联关系
    - 读取前端组件代码：理解表单校验、状态展示、交互流程

    对每层代码，提取：

    | 代码层        | 提取内容                                                           |
    |--------------|-------------------------------------------------------------------|
    | Service      | 业务规则、校验逻辑、状态流转、错误处理、事务使用                     |
    | Controller   | API参数、鉴权逻辑、状态码、错误码映射                               |
    | Model/Schema | 字段约束（类型/长度/必填/唯一）、关联关系、默认值                    |
    | 前端组件      | 表单校验规则、状态展示映射、交互流程、错误提示                       |

3.3 代码 vs 文档差异识别
    对比需求文档中的 AC 和代码实际实现：

    | AC 描述          | 代码实际           | 差异     | 风险 |
    |------------------|--------------------|----------|------|
    | 密码>=6位        | service.ts 校验>=8位 | 不一致   | 高   |
    | 超时30分钟取消    | 代码里是60分钟      | 不一致   | 中   |
    | 无此AC           | 代码有记住我逻辑    | 隐含需求 | 低   |

    差异处理：
    - 代码更严格 → 以代码为准，更新 AC
    - 代码更宽松 → 标注为 Bug，需修复
    - 代码有额外逻辑 → 标注为隐含需求，需补充测试

3.4 测试风险清单
    基于代码分析，列出代码中最可能出 bug 的地方：

    | 风险           | 位置                     | 原因           | 测试优先级 |
    |----------------|--------------------------|----------------|------------|
    | 缺少并发控制    | order.service.ts:45      | 库存扣减无锁   | P0         |
    | 异常被吞       | payment.service.ts:67    | 空 catch 块    | P0         |
    | 校验不完整     | user.controller.ts:23    | 缺少邮箱格式校验 | P1       |
    | 硬编码超时     | cache.service.ts:12      | 60s硬编码      | P2         |
```

### Step 4: 确认测试环境

```
检查项：
- [ ] 测试环境已部署最新版本
- [ ] 测试数据库已初始化
- [ ] 测试账号可用
- [ ] 第三方服务（支付/短信/邮件）mock 或沙箱就绪
- [ ] 配置项正确（Feature Flag 等）
```

### Step 5: 输出测试上下文报告

## 使用场景

### 场景 1: 新功能测试
```
用户: "准备测试用户登录功能"

执行:
1. 读取 requirement-login.md，获取验收标准
2. 检查提测单 handoff-dev-to-qa.md
3. 深入分析实现：
   - 读取 implementation-login.md（如有）
   - 读取 auth.service.ts / auth.controller.ts / user.model.ts / LoginForm.tsx
   - 提取实际业务规则，对比文档
   - 列出测试风险
4. 确认测试环境就绪
5. 输出完整测试上下文
```

### 场景 2: Bug 修复验证
```
用户: "验证支付超时 bug 是否修复"

执行:
1. 读取原始 bug 描述和复现步骤
2. 检查提测单
3. 深入分析修复代码：
   - 读取 payment.service.ts 相关修改
   - 确认超时逻辑是否与 bug 描述一致
   - 检查是否引入新的并发或异常处理风险
4. 评估回归影响
5. 输出精简测试上下文（聚焦 bug 相关场景）
```

### 场景 3: 回归测试
```
用户: "了解本次上线的影响范围"

执行:
1. 读取需求文档
2. 读取提测单
3. 深入分析实现：
   - 扫描所有变更文件
   - 重点读取 service 层变更，提取业务规则变化
   - 识别受影响的关联模块
4. 评估回归测试范围
5. 输出回归测试建议
```

### 场景 4: 紧急测试（时间有限）
```
用户: "快速测一下，马上要上线"

执行:
1. 只读取验收标准（跳过详细文档分析）
2. 快速扫描 service 层核心代码（只看校验和状态流转）
3. 确认环境可用
4. 输出冒烟测试清单（只含 P0 场景 + 代码中 P0 风险项）
```

## 依赖文档检查清单

在执行测试前，必须确认以下文档存在：

- [ ] 需求文档 `requirement-{feature}.md` — **必须**
- [ ] 提测单 `handoff-dev-to-qa.md` — 推荐
- [ ] 实现文档 `implementation-{feature}.md` — 可选
- [ ] 测试用例 `test-cases-{feature}.md` — 后续需要

缺失必要文档时，提示补全后再开始测试。

## 最小上下文原则

1. **先查需求文档** — 验收标准是测试设计的核心依据
2. **代码优先** — 有代码时不只依赖文档，从代码提取实际业务规则
3. **聚焦业务规则** — 不深入代码实现，关注输入/输出/约束
4. **明确测试边界** — 清楚要测什么、不测什么
5. **确认环境就绪** — 避免因环境问题浪费测试时间

## 无需求文档时

如果需求文档不存在，自动推荐：
-> 执行 `qa/context/qa-context-from-code` 从代码逆向推导测试上下文

## 工具可用
- read: 读取需求文档、提测单、实现代码
- grep: 搜索代码变更范围、定位校验逻辑和错误处理
- glob: 扫描项目结构、定位相关代码文件
- bash: 运行测试脚本验证环境连通性
