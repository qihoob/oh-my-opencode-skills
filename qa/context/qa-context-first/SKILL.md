# Skill: qa-context-first

**角色**: 测试 (QA)
**功能**: 测试前获取最小上下文
**触发关键词**: 测试上下文、了解被测功能、测试范围、开始测试前

## 产出文档

无固定产出文件。输出为即时测试上下文报告，供 test-case-design 和 test-executor 使用。

## 依赖文档

按优先级检查：
1. `.opencode/docs/requirement-{feature}.md` — 需求文档（必须）
2. `.opencode/docs/handoff-dev-to-qa.md` — 提测单（如有）
3. `.opencode/docs/implementation-{feature}.md` — 实现文档（如有）
4. `.opencode/docs/code-review-{feature}.md` — 代码审查报告（如有）

## 核心原则

**测试最小上下文原则**：只获取设计和执行测试所需的关键信息，不深入实现细节，聚焦验收标准和业务规则。

## 执行步骤

### Step 1: 读取需求文档

```
检查路径: .opencode/docs/requirement-{feature}.md

必须获取的信息：
- 用户故事（As a... I want to... So that...）
- 验收标准（Given/When/Then 场景列表）
- 优先级（MoSCoW 分级）
- 边界条件和异常流程

如不存在需求文档 → 提示用户先执行 product-requirement-analysis
```

### Step 2: 读取提测单（如有）

```
检查路径: .opencode/docs/handoff-dev-to-qa.md

获取信息：
- 功能完成情况
- 代码审查结果
- 测试重点和建议
- 已知问题和风险点
- 测试环境和测试数据
- 测试账号信息
```

### Step 3: 分析代码变更范围

```
目的：确定测试边界和回归范围

获取信息：
- 涉及的模块/文件列表
- 变更类型（新增/修改/删除）
- 影响范围评估
- 依赖模块是否变更
```

### Step 4: 确认测试环境

```
检查项：
- [ ] 测试环境已部署最新版本
- [ ] 测试数据库已初始化
- [ ] 测试账号可用
- [ ] 第三方服务（支付/短信/邮件）mock 或沙箱就绪
- [ ] 配置项正确（Feature Flag 等）
```

### Step 5: 输出测试上下文报告

## 输出格式

```markdown
## 测试上下文报告：{功能名称}

### 需求概述
- **用户故事**: As a {角色}, I want to {功能}, so that {价值}
- **需求文档**: requirement-{feature}.md
- **优先级**: Must Have / Should Have

### 验收标准清单

| # | 验收标准 | 类型 | 测试优先级 |
|---|----------|------|------------|
| AC1 | {描述} | 正常流程 | P0 |
| AC2 | {描述} | 异常处理 | P0 |
| AC3 | {描述} | 边界条件 | P1 |

### 代码变更范围

| 变更模块/文件 | 变更类型 | 影响评估 |
|---------------|----------|----------|
| src/modules/{module}/service.ts | 修改 | 核心逻辑变更 |
| src/modules/{module}/controller.ts | 新增 | 新接口 |

### 回归测试建议

| 影响模块 | 影响程度 | 建议测试范围 |
|----------|----------|--------------|
| 模块 A | 高 | 全量用例 |
| 模块 B | 中 | 核心用例 |
| 模块 C | 低 | 冒烟测试 |

### 测试环境状态
- **环境地址**: {url}
- **部署版本**: {v1.0.0}
- **数据状态**: {已初始化/需准备}
- **第三方服务**: {mock/沙箱}

### 测试数据需求

| 数据类型 | 数量 | 用途 | 准备状态 |
|----------|------|------|----------|
| 测试用户 | 3 个 | 正常/异常/权限 | 已准备 |
| 测试商品 | 10 个 | 列表查询 | 需创建 |

### 风险项
- ⚠️ {提测单中标注的风险}
- ⚠️ {环境不稳定等已知问题}

### 推荐下一步
→ 执行 `qa/test-case/test-case-design` 设计测试用例
```

## 使用场景

### 场景 1: 新功能测试
```
用户: "准备测试用户登录功能"

执行:
1. 读取 requirement-login.md，获取验收标准
2. 检查提测单 handoff-dev-to-qa.md
3. 分析代码变更范围
4. 确认测试环境就绪
5. 输出完整测试上下文
```

### 场景 2: Bug 修复验证
```
用户: "验证支付超时 bug 是否修复"

执行:
1. 读取原始 bug 描述和复现步骤
2. 检查代码修复范围
3. 评估回归影响
4. 输出精简测试上下文（聚焦 bug 相关场景）
```

### 场景 3: 回归测试
```
用户: "了解本次上线的影响范围"

执行:
1. 分析所有变更文件
2. 识别受影响的模块
3. 评估回归测试范围
4. 输出回归测试建议
```

### 场景 4: 紧急测试（时间有限）
```
用户: "快速测一下，马上要上线"

执行:
1. 只读取验收标准（跳过详细分析）
2. 确认环境可用
3. 输出冒烟测试清单（只含 P0 场景）
```

## 依赖文档检查清单

在执行测试前，必须确认以下文档存在：

- [ ] 需求文档 `requirement-{feature}.md` — **必须**
- [ ] 提测单 `handoff-dev-to-qa.md` — 推荐
- [ ] 实现文档 `implementation-{feature}.md` — 可选
- [ ] 测试用例 `test-cases-{feature}.md` — 后续需要

缺失必要文档时，提示补全后再开始测试。

## 最小上下文原则

1. **先查需求文档** — 验收标准是测试设计的核心依据
2. **聚焦业务规则** — 不深入代码实现，关注输入/输出/约束
3. **明确测试边界** — 清楚要测什么、不测什么
4. **确认环境就绪** — 避免因环境问题浪费测试时间

## 工具可用
- read: 读取需求文档和提测单
- grep: 搜索代码变更范围
- glob: 扫描项目结构
