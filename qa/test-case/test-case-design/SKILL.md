# Skill: test-case-design

**角色**: 测试 (QA)
**功能**: 测试用例设计（含九维度覆盖检查）
**触发关键词**: 设计测试用例、测试场景、用例设计、写测试用例
**版本**: 2.0 (新增九维度覆盖检查)

## 产出文档
**路径**: `.opencode/docs/test-cases-{feature-name}.md`

## 依赖文档
**必须读取**: `.opencode/docs/requirement-{feature-name}.md`

**可选读取**:
- `.opencode/docs/contract-{feature}.md` — 契约文档（状态流转、错误码）
- `.opencode/docs/implementation-{feature}.md` — 实现文档（代码变更范围）

## 核心原则

**测试有效性 = 覆盖全面 + 场景真实 + 结果可验证**

不只是"设计了用例"，还要确保用例真正覆盖了所有需要测试的内容。

## 执行流程

### Step 1: 提取验收标准

```
从 requirement-{feature}.md 提取所有验收标准（AC1, AC2, ...）

对每条 AC，标注：
- 类型: 功能 / 约束 / 异常 / 状态 / 权限
- 优先级: P0 / P1 / P2

如存在 contract-{feature}.md，额外提取：
- 状态流转定义（状态机）
- 错误码定义
- 数据模型约束（字段类型、必填、范围）
```

### Step 2: 九维度覆盖设计

对每条验收标准，从九个维度设计用例：

#### 维度 1: 验收标准覆盖

```
目的: 确保每条 AC 至少有一个正向用例验证

检查规则:
- 每条 AC 必须有 ≥ 1 个正向测试用例
- AC 描述的每个细节点都要有对应验证
- 优先级与需求一致

| AC# | 验收标准 | 正向用例 | 覆盖 |
|-----|----------|----------|------|
| AC1 | 用户可用邮箱登录 | TC001 | ✅ |
| AC2 | 密码错误3次锁定 | TC005 | ✅ |
| AC3 | 支持"记住我" | TC008 | ✅ |

如某条 AC 无对应用例 → 标注 ⚠️ 未覆盖，必须补充
```

#### 维度 2: 正向 + 反向

```
目的: 不能只测正常流程

规则:
- 每个正向用例（正常输入→正常输出）必须搭配反向用例
- 反向用例类型:

| 反向类型 | 示例 |
|----------|------|
| 无效输入 | 错误密码、不存在的邮箱、非法字符 |
| 缺失输入 | 空用户名、空密码、缺少必填字段 |
| 超范围输入 | 超长字符串、负数、超大数据量 |
| 错误顺序 | 未登录直接访问、跳过必填步骤 |

检查:
| 正向用例 | 对应反向用例 | 状态 |
|----------|-------------|------|
| TC001 正常登录 | TC002 错误密码 / TC003 空密码 / TC004 不存在用户 | ✅ |
| TC006 创建订单 | TC007 缺少必填项 / TC008 库存不足 / TC009 超量购买 | ✅ |
```

#### 维度 3: 边界值

```
目的: 极端输入最容易出 bug

对每个有范围/数量/长度限制的字段，设计边界用例:

| 字段 | 约束 | 边界用例 |
|------|------|----------|
| 密码长度 | 6-20字符 | 5字符(不足) / 6字符(最小) / 20字符(最大) / 21字符(超限) |
| 商品数量 | 1-99 | 0(非法) / 1(最小) / 99(最大) / 100(超限) |
| 列表分页 | 每页20条 | 第0页 / 第1页 / 最后一页 / 超出总页数 |
| 金额 | >0 | 0 / 0.01 / 999999.99 / 负数 |
| 时间范围 | 开始<结束 | 开始=结束 / 开始>结束 / 跨年 / 跨时区 |

无明确约束的字段 → 测试空值、特殊字符(<script>、SQL注入、emoji、超长)
```

#### 维度 4: 状态流转

```
目的: 确保所有状态转换路径都被测试

从契约文档（如有）或需求中提取状态机:

  [pending] → [paid] → [shipped] → [completed]
       ↓           ↓
  [cancelled]  [refunded]

为每条转换路径设计用例:

| 转换路径 | 触发条件 | 用例 |
|----------|----------|------|
| pending → paid | 支付成功 | TC020 |
| pending → cancelled | 超时取消 | TC021 |
| paid → shipped | 发货 | TC022 |
| paid → refunded | 申请退款 | TC023 |
| shipped → completed | 签收 | TC024 |

非法转换也必须测试:
| 非法转换 | 预期结果 | 用例 |
|----------|----------|------|
| completed → cancelled | 拒绝/报错 | TC025 |
| cancelled → paid | 拒绝/报错 | TC026 |
```

#### 维度 5: 前后端一致性

```
目的: 界面展示的数据必须和接口返回的数据一致

检查点:
- [ ] 列表页展示的字段 = 接口返回的字段（无遗漏、无多余）
- [ ] 详情页展示的数据 = 接口返回的完整数据
- [ ] 状态文案与后端状态码对应（如 status:2 显示"已发货"）
- [ ] 错误提示与后端错误码对应（如 code:2001 显示"用户名或密码错误"）
- [ ] 金额/日期/数字格式化一致（如后端返回分，前端显示元）
- [ ] 空数据展示一致（空列表、空详情、无权限）

用例示例:
| 检查项 | 操作 | 验证方式 | 用例 |
|--------|------|----------|------|
| 列表字段一致 | 查看用户列表 | 对比页面字段与接口响应 | TC030 |
| 状态映射正确 | 创建待支付订单 | 页面显示"待支付" | TC031 |
| 金额格式化 | 创建 99.90 元订单 | 页面显示 ¥99.90 | TC032 |
```

#### 维度 6: 数据完整性

```
目的: 增删改查后数据必须正确持久化

检查点:
- [ ] 创建 → 数据库/列表中能查到，所有字段正确
- [ ] 修改 → 修改后查询，只有被修改的字段变了，其他不变
- [ ] 删除 → 删除后查询不到，关联数据正确处理（级联/置空）
- [ ] 并发操作 → 两人同时修改同一数据，结果一致不冲突
- [ ] 事务完整性 → 多步操作中间失败，数据回滚不留垃圾

用例示例:
| 操作 | 验证 | 用例 |
|------|------|------|
| 创建订单 | 数据库订单表有记录，状态正确 | TC040 |
| 修改地址 | 订单地址变了，金额不变 | TC041 |
| 删除订单 | 订单和订单项都删除 | TC042 |
| 取消订单 | 库存恢复 | TC043 |
```

#### 维度 7: 权限控制

```
目的: 未授权访问必须被正确拒绝

检查点:
- [ ] 未登录访问受保护页面 → 跳转登录 / 返回 401
- [ ] 普通用户访问管理功能 → 拒绝 / 返回 403
- [ ] 越权操作（A用户修改B用户的数据）→ 拒绝
- [ ] 接口直接调用（跳过前端）→ 同样被拒绝
- [ ] Token 过期 → 正确处理（刷新或跳转登录）
- [ ] 角色权限矩阵覆盖（每个角色的每个权限点）

用例示例:
| 场景 | 操作 | 预期 | 用例 |
|------|------|------|------|
| 未登录访问 | 直接访问 /dashboard | 跳转登录页 | TC050 |
| 越权删除 | A用户调 DELETE /api/users/B | 403 拒绝 | TC051 |
| Token过期 | 过期Token调接口 | 401 + 刷新机制 | TC052 |
| 接口直调 | curl 无Token调管理接口 | 401 拒绝 | TC053 |
```

#### 维度 8: 并发与竞态

```
目的: 多用户同时操作同一数据时不出错

场景:
- 两人同时购买最后一件商品 → 只有一人成功
- 两人同时修改同一条记录 → 后者不覆盖前者（或提示冲突）
- 同一用户快速重复点击提交 → 不产生重复数据
- 定时任务与手动操作冲突 → 数据不混乱

用例示例:
| 场景 | 操作 | 预期 | 用例 |
|------|------|------|------|
| 库存竞态 | 2人同时买最后1件 | 1人成功1人提示库存不足 | TC060 |
| 重复提交 | 快速点3次提交按钮 | 只创建1条数据 | TC061 |
| 并发修改 | 2人同时改同一条记录 | 后者提示冲突或合并 | TC062 |
```

#### 维度 9: 回归影响

```
目的: 改了这个功能，确认其他功能不受影响

来源:
- implementation-{feature}.md 中的「变更影响分析」
- 涉及文件列表 → 识别受影响的关联模块

检查:
| 受影响模块 | 影响程度 | 回归用例 | 状态 |
|------------|----------|----------|------|
| 登录模块 | 高（改了认证逻辑） | 全量回归 | TC070-TC075 |
| 用户列表 | 中（改了用户模型） | 核心用例 | TC076-TC078 |
| 权限系统 | 低（未改） | 冒烟测试 | TC079 |
```

### Step 3: 覆盖度检查（强制）

```
九维度全部设计完成后，执行覆盖度检查：

## 覆盖度检查清单

| 维度 | 检查项 | 覆盖状态 |
|------|--------|----------|
| 1. AC覆盖 | 每条 AC 都有正向用例？ | ✅/❌ |
| 2. 正向+反向 | 每个正向用例都有反向搭配？ | ✅/❌ |
| 3. 边界值 | 所有限制字段都测了边界？ | ✅/❌ |
| 4. 状态流转 | 所有转换路径（含非法）都测了？ | ✅/⚠️无状态/❌ |
| 5. 前后端一致 | 展示数据与接口数据对齐验证了？ | ✅/❌ |
| 6. 数据完整 | 增删改查持久化都验证了？ | ✅/❌ |
| 7. 权限控制 | 未登录/越权/Token过期都测了？ | ✅/❌ |
| 8. 并发竞态 | 关键竞态场景都考虑了？ | ✅/⚠️不涉及/❌ |
| 9. 回归影响 | 受影响模块都规划了回归用例？ | ✅/❌ |

❌ 的维度 → 必须补充用例，不允许遗漏
⚠️ 的维度 → 标注原因（如无状态/不涉及），可跳过
✅ 的维度 → 确认覆盖

覆盖度 = ✅ 数量 / (9 - ⚠️ 数量) × 100%
目标: ≥ 90%
```

### Step 4: 产出测试用例文档

### Step 5: 推荐下一步

```
用例设计完成 → 推荐 test-case-prioritization（排序后执行）
```

## 测试用例文档模板

```markdown
# 测试用例：{功能名称}

**设计日期**: {YYYY-MM-DD}
**需求来源**: requirement-{feature}.md
**契约参考**: contract-{feature}.md（如有）

## 覆盖度检查

| 维度 | 状态 | 备注 |
|------|------|------|
| 1. AC覆盖 | ✅ | 8条AC，8个正向用例 |
| 2. 正向+反向 | ✅ | 8正向 + 12反向 |
| 3. 边界值 | ✅ | 密码长度、分页、金额边界 |
| 4. 状态流转 | ✅ | 5条合法转换 + 2条非法转换 |
| 5. 前后端一致 | ✅ | 状态映射、金额格式、错误提示 |
| 6. 数据完整 | ✅ | 增删改查 + 事务回滚 |
| 7. 权限控制 | ✅ | 未登录、越权、Token过期 |
| 8. 并发竞态 | ⚠️ 不涉及 | 单用户操作 |
| 9. 回归影响 | ✅ | 登录模块全量回归 |

**覆盖度**: 8/8 = 100%

## 用例清单

### TC001: 正常登录成功
- **优先级**: P0
- **维度**: AC覆盖(正向)
- **执行模式**: 🌐 浏览器测试
- **前置条件**: 用户已注册 (test@example.com / Password123)
- **测试步骤**:
  1. 打开登录页
  2. 输入邮箱 test@example.com
  3. 输入密码 Password123
  4. 点击"登录"
- **预期结果**: 跳转到首页，右上角显示用户头像
- **验证方式**: URL 包含 /dashboard，头像元素可见

### TC002: 错误密码登录失败
- **优先级**: P0
- **维度**: 正向+反向(反向)
- **执行模式**: 🌐 浏览器测试
- **前置条件**: 用户已注册
- **测试步骤**:
  1. 打开登录页
  2. 输入邮箱 test@example.com
  3. 输入密码 WrongPassword
  4. 点击"登录"
- **预期结果**: 显示错误提示"用户名或密码错误"
- **验证方式**: 错误提示元素包含对应文案

### TC003: 密码错误3次锁定
- **优先级**: P0
- **维度**: AC覆盖 + 状态流转
- **执行模式**: 🌐 浏览器测试 + 🔌 接口测试
- **前置条件**: 用户已注册
- **测试步骤**:
  1. 连续3次输入错误密码
  2. 第4次输入正确密码
- **预期结果**: 第4次登录被拒绝，提示"账号已锁定，请30分钟后重试"
- **验证方式**: 错误提示文案 + 30分钟后再试可登录

### TC004: 密码长度边界
- **优先级**: P1
- **维度**: 边界值
- **执行模式**: 🌐 浏览器测试
- **前置条件**: 用户已注册
- **测试步骤**:
  1. 输入5位密码 → 预期: 提示"密码至少6位"
  2. 输入6位密码 → 预期: 允许提交
  3. 输入20位密码 → 预期: 允许提交
  4. 输入21位密码 → 预期: 提示"密码最多20位"

### TC005: 未登录访问受保护页面
- **优先级**: P0
- **维度**: 权限控制
- **执行模式**: 🌐 浏览器测试
- **测试步骤**:
  1. 未登录状态直接访问 /dashboard
- **预期结果**: 跳转到登录页

### TC006: 接口Token过期处理
- **优先级**: P1
- **维度**: 权限控制
- **执行模式**: 🔌 接口测试
- **测试步骤**:
  1. 使用过期Token调用 GET /api/users/me
- **预期结果**: HTTP 401，前端触发Token刷新或跳转登录

### TC007: 前后端状态映射一致
- **优先级**: P1
- **维度**: 前后端一致性
- **执行模式**: 🌐 浏览器测试 + 🔌 接口测试
- **测试步骤**:
  1. 创建订单（状态 pending）
  2. 查看订单列表页面的状态显示
  3. 调用接口查看返回的 status 字段
- **预期结果**: 页面显示"待支付"，接口返回 status:1，映射一致

### TC008: 并发购买最后一件商品
- **优先级**: P1
- **维度**: 并发竞态
- **执行模式**: 🔌 接口测试
- **测试步骤**:
  1. 商品库存设为1
  2. 两个用户同时下单购买
- **预期结果**: 1人成功，1人收到"库存不足"提示
```
