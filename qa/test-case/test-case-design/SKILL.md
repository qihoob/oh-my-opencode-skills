# Skill: test-case-design

**角色**: 测试 (QA)
**功能**: 测试用例设计（代码理解驱动 + 九维度覆盖检查）
**触发关键词**: 设计测试用例、测试场景、用例设计、写测试用例
**版本**: 3.0 (新增 Step 0 功能深度理解，九维度与代码理解联动)

## 产出文档
**路径**: `.opencode/docs/test-cases-{feature}.md`

## 依赖文档
**必须读取**: `.opencode/docs/requirement-{feature}.md`

**推荐读取**:
- **必须读取**: `.opencode/docs/contract-{feature}.md``
- **必须读取**: `.opencode/docs/implementation-{feature}.md``

## 核心原则

**测试有效性 = 理解代码 + 覆盖全面 + 场景真实 + 结果可验证**

用例不是凭空想象的，而是基于代码实际实现设计的。不理解代码就写用例，等于在测幻想而不是测系统。

## 执行流程

### Step 0: 功能深度理解（必须，不可跳过）

在写任何用例之前，先搞清楚这个功能到底做了什么。用例的设计依据来自代码，不是来自猜测。

#### 0.1 读取实现代码（必须，非可选）

```
通过 grep 搜索定位功能相关文件：
- 搜索 service、controller、handler、model、entity、前端组件等关键词
- 搜索功能名称、路由路径、API 端点

读取核心实现文件，重点理解：
- 实际的验证逻辑：哪些字段校验了、校验规则是什么、哪些字段没校验
- 实际的状态流转代码：状态机是否完整、转换条件是否完备
- 实际的错误处理路径：哪些异常被捕获了、哪些被吞了、哪些被正确传播了
- 实际的数据操作：是否有事务、级联删除、乐观锁/悲观锁
- 实际的权限检查：哪些接口有鉴权装饰器、哪些接口裸奔
```

#### 0.2 提取实际业务规则

```
从代码中提取业务规则，不是从文档中抄。文档可能过时，代码才是真相。

输出格式：

| 规则类型 | 从代码提取 | 代码位置 |
|----------|-----------|----------|
| 字段约束 | 密码: validator.isLength(val, {min:8, max:20}) | auth.service.ts:23 |
| 状态流转 | pending → paid (支付成功回调触发) | order.service.ts:56 |
| 权限检查 | @RequireAuth, @RequireRole('admin') | user.controller.ts:12 |
| 错误码 | 2001: 用户名或密码错误 | error-codes.ts:5 |
| 副作用 | 下单后扣减库存、发送通知 | order.service.ts:78-82 |
| 数据格式 | 金额以分为单位存储，返回时转为元 | payment.utils.ts:15 |
| 并发控制 | UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0 | stock.repo.ts:34 |
```

#### 0.3 识别代码中的风险点

```
在阅读代码时，主动寻找以下风险信号：

- 缺少校验的字段：代码里没校验但应该校验的（如用户输入直接拼接SQL、未验证枚举值）
- 未处理的异常路径：空 catch 块、缺少 try/catch、Promise 未处理的 rejection
- 并发不安全的操作：共享状态无锁、读写无事务保护、先查后改非原子操作
- 硬编码的值：超时时间、阈值、URL、密钥（应从配置读取）
- 被注释掉的旧逻辑：可能影响行为，或暗示功能不完整
- TODO/FIXME/HACK 标记：明确标注的未完成功能或已知问题
- 隐式依赖：未在接口中暴露但实际依赖的外部服务或状态

每个风险点记录：
| 风险点 | 类型 | 代码位置 | 建议测试场景 |
|--------|------|----------|-------------|
| 密码字段无XSS过滤 | 缺少校验 | auth.service.ts:45 | 输入<script>标签 |
| 库存扣减无事务 | 并发不安全 | order.service.ts:78 | 并发下单测试 |
| catch块为空 | 异常未处理 | payment.service.ts:90 | 支付网关超时场景 |
```

#### 0.4 文档 vs 代码对齐检查

```
对比需求文档和代码实现，找出差异：

检查项：
- AC 说的和代码做的是否一致？
- 代码有没有额外逻辑是 AC 没提到的？
- 代码有没有遗漏 AC 要求的功能？

判定规则：
- 用例设计以代码为准（因为代码才是实际运行的东西）
- 但必须标注差异，差异本身就是一个测试发现

输出：

| AC编号 | AC描述 | 代码实现 | 是否一致 | 差异说明 |
|--------|--------|----------|----------|----------|
| AC1 | 密码长度6-20 | 代码实际为8-20 | 不一致 | 代码比AC更严格 |
| AC2 | 错误3次锁定 | 代码实际为5次 | 不一致 | 代码比AC更宽松 |
| AC5 | 发送邮件通知 | 代码未实现 | 缺失 | 功能未开发 |
```

#### 0.5 输出功能理解清单

```
将 0.1-0.4 的发现整合为一份功能理解清单，供九维度设计时引用：

## 功能理解清单

### 核心逻辑
- 功能做什么（一句话）
- 涉及哪些文件（列出关键文件及职责）
- 主要数据流（输入 → 处理 → 输出）

### 业务规则表（来自 0.2）
（直接引用 0.2 的表格）

### 风险点列表（来自 0.3）
（直接引用 0.3 的表格）

### 文档代码差异（来自 0.4）
（直接引用 0.4 的表格）

这份清单就是后续九维度设计的输入。每个用例必须引用清单中的具体条目。
```

### Step 1: 提取验收标准

```
从 requirement-{feature}.md 提取所有验收标准（AC1, AC2, ...）

对每条 AC，标注：
- 类型: 功能 / 约束 / 异常 / 状态 / 权限
- 优先级: P0 / P1 / P2

如存在 contract-{feature}.md，额外提取：
- 状态流转定义（状态机）
- 错误码定义
- 数据模型约束（字段类型、必填、范围）

注意：此处提取的 AC 需与 Step 0.4 的对齐结果交叉验证。
如果 AC 与代码实现不一致，以代码为准设计用例，同时增加差异验证用例。
```

---

### Step 2: 九维度覆盖设计

对每条验收标准，从九个维度设计用例。**每个维度的用例设计必须引用 Step 0 中发现的具体内容。**

#### 维度 1: 验收标准覆盖

```
目的: 确保每条 AC 至少有一个正向用例验证

检查规则:
- 每条 AC 必须有 ≥ 1 个正向测试用例
- AC 描述的每个细节点都要有对应验证
- 优先级与需求一致
- 参考 Step 0.4 的对齐结果：如果 AC 与代码不一致，以代码行为为准设计用例

| AC# | 验收标准 | 代码依据 | 正向用例 | 覆盖 |
|-----|----------|----------|----------|------|
| AC1 | 用户可用邮箱登录 | auth.service.ts:15 login() | TC001 | ✅ |
| AC2 | 密码错误3次锁定 | auth.service.ts:40 lockAfterFailed() | TC005 | ✅ |
| AC3 | 支持"记住我" | auth.controller.ts:28 rememberToken | TC008 | ✅ |

如某条 AC 无对应用例 → 标注 ⚠️ 未覆盖，必须补充
如 AC 与代码不一致 → 以代码为准设计，标注差异
```

#### 维度 2: 正向 + 反向

```
目的: 不能只测正常流程

规则:
- 每个正向用例（正常输入→正常输出）必须搭配反向用例
- 反向用例设计参考 Step 0.2 提取的验证逻辑：
  代码做了哪些校验，就针对这些校验设计反向用例
  代码没做但应该做的校验，设计反向用例并标注为"代码缺失"

反向用例类型:

| 反向类型 | 设计依据 | 示例 |
|----------|----------|------|
| 无效输入 | Step 0.2 字段约束表 | 不符合正则的邮箱、不在枚举范围内的值 |
| 缺失输入 | Step 0.2 中有必填标记的字段 | 空用户名、空密码、缺少必填字段 |
| 超范围输入 | Step 0.2 中有范围限制的字段 | 超长字符串、负数、超大数据量 |
| 错误顺序 | Step 0.2 状态流转表 | 未登录直接访问、跳过必填步骤 |

| 正向用例 | 对应反向用例 | 代码依据 | 状态 |
|----------|-------------|----------|------|
| TC001 正常登录 | TC002 错误密码 / TC003 空密码 / TC004 不存在用户 | auth.service.ts:23 validatePassword() | ✅ |
| TC006 创建订单 | TC007 缺少必填项 / TC008 库存不足 / TC009 超量购买 | order.service.ts:45 createOrder() | ✅ |
```

#### 维度 3: 边界值

```
目的: 极端输入最容易出 bug

设计方法：从 Step 0.2 提取的字段约束表中，对每个有范围限制的字段设计边界用例。
如果代码校验和文档不一致，以代码为准。

| 字段 | Step 0.2 约束来源 | 边界用例 |
|------|-------------------|----------|
| 密码长度 | auth.service.ts:23 validator.isLength(val, {min:8, max:20}) | 7字符(不足) / 8字符(最小) / 20字符(最大) / 21字符(超限) |
| 商品数量 | order.service.ts:30 MAX_QTY=99 | 0(非法) / 1(最小) / 99(最大) / 100(超限) |
| 列表分页 | list.component.ts:12 PAGE_SIZE=20 | 第0页 / 第1页 / 最后一页 / 超出总页数 |
| 金额 | payment.utils.ts:8 amount > 0 | 0 / 0.01 / 999999.99 / 负数 |
| 时间范围 | date.validator.ts:5 start < end | 开始=结束 / 开始>结束 / 跨年 / 跨时区 |

额外检查 Step 0.3 中的风险点：
- 代码中缺少边界校验的字段 → 设计边界用例，标注"代码未校验此边界"
- 无明确约束的字段 → 测试空值、特殊字符(<script>、SQL注入、emoji、超长)
```

#### 维度 4: 状态流转

```
目的: 确保所有状态转换路径都被测试

设计方法：从 Step 0.2 提取的状态流转表中，为每条转换路径设计用例。
额外检查代码中是否有未在文档中说明的状态转换（Step 0.4 差异表）。

  [pending] → [paid] → [shipped] → [completed]
       ↓           ↓
  [cancelled]  [refunded]

为每条转换路径设计用例:

| 转换路径 | 代码依据 | 触发条件 | 用例 |
|----------|----------|----------|------|
| pending → paid | order.service.ts:56 handlePaymentCallback() | 支付成功 | TC020 |
| pending → cancelled | order.service.ts:70 cancelTimeout() | 超时取消 | TC021 |
| paid → shipped | order.service.ts:85 ship() | 发货 | TC022 |
| paid → refunded | order.service.ts:92 refund() | 申请退款 | TC023 |
| shipped → completed | order.service.ts:100 confirmReceive() | 签收 | TC024 |

非法转换也必须测试（代码中是否有防护？检查 Step 0.3 风险点）:

| 非法转换 | 代码是否有防护 | 预期结果 | 用例 |
|----------|---------------|----------|------|
| completed → cancelled | order.service.ts:110 throw InvalidTransition | 拒绝/报错 | TC025 |
| cancelled → paid | 未找到防护代码 | 待验证 | TC026 |
```

#### 维度 5: 前后端一致性

```
目的: 界面展示的数据必须和接口返回的数据一致

检查点（参考 Step 0.2 中提取的数据格式规则）:
- [ ] 列表页展示的字段 = 接口返回的字段（无遗漏、无多余）
- [ ] 详情页展示的数据 = 接口返回的完整数据
- [ ] 状态文案与后端状态码对应（如 status:2 显示"已发货"）
- [ ] 错误提示与后端错误码对应（如 code:2001 显示"用户名或密码错误"）
      参考 Step 0.2 提取的错误码表
- [ ] 金额/日期/数字格式化一致（如后端返回分，前端显示元）
      参考 Step 0.2 提取的数据格式规则
- [ ] 空数据展示一致（空列表、空详情、无权限）

用例示例:
| 检查项 | 代码依据 | 操作 | 验证方式 | 用例 |
|--------|----------|------|----------|------|
| 列表字段一致 | user.component.ts:20 columns | 查看用户列表 | 对比页面字段与接口响应 | TC030 |
| 状态映射正确 | order.component.ts:35 statusMap | 创建待支付订单 | 页面显示"待支付" | TC031 |
| 金额格式化 | payment.utils.ts:15 分转元 | 创建 99.90 元订单 | 页面显示 ¥99.90 | TC032 |
```

#### 维度 6: 数据完整性

```
目的: 增删改查后数据必须正确持久化

检查点（参考 Step 0.2 提取的数据操作和副作用）:
- [ ] 创建 → 数据库/列表中能查到，所有字段正确
- [ ] 修改 → 修改后查询，只有被修改的字段变了，其他不变
- [ ] 删除 → 删除后查询不到，关联数据正确处理（级联/置空）
      代码中是否有级联删除逻辑？检查 Step 0.2 的副作用表
- [ ] 并发操作 → 两人同时修改同一数据，结果一致不冲突
      代码中是否有乐观锁/悲观锁？检查 Step 0.2 的并发控制
- [ ] 事务完整性 → 多步操作中间失败，数据回滚不留垃圾
      代码中是否使用了事务？检查 Step 0.3 的风险点

用例示例:
| 操作 | 代码依据 | 验证 | 用例 |
|------|----------|------|------|
| 创建订单 | order.service.ts:45 createOrder() + transaction | 数据库订单表有记录，状态正确 | TC040 |
| 修改地址 | order.service.ts:60 updateAddress() | 订单地址变了，金额不变 | TC041 |
| 删除订单 | order.service.ts:75 deleteOrder() CASCADE | 订单和订单项都删除 | TC042 |
| 取消订单 | order.service.ts:70 cancelOrder() → stock.restore() | 库存恢复 | TC043 |
```

#### 维度 7: 权限控制

```
目的: 未授权访问必须被正确拒绝

设计方法：从 Step 0.2 提取的权限检查表中，为每个鉴权点设计用例。
额外检查 Step 0.3 中发现的缺少鉴权的接口（裸奔接口）。

检查点:
- [ ] 每个 @RequireAuth / @RequireRole 装饰器保护的接口 → 验证未授权时正确拒绝
      参考 Step 0.2 权限检查表
- [ ] 缺少鉴权装饰器的接口 → 标注为风险，设计用例验证是否真的不需要权限
      参考 Step 0.3 风险点列表
- [ ] 越权操作（A用户修改B用户的数据）→ 拒绝
- [ ] 接口直接调用（跳过前端）→ 同样被拒绝
- [ ] Token 过期 → 正确处理（刷新或跳转登录）
- [ ] 角色权限矩阵覆盖（每个角色的每个权限点）

用例示例:
| 场景 | 代码依据 | 操作 | 预期 | 用例 |
|------|----------|------|------|------|
| 未登录访问 | auth.guard.ts:8 @RequireAuth | 直接访问 /dashboard | 跳转登录页 | TC050 |
| 越权删除 | 无鉴权（Step 0.3 风险点） | A用户调 DELETE /api/users/B | 待验证 | TC051 |
| Token过期 | auth.middleware.ts:20 verifyToken() | 过期Token调接口 | 401 + 刷新机制 | TC052 |
| 接口直调 | auth.guard.ts:8 | curl 无Token调管理接口 | 401 拒绝 | TC053 |
| 裸奔接口 | 无装饰器（Step 0.3 风险点） | 直接调 /api/export-data | 待验证是否应鉴权 | TC054 |
```

#### 维度 8: 并发与竞态

```
目的: 多用户同时操作同一数据时不出错

设计方法：从 Step 0.3 识别的并发风险点出发，针对每个不安全的操作设计竞态用例。

场景（根据代码实际情况选择）:
- 两人同时购买最后一件商品 → 只有一人成功
      检查 Step 0.2 中库存扣减是否为原子操作
- 两人同时修改同一条记录 → 后者不覆盖前者（或提示冲突）
      检查 Step 0.2 中是否有乐观锁（version字段）或悲观锁
- 同一用户快速重复点击提交 → 不产生重复数据
      检查代码中是否有幂等性保护（idempotency key、前端防抖、后端去重）
- 定时任务与手动操作冲突 → 数据不混乱

用例示例:
| 场景 | 代码依据 | 操作 | 预期 | 用例 |
|------|----------|------|------|------|
| 库存竞态 | stock.repo.ts:34 无锁UPDATE | 2人同时买最后1件 | 1人成功1人提示库存不足 | TC060 |
| 重复提交 | order.service.ts:45 无幂等key | 快速点3次提交按钮 | 只创建1条数据 | TC061 |
| 并发修改 | 无乐观锁（Step 0.3 风险点） | 2人同时改同一条记录 | 待验证行为 | TC062 |
```

#### 维度 9: 回归影响

```
目的: 改了这个功能，确认其他功能不受影响

来源:
- implementation-{feature}.md 中的「变更影响分析」（如有）
- Step 0.1 中读取的涉及文件列表 → 识别受影响的关联模块

检查:
| 受影响模块 | 影响程度 | 影响来源 | 回归用例 | 状态 |
|------------|----------|----------|----------|------|
| 登录模块 | 高（改了认证逻辑） | auth.service.ts 变更 | 全量回归 | TC070-TC075 |
| 用户列表 | 中（改了用户模型） | user.model.ts 字段变更 | 核心用例 | TC076-TC078 |
| 权限系统 | 低（未改） | 无变更 | 冒烟测试 | TC079 |
```

### Step 3: 覆盖度检查（强制）

```
九维度全部设计完成后，执行覆盖度检查：

## 覆盖度检查清单

| 维度 | 检查项 | 代码依据是否引用 | 覆盖状态 |
|------|--------|-----------------|----------|
| 1. AC覆盖 | 每条 AC 都有正向用例？ | ✅/❌ | ✅/❌ |
| 2. 正向+反向 | 每个正向用例都有反向搭配？ | ✅/❌ | ✅/❌ |
| 3. 边界值 | 所有限制字段都测了边界？ | ✅/❌ | ✅/❌ |
| 4. 状态流转 | 所有转换路径（含非法）都测了？ | ✅/❌ | ✅/⚠️无状态/❌ |
| 5. 前后端一致 | 展示数据与接口数据对齐验证了？ | ✅/❌ | ✅/❌ |
| 6. 数据完整 | 增删改查持久化都验证了？ | ✅/❌ | ✅/❌ |
| 7. 权限控制 | 未登录/越权/Token过期都测了？裸奔接口测了？ | ✅/❌ | ✅/❌ |
| 8. 并发竞态 | 关键竞态场景都考虑了？代码风险点覆盖了？ | ✅/❌ | ✅/⚠️不涉及/❌ |
| 9. 回归影响 | 受影响模块都规划了回归用例？ | ✅/❌ | ✅/❌ |

❌ 的维度 → 必须补充用例，不允许遗漏
⚠️ 的维度 → 标注原因（如无状态/不涉及），可跳过
✅ 的维度 → 确认覆盖

"代码依据是否引用" 列检查：每个维度的用例是否都引用了 Step 0 中的具体代码发现。
如果没有引用 → 说明用例是凭空设计的，需要回溯到 Step 0 补充代码依据。

覆盖度 = ✅ 数量 / (9 - ⚠️ 数量) × 100%
目标: ≥ 90%
```
