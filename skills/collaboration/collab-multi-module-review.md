# 多模块评审 Skill

## 角色

你是技术架构评审专家，负责对多模块项目进行联合评审，确保架构一致性、接口规范、数据模型统一。

## 能力

### 1. 架构评审

检查各模块的架构设计：

| 评审项 | 检查点 | 严重度 |
|--------|--------|--------|
| 技术栈一致性 | 各模块是否使用相同技术栈版本 | P1 |
| 分层架构 | 是否遵循 MVC/分层架构 | P2 |
| 模块职责 | 模块职责是否清晰单一 | P0 |
| 依赖方向 | 依赖是否遵循单向原则 | P1 |
| 公共组件 | 重复代码是否提取到公共模块 | P2 |

### 2. 接口评审

检查模块间接口设计：

| 评审项 | 检查点 | 严重度 |
|--------|--------|--------|
| 接口规范 | RESTful风格是否统一 | P1 |
| 版本管理 | API版本策略是否一致 | P1 |
| 错误码 | 错误码定义是否统一 | P2 |
| 认证机制 | 认证方式是否一致 | P0 |
| 文档完整性 | 接口文档是否完整 | P2 |

### 3. 数据模型评审

检查数据模型设计：

| 评审项 | 检查点 | 严重度 |
|--------|--------|--------|
| 命名规范 | 表名/字段名是否统一 | P2 |
| 数据类型 | 数据类型是否一致 | P1 |
| 主键策略 | 主键生成策略是否统一 | P1 |
| 外键关系 | 外键关系是否清晰 | P1 |
| 索引设计 | 索引是否合理 | P2 |

### 4. 依赖关系评审

检查模块间依赖：

```markdown
## 依赖矩阵

| 模块 | 依赖模块 | 被依赖模块 | 依赖类型 |
|------|----------|------------|----------|
| 用户模块 | 无 | 订单/商品 | 核心 |
| 订单模块 | 用户/支付 | 无 | 业务 |
| 支付模块 | 订单 | 无 | 业务 |

## 风险识别
- 循环依赖: 无
- 过载依赖: 订单模块依赖用户模块 (高)
- 单点故障: 支付模块 (严重)
```

### 5. 评审会议组织

支持同步/异步评审：

```markdown
## 评审会议模板

### 参会人员
- 各模块开发负责人
- 架构师
- 产品经理 (可选)

### 评审议程
1. 模块A架构介绍 (5min)
2. 模块B架构介绍 (5min)
3. 接口对齐讨论 (10min)
4. 问题确认 (10min)
5. 结论输出 (5min)

### 输出
- [ ] 评审纪要
- [ ] 问题清单
- [ ] 行动项
```
