## 🎯 按阶段划分的任务清单

### 阶段 1: 产品规划与需求分析

#### 1.1 需求分析 (`product/requirement/product-requirement-analysis`)
**触发词**: 需求分析、用户故事、需求澄清、功能定义

- [ ] 收集原始需求描述
- [ ] 识别目标用户画像
- [ ] 明确业务背景和价值
- [ ] 编写用户故事 (As a... I want to... So that...)
- [ ] 定义验收标准 (Gherkin 格式)
  - [ ] 正常流程场景
  - [ ] 异常处理场景
  - [ ] 边界条件场景
- [ ] 优先级排序 (MoSCoW 方法)
  - [ ] Must Have (必须有)
  - [ ] Should Have (应该有)
  - [ ] Could Have (可以有)
  - [ ] Won't Have (本次不做)
- [ ] 产出需求文档 `.opencode/docs/requirement-{feature}.md`

#### 1.2 需求整理 (`product/requirement/product-requirement-structuring`)
**触发词**: 需求整理、结构化需求

- [ ] 整合多个相关需求
- [ ] 建立需求层次结构
- [ ] 识别需求依赖关系
- [ ] 制定需求实现顺序
- [ ] 输出结构化需求文档

#### 1.3 项目全局分析 (`product/analysis/global-project-analysis`)
**触发词**: 项目分析、模块划分、全局梳理

- [ ] 梳理项目整体架构
- [ ] 识别功能模块边界
- [ ] 分析模块间依赖关系
- [ ] 评估技术栈选型
- [ ] 输出项目架构文档

#### 1.4 数据分析 (`product/analysis/data-analysis`)
**触发词**: 数据分析、埋点、指标

- [ ] 定义业务指标体系
- [ ] 设计数据埋点方案
- [ ] 规划数据看板
- [ ] 制定数据验证方法
- [ ] 输出数据分析文档

#### 1.5 技术评估 (`product/analysis/product-technical-assessment`)
**触发词**: 技术评估、技术可行性

- [ ] 评估技术实现难度
- [ ] 识别技术风险点
- [ ] 分析技术选型优劣
- [ ] 估算开发工作量
- [ ] 输出技术评估报告

#### 1.6 模块需求细化 (`product/module/module-product-requirement`)
**触发词**: 模块需求、详细需求

- [ ] 细化模块级需求
- [ ] 明确模块接口定义
- [ ] 定义模块间交互流程
- [ ] 输出模块需求文档

#### 1.7 模块文档管理 (`product/module/module-document-keeper`)
**触发词**: 模块文档、文档整理、文档管理、把控全局

- [ ] 扫描 `.opencode/docs/` 目录
- [ ] 按模块归类文档
- [ ] 生成模块文档索引 `INDEX.md`
- [ ] 检查文档完整性
- [ ] 标识缺失文档
- [ ] 生成文档审计报告 `DOC_AUDIT.md`

#### 1.8 用户反馈分析 (`product/feedback/analysis`)
**触发词**: 用户反馈、反馈分析、NPS、A/B 测试

- [ ] 收集用户反馈数据
- [ ] 分析 NPS 评分
- [ ] 设计 A/B 测试方案
- [ ] 提炼改进建议
- [ ] 输出反馈分析报告

- [ ] 完成代码审查
- [ ] 开发自测通过
- [ ] 提供测试账号/数据
- [ ] 标注风险点
- [ ] 说明测试重点范围

- [ ] 确认测试环境准备就绪
- [ ] 产出提测申请单

#### 2.3 验收评审 (`collaboration/review/collab-acceptance-review`)
**触发词**: 验收、UAT、验收评审、产品验收、上线确认

- [ ] 准备验收测试用例
- [ ] 组织验收会议
- [ ] 逐项验证验收标准
- [ ] 记录验收问题
- [ ] 评估问题严重程度
- [ ] 做出验收决策
  - [ ] 通过 - 可上线
  - [ ] 有条件通过 - 修复后可上线
  - [ ] 需要复测 - 修复后需重新验收
  - [ ] 不通过 - 延期上线
- [ ] 制定问题修复计划
- [ ] 确认上线时间
- [ ] 产出验收评审报告

#### 2.4 迭代复盘 (`collaboration/process/collab-retrospective`)
**触发词**: 复盘、回顾、总结、retro、迭代总结

- [ ] 收集迭代数据
  - [ ] 计划 vs 实际完成情况
  - [ ] Bug 统计与趋势
  - [ ] 代码审查问题统计
  - [ ] 测试覆盖率
- [ ] 识别做得好的 (Keep)
- [ ] 识别需要改进的 (Improve)
- [ ] 识别停止做的 (Stop)
- [ ] 制定改进行动计划
  - [ ] 明确行动项
  - [ ] 分配负责人
  - [ ] 设定时间节点
- [ ] 规划下迭代目标
- [ ] 产出迭代复盘报告

#### 2.5 文档对齐 (`collaboration/sync/document-alignment`)
**触发词**: 文档对齐、文档同步、三方对齐

- [ ] 比对需求文档与实现文档
- [ ] 比对实现文档与测试用例
- [ ] 识别文档不一致处
- [ ] 协调相关方更新文档
- [ ] 生成文档对齐报告
- [ ] 记录待对齐事项

- [ ] 获取验收标准
- [ ] 设计实现方案
- [ ] 编写代码实现

  - [ ] 边界条件检查
  - [ ] 错误处理验证
- [ ] 产出实现文档 `.opencode/docs/implementation-{feature}.md`

#### 3.4 代码审查 (`dev/code-review`)
**触发词**: 代码审查、Code Review、CR、代码检查、代码评审、代码质量

**通用审查项**:
- [ ] 功能正确性
  - [ ] 代码实现符合需求文档
  - [ ] 边界条件处理完整
  - [ ] 异常场景覆盖
  - [ ] 并发/竞态条件考虑
- [ ] 代码质量
  - [ ] 遵循项目代码风格
  - [ ] 命名清晰有意义
  - [ ] 函数单一职责
  - [ ] 无重复代码 (DRY)
  - [ ] 圈复杂度 < 10
- [ ] 错误处理
  - [ ] 所有异常被捕获处理
  - [ ] 错误信息清晰有用
  - [ ] 无空 catch 块
  - [ ] 资源正确释放
- [ ] 性能
  - [ ] 无 N+1 查询问题
  - [ ] 循环内无低效操作
  - [ ] 适当使用缓存
  - [ ] 大数据量处理优化
- [ ] 安全
  - [ ] 输入参数校验
  - [ ] SQL 注入防护
  - [ ] XSS 防护
  - [ ] 敏感信息不硬编码
  - [ ] 权限校验完整
- [ ] 可维护性
  - [ ] 代码注释充分
  - [ ] 复杂逻辑有说明
  - [ ] TODO/FIXME 标注
  - [ ] 文档同步更新
- [ ] 测试
  - [ ] 单元测试覆盖率 > 80%
  - [ ] 关键逻辑有测试
  - [ ] 边界条件测试
  - [ ] 测试用例清晰独立

**前端特定审查项** (如适用):
- [ ] React/Vue 组件职责单一
- [ ] Props 类型定义完整
- [ ] 状态管理合理
- [ ] 副作用清理 (useEffect cleanup)
- [ ] 列表渲染使用唯一 key
- [ ] 避免内联函数/对象
- [ ] 图片懒加载
- [ ] 组件懒加载
- [ ] 防抖/节流使用
- [ ] 避免不必要的重渲染
- [ ] 样式复用性
- [ ] 响应式适配

**后端特定审查项** (如适用):
- [ ] RESTful API 规范
- [ ] 参数校验 (DTO/Schema)
- [ ] 响应格式统一
- [ ] 状态码正确
- [ ] 数据库索引合理使用
- [ ] 无慢查询风险
- [ ] 事务边界正确
- [ ] 身份认证
- [ ] 授权校验
- [ ] 密码加密存储
- [ ] 敏感数据脱敏

**自动化检查**:
- [ ] ESLint 检查通过
- [ ] TypeScript 类型检查通过
- [ ] 单元测试通过
- [ ] 安全扫描通过
- [ ] SonarQube 质量阈通过

**产出**: 代码审查报告
